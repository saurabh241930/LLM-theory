Cheat Sheet

What is RAG?

Retrieval-Augmented Generation. Instead of relying on pre-training, you retrieve relevant context first, then ask the LLM to answer based on that context.

💡 Solves hallucination by grounding answers in real documents.

The Naive RAG Pipeline

A 5-step process: User Query → Retrieve (Vector DB) → Build Prompt → Generate → Return.

  • Simple QA: Best for "What is X?" style questions.
  • Fast: Direct vector search is very fast.

Offline Indexing

One-time setup. Load documents → Split into chunks → Embed chunks → Store in Vector DB.

  • Pseudocode: for doc in docs: chunks = chunk(doc); db.add(embed(chunks))
  • Goal: Create a searchable map of your knowledge base.

Runtime Query Processing

1. Embed User Query. 2. Search DB for top-k similar vectors. 3. Join chunks into context string. 4. Send to LLM.

💡 The quality of the "Top-k" retrieval directly determines the answer quality.

Limitations & Pitfalls

Naive RAG is simple but has failure modes:

  • Silent Failure: Retrieves wrong chunk confidently.
  • No Reranking: Top-k might not be the best.
  • Complex Queries: Fails on multi-hop reasoning.

Key Takeaways

✅Three Phases: Indexing (Offline), Retrieval (Runtime), Generation.
✅Simple QA: Works best for straightforward questions.
✅Risk: Silent failures (wrong retrieval) are the biggest danger.
✅Next Step: If accuracy is critical, move to Advanced RAG.
