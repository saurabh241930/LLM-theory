04. Chunking Strategies

Splitting Text • Semantic Quality • RAG Foundation
Cheat Sheet

Why Chunking Exists

Documents (10k+ tokens) exceed the Context Window (e.g., 2k-4k tokens). We must split them into manageable pieces.

The Goal

Preserve semantic meaning while keeping chunks small enough to fit.

Fixed-Size & Sliding Window

Fixed-Size

Simple, fast, predictable. Cons: Breaks semantic boundaries (splits sentences mid-way).

Sliding Window

Fixed size but with overlap (e.g., Chunk 2 starts at 50% of Chunk 1). Preserves context at boundaries.

Semantic Chunking

Split based on meaning, not fixed length. Embed every sentence, calculate cosine similarity. If similarity < threshold → Start new chunk.

  • Pros: High semantic coherence, variable sizes.
  • Cons: Expensive (requires embedding all sentences).
Semantic Boundary
Low Similarity Trigger → CUT

Recursive Chunking

Hierarchical splitting. Try Paragraphs → If too big, try Sentences → If too big, try Words.

Respects document structure while handling edge cases of long sections.

Recursive Split

Document-Aware Chunking

Split based on document structure (Markdown headers, HTML tags). Each chunk retains the header context.

  • Best for: Structured docs (Legal, Technical Manuals).
  • Pros: Preserves hierarchy perfectly.

Common Pitfalls

  • Too Small: < 100 tokens loses context.
  • Too Large: > 5k tokens won't fit context window.
  • No Overlap: Loses meaning at edges.
  • No Metadata: Can't trace chunk back to source.

Key Takeaways

✅Necessary Evil: Long docs must be split for RAG.
✅Strategy Choice: Fixed (Fast) vs. Semantic (Quality) vs. Structured (Hierarchy).
✅Size Matters: Target ~1/3 of context window.
✅Metadata is King: Always attach source ID to chunks.
