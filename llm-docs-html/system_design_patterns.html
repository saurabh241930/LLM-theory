Core Architectures & Components

Q1: Design a scalable microservices architecture for a BFSI app

Break the domain into bounded contexts, each owning its data, exposed via API Gateway with async event backbone.

Core Points

  •  Bounded contexts: Auth, Accounts, Transactions, Notifications — each a separate service with its
  • own DB
  •  API Gateway handles routing, rate limiting, auth token validation before requests hit services
  •  Async communication via Kafka/RabbitMQ for cross-service events (e.g., transaction created →
  • notification triggered)
flowchart TD Client --> Gateway["API Gateway"] Gateway --> Auth["Auth Service"] Gateway --> Account["Account Service"] Gateway --> Tx["Transaction Service"] Account --> DB1[("Account DB")] Tx --> DB2[("Tx DB")] Tx -- "Event: Created" --> Kafka[("Kafka / Event Bus")] Kafka --> Notif["Notification Service"]
Depth (Senior Signal):
  • Use Circuit Breaker (Resilience4j) to prevent cascading failures between services
  • Service mesh (Istio) for mTLS, observability, and traffic management at scale
Trap / Watch Out: Don't say 'microservices for everything' — interviewers want to hear when NOT to use them (small team, low complexity = monolith first)

Q2: API Gateway vs Backend for Frontend (BFF) — when to use which

API Gateway is a single entry point for all clients; BFF is a tailored gateway per client type.

Core Points

  •  API Gateway: rate limiting, auth, routing — generic for all consumers
  •  BFF: separate backends for mobile vs web vs third-party — each optimizes its own payload shape
  •  BFF reduces over-fetching: mobile gets a slim response, web gets a richer one from the same
  • underlying microservices
flowchart TD MobileClient["Mobile App"] --> BFF_Mobile["Mobile BFF"] WebClient["Web App"] --> BFF_Web["Web BFF"] BFF_Mobile --> ServiceA["Service A"] BFF_Web --> ServiceA BFF_Web --> ServiceB["Service B"]
Depth (Senior Signal):
  • BFF pattern shines in teams where frontend and backend teams are aligned — each team owns
  • their BFF
  • GraphQL can act as a BFF-like layer, letting clients request exactly what they need
Trap / Watch Out: Don't confuse BFF with API Gateway — they can coexist. API Gateway sits in front of BFFs.

Q3: Explain CQRS and Event Sourcing — when to apply

CQRS separates reads from writes; Event Sourcing stores state as a log of events rather than current values.

Core Points

  •  CQRS: Command model handles writes (normalized DB), Query model handles reads
  • (denormalized/cached) — separate scaling
  •  Event Sourcing: every state change is an immutable event; current state is derived by replaying
  • events
  •  Together they enable full audit trails, temporal queries ('what was the state at T?'), and replay-based
  • debugging
flowchart LR Command["Write Request"] --> CmdService["Command Service"] CmdService --> EventStore[("Event Store (Append Only)")] EventStore -- "Publishes Event" --> MessageBus["Message Bus"] MessageBus --> QueryService["Query Service"] QueryService --> ReadDB[("Read-Optimized DB")] Query["Read Request"] --> QueryService
Depth (Senior Signal):
  • Eventual consistency is the tradeoff — reads may lag behind writes by milliseconds to seconds
  • Use projections to build read models from event streams; rebuild projections anytime by replaying
Trap / Watch Out: CQRS adds complexity — only apply when read/write loads differ significantly or audit trail is a hard requirement (BFSI, compliance)

Q4: How do you handle distributed transactions across microservices?

Use the Saga pattern — a sequence of local transactions coordinated via events, with compensating transactions for rollback.

Core Points

  •  Choreography Saga: each service publishes events and reacts to others' events — no central
  • coordinator
  •  Orchestration Saga: a central orchestrator (e.g., LangGraph, Temporal) tells each service what to do
  • next
  •  Compensating transactions: if step N fails, steps N-1 to 1 are reversed via explicit undo operations
sequenceDiagram participant Order participant Payment participant Inventory Order->>Payment: Reserve Funds Payment-->>Order: OK Order->>Inventory: Reserve Stock Inventory-->>Order: Failed Order->>Payment: Rollback (Compensate)
Depth (Senior Signal):
  • 2PC (Two-Phase Commit) is an alternative but creates tight coupling and blocking — avoid in
  • microservices
  • Outbox pattern ensures atomicity between DB write and event publish — write event to outbox table
  • in same transaction
Trap / Watch Out: Never say 'use a shared DB' to solve distributed transactions — that defeats microservices isolation entirely

Q5: Designing for high availability — failover, circuit breakers, retries

HA is about eliminating single points of failure at every layer — compute, data, network, and dependencies.

Core Points

  •  Circuit Breaker: after N failures, open the circuit (fail fast), then half-open after timeout to test
  • recovery
  •  Retry with exponential backoff + jitter: prevents thundering herd on recovery
  •  Multi-AZ deployments, health checks, and auto-scaling groups ensure compute HA on AWS
flowchart TD Client --> ServiceA["Service A"] ServiceA -- "Call Service B" --> CB{"Circuit Breaker"} CB -- "Closed (Healthy)" --> ServiceB["Service B"] CB -. "Open (Failing)" .-x ServiceB CB -- "Open" --> Fallback["Return Fallback Data"]
Depth (Senior Signal):
  • Bulkhead pattern: isolate thread pools per downstream dependency so one slow service doesn't
  • exhaust all threads
  • Chaos engineering (Netflix Simian Army) — proactively inject failures to validate resilience
Trap / Watch Out: Retries without backoff can amplify failures — always add jitter to prevent synchronized retry storms

Q6: Monolith to microservices migration strategy

Use the Strangler Fig pattern — incrementally extract services while keeping the monolith running in parallel.

Core Points

  •  Strangler Fig: route specific endpoints to new microservices behind a facade; monolith handles the
  • rest
  •  Start with the most independently deployable, lowest-coupling domain (e.g., Notifications, Reporting)
  •  Anti-corruption layer between old and new: translate models so new service doesn't inherit legacy
  • domain concepts
flowchart TD Monolith["Legacy Monolith"] Monolith --> Strangler["Strangler Fig Proxy"] Strangler -- "Old Routes" --> Monolith Strangler -- "/new-api" --> Microservice["New Microservice"]
Depth (Senior Signal):
  • Database decomposition is the hardest part — shared DB must eventually split; use dual writes
  • transitionally
  • Track migration progress with a 'decomposition scorecard' per bounded context
Trap / Watch Out: Don't big-bang rewrite — it almost always fails. Incremental migration with parallel running is safer and provable.

Q7: How would you design a rate limiter for a high-traffic API?

Token Bucket or Sliding Window algorithm, enforced at API Gateway level with Redis as shared state.

Core Points

  •  Token Bucket: refills at fixed rate; requests consume tokens — allows bursting up to bucket capacity
  •  Sliding Window Counter: more accurate than fixed window, tracks request count in rolling time
  • window per key
  •  Redis + Lua script for atomic check-and-decrement across distributed gateway instances
flowchart TD Req["Incoming Request"] --> Gateway["API Gateway"] Gateway --> Redis[("Redis (Token Bucket)")] Redis -- "Tokens Available" --> Forward["Forward to Backend"] Redis -- "No Tokens" --> Reject["429 Too Many Requests"]
Depth (Senior Signal):
  • Rate limit by: IP, user ID, API key, or tenant — different granularity for different threat models
  • Return Retry-After header with 429 response so well-behaved clients back off gracefully
Trap / Watch Out: Fixed window algorithm has edge-case: 2x requests possible at window boundary — sliding window solves this

Q8: Explain the Outbox Pattern and why it matters

Write DB record and event to an outbox table in one transaction; a relay process publishes the event — guarantees exactly-once semantics.

Core Points

  •  Problem: after DB write, app crashes before publishing event — data and messaging fall out of sync
  •  Outbox: write business data AND event record in same DB transaction — atomically consistent
  •  Debezium (CDC) or polling relay reads outbox table and publishes to Kafka/RabbitMQ
flowchart TD Service["Service (Tx)"] --> DB[("Primary DB")] DB -- "Writes to Outbox Table" --> DB Relay["Message Relay / CDC"] -- "Reads Outbox" --> DB Relay --> Kafka[("Kafka Broker")]
Depth (Senior Signal):
  • Transactional outbox enables at-least-once delivery; idempotency keys on consumers handle
  • deduplication
  • Alternative: event-first approach (publish to Kafka first, then write to DB) — different tradeoffs
Trap / Watch Out: Dual writes (write to DB and Kafka independently) are not atomic — one can succeed while the other fails; outbox fixes this

Q9: How do you design a notification system at scale (email, SMS, push)?

Async event-driven pipeline with channel-specific workers, deduplication, and delivery tracking.

Core Points

  •  Producer publishes notification events to Kafka topic; channel consumers (email/SMS/push) process
  • independently
  •  Deduplication via idempotency key — prevents duplicate sends on retry
  •  Delivery status tracked in DB: sent, delivered, failed, bounced — enables retry and audit
flowchart TD Producer["Any Service"] --> Queue["Notification Queue"] Queue --> Worker["Worker Pool"] Worker --> Pref["User Preferences DB"] Worker -- "If SMS" --> Twilio["Twilio API"] Worker -- "If Email" --> SendGrid["SendGrid API"]
Depth (Senior Signal):
  • Priority queues: critical notifications (OTP, fraud alert) get separate high-priority topic
  • Template rendering service decoupled from delivery — supports A/B testing of content
Trap / Watch Out: Don't call email/SMS providers synchronously in request path — always async; user doesn't need to wait for delivery

Q10: What is the Saga pattern vs Two-Phase Commit?

Saga is async and eventually consistent; 2PC is synchronous and strongly consistent but blocks resources.

Core Points

  •  2PC: coordinator asks all nodes to 'prepare', then 'commit' — all or nothing; blocks until all respond
  •  Saga: series of local transactions; failure triggers compensating transactions instead of global
  • rollback
  •  Saga is preferred in microservices — no distributed lock, no blocking, services stay autonomous
flowchart LR Orchestrator["Saga Orchestrator"] Orchestrator --> S1["Service 1: Book Hotel"] Orchestrator --> S2["Service 2: Book Flight"] S2 -- "Fails" --> Orchestrator Orchestrator --> S1_Cancel["Service 1: Cancel Hotel"]
Depth (Senior Signal):
  • 2PC has single point of failure: if coordinator crashes between prepare and commit, nodes are stuck
  • Saga requires designing compensating transactions upfront — not always possible (e.g., sent email
  • can't be unsent)
Trap / Watch Out: 2PC is still valid in single-DB multi-table scenarios or within a single service — just not across microservice boundaries

Q11: How would you design a URL shortener at scale?

Hash-based ID generation, KV store for redirects, CDN at edge for sub-millisecond lookups.

Core Points

  •  Base62 encode a counter or hash to generate short codes — 7 chars gives 62^7 = 3.5 trillion URLs
  •  Redis as primary KV store for redirect lookups — sub-millisecond reads at scale
  •  Cassandra or DynamoDB for persistent storage — wide-column scales horizontally
flowchart TD User["User /long-url"] --> API["URL Service"] API --> KGS["Key Generation Service (KGS)"] KGS --> DB[("URL Mapping DB")] User2["User /xyz123"] --> Cache[("Redis Cache")] Cache -- "Miss" --> DB
Depth (Senior Signal):
  • Counter-based IDs risk enumeration attacks — use random salted hashing instead
  • Analytics pipeline: async Kafka event per click → Spark aggregation → dashboard
Trap / Watch Out: Don't use SQL with auto-increment as the only ID source at scale — distributed counter or pre-generated ID pool is safer

Q12: Explain the Bulkhead pattern

Isolate resources (thread pools, connection pools) per downstream dependency so one failure doesn't starve others.

Core Points

  •  Named after ship bulkheads — a breach in one compartment doesn't flood the whole ship
  •  Each external service gets its own thread pool; if service A is slow, only its pool fills — service B is
  • unaffected
  •  Implemented in Resilience4j, Hystrix (deprecated), or custom semaphore-based limiting
flowchart TD App["Application Node"] App --> Pool1["ThreadPool 1 (Service A)"] App --> Pool2["ThreadPool 2 (Service B)"] Pool1 -- "Exhausted" --> Fail["Fails Fast (Service B is safe)"]
Depth (Senior Signal):
  • Combine with Circuit Breaker: Bulkhead limits concurrent calls; Circuit Breaker stops calls when
  • error threshold is hit
  • Tune pool sizes based on SLA — critical services get larger pools
Trap / Watch Out: Bulkhead without monitoring is incomplete — you need metrics on pool utilization to tune sizes correctly

Q13: How do you handle service discovery in microservices?

Client-side (Eureka/Consul) or server-side (AWS ALB, Kubernetes Service) discovery to resolve service instances dynamically.

Core Points

  •  Client-side: service registry (Eureka) holds instance list; client queries registry and load-balances
  • itself
  •  Server-side: load balancer (ALB, Nginx) handles discovery transparently — client calls LB, LB routes
  •  Kubernetes uses DNS-based discovery — service name resolves to ClusterIP which routes to pods
flowchart LR ServiceA["Service A"] --> Registry["Service Registry (Consul)"] Registry -- "Returns IP" --> ServiceA ServiceA --> ServiceB["Service B"]
Depth (Senior Signal):
  • Health checks in registry: unhealthy instances are deregistered automatically
  • Service mesh (Istio) adds mTLS, retries, and traffic splitting on top of discovery
Trap / Watch Out: Hardcoding IPs or DNS names in config is an anti-pattern — any IP change breaks the service
