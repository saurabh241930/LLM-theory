System Design: E-Commerce AI Agent Room

Multi-Agent System Handling Marketing, Support, & Inventory for 1M Users

Q84: Design an "AI Agent Room" to manage an E-commerce platform (like eBay)

An event-driven architecture using Kafka to orchestrate a swarm of specialized LLM Agents (Marketing, Support, Inventory), utilizing a shared Vector Context DB to maintain global state and resolve conflicts.

This is an advanced Multi-Agent System (MAS) problem. It's wildly different from a standard chatbot because agents operate asynchronously, interact with APIs (tools), and must collaborate with each other without overwhelming the system into infinite loops.

1. High-Level Concept: The "Room" Architecture

Instead of one massive, confusing LLM trying to do everything, we split responsibilities. We use a Publisher/Subscriber model where an Orchestrator routes tasks, and specialized Agents perform them using specific tools.

2. Key Components

A. The Orchestrator (Supervisor) Agent
  • The central nervous system. It watches incoming events (e.g., a "Customer Complaint" payload) and decides which agent gets the job.
  • Implementation: A fast, cheap LLM (like Claude Haiku or GPT-4o-mini) running a strict classification prompt. It outputs JSON routing instructions to message queues.
B. Specialized Department Agents

Each agent is an LLM wrapper configured with a specific persona, system prompt, and access to a strict subset of tools (function calling).

  • Support Agent: Has access to `get_order_status()`, `issue_refund(amount)`, `send_message(user_id)`.
  • Inventory Agent: Has access to `update_stock(sku, qty)`, `request_supplier_reorder()`.
  • Marketing Agent: Has access to `generate_coupon()`, `email_user()`.
C. Shared Context / Memory (The "Whiteboard")

To scale to 1M users, agents cannot constantly talk to each other directly—this creates massive latency and cost. Instead, they write to a shared memory store.

  • When the Support Agent issues a $50 refund to a VIP user, it writes this action to a Vector Database partitioned by `user_id`.
  • If the Marketing Agent later wants to target that user, it runs a semantic search on the user's history in the Vector DB, sees the recent refund, and decides to generate a unique "Sorry for the trouble!" 20% off coupon instead of a generic promo.

Depth (Senior Signal) - Guardrails and Hallucination Control:
  • Infinite Loops: A major risk in Multi-Agent systems is Agent A requesting info from Agent B, who asks A for clarification, forever. Fix: Hard limits on conversation turns (Depth limit = 5) and strict timeout circuits.
  • Human-in-the-Loop (HITL): Define "High-Risk Operations." If the Marketing Agent decides to email a 90% discount code to 100,000 users, the tool executor halts and pushes a notification to a Slack channel for human approval before execution.
  • Observability: You must trace the "Thought Process." Use tracing tools (like LangSmith or Arize Phoenix) to log exactly what context was retrieved and what function was called at every step.
Trap / Watch Out: Interviewers will press you on latency and cost. 1M active users generating events will bankrupt you if every click hits GPT-4. Counter this by emphasizing Heuristic Gateways: Use standard code (cheap if/else rules) to handle simple events, and only trigger the expensive LLM Agents for complex anomalies or unstructured tasks.
