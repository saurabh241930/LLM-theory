2. CTEs: Optimization Boundaries

Common Table Expressions (WITH) are often thought of as "Views." However, in Postgres < v12, they were Optimization Fences—materialized entirely before the main query, which could cause massive memory spikes.

Modern Tip: Use WITH ... AS NOT MATERIALIZED to allow the planner to "inline" the CTE into the main query for better optimization. 
-- Recursive CTE for Hierarchical Comments
WITH RECURSIVE CommentTree AS (
    SELECT id, body, parent_id, 1 as depth
    FROM comments WHERE parent_id IS NULL
    UNION ALL
    SELECT c.id, c.body, c.parent_id, ct.depth + 1
    FROM comments c
    JOIN CommentTree ct ON c.parent_id = ct.id
)
SELECT * FROM CommentTree WHERE depth <= 3;

3. Multi-Version Concurrency (MVCC)

Postgres never overwrites data. When you UPDATE a row, it creates a new version (tuple) and marks the old one as dead. This is why VACUUM exists—to reclaim the disk space from "dead" tuples.

Senior Patterns: FOR UPDATE SKIP LOCKED

Building a high-throughput job queue in SQL? SKIP LOCKED is the gold standard. It allows workers to ignore locked rows, preventing contention bottlenecks.

-- Pick the next 5 available jobs without blocking other workers
SELECT * FROM job_queue
WHERE status = 'pending'
FOR UPDATE SKIP LOCKED
LIMIT 5;

4. The Bridge: pgvector Internals

Postgres is now a first-class Vector DB. With pgvector, you store embeddings alongside business data, enabling Hybrid Filters (e.g., "Find similar images WHERE price < $50").

-- Efficiently query 1,536-dim vectors
CREATE INDEX idx_vec ON products 
USING hnsw (embedding vector_cosine_ops);

SELECT name FROM products 
ORDER BY embedding <=> '[0.12, -0.5, ...]' 
LIMIT 5;
