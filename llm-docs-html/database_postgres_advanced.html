Window Functions, CTEs, Concurrency, and Execution Plans

1. Window Functions

Unlike aggregate functions (GROUP BY) which collapse multiple rows into a single output row, Window Functions compute an aggregate value based on a group of rows but keep the individual rows intact.

SELECT employee_id, department, salary,
       ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) as rank_num,
       LAG(salary) OVER (PARTITION BY department ORDER BY salary DESC) as previous_salary,
       LEAD(salary) OVER (PARTITION BY department ORDER BY salary DESC) as next_salary
FROM employees;

ROW_NUMBER vs RANK vs DENSE_RANK

  • ROW_NUMBER(): Basic sequential counter (1, 2, 3, 4). No ties.
  • RANK(): Assigns the same rank for ties, but skips the next numbers (1, 2, 2, 4).
  • DENSE_RANK(): Assigns the same rank for ties, but does NOT skip numbers (1, 2, 2, 3).

LAG and LEAD

Fetches data from the previous physically contiguous row (LAG) or the next row (LEAD) within the specified partition. Exceptional for calculating month-over-month growth.

2. CTEs & Recursive CTEs

Common Table Expressions (WITH) act as temporary views scoped to a single query execution. They massively improve query readability over deeply nested subqueries.

Recursive CTEs for Hierarchical Data

Recursive CTEs are the universally accepted way to query tree-structured data (like an Org Chart, File System paths, or Comment Replies) in SQL.
WITH RECURSIVE OrgChart AS (
  -- 1. Base Case (Anchor Member): Find the CEO
  SELECT id, name, manager_id, 1 AS level
  FROM employees WHERE manager_id IS NULL
  
  UNION ALL
  
  -- 2. Recursive Step: Find subordinates of everyone currently in the OrgChart
  SELECT e.id, e.name, e.manager_id, oc.level + 1
  FROM employees e
  INNER JOIN OrgChart oc ON e.manager_id = oc.id
)
SELECT * FROM OrgChart;

3. Transaction Isolation Levels

PostgreSQL handles concurrent transactions referencing the same rows using MVCC (Multi-Version Concurrency Control).

Concurrency Phenomena

  • Dirty Read: Reading uncommitted changes from another transaction. (PostgreSQL completely prevents this at all levels).
  • Non-Repeatable Read: A transaction re-reads a row it read previously, finding it was modified by a committed transaction.
  • Phantom Read: A query is executed twice, but a second transaction inserted/deleted rows that match the query criteria in the meantime.
Isolation Level Dirty Read Non-Repeatable Read Phantom Read
Read Committed (PG Default) Prevented Possible Possible
Repeatable Read Prevented Prevented Prevented (in PG, although SQL spec allows it)
Serializable (Strictest) Prevented Prevented Prevented (Guarantees true serial execution outcome)

4. Deadlocks & Row-Level Locking

graph TD Tx1[Transaction A] -- Locks --> Row1 Tx2[Transaction B] -- Locks --> Row2 Tx1 -- Waits for --> Row2 Tx2 -- Waits for --> Row1

Prevention: Always lock rows in the exact same order across all application code. Avoid scattered random updates.

FOR UPDATE SKIP LOCKED

In job queue implementations built on SQL databases, multiple worker threads picking jobs will naturally block each other on SELECT ... FOR UPDATE. SKIP LOCKED tells the worker to ignore rows currently locked by other transactions, instantly grabbing the next available job without waiting.

Advisory Locks

pg_advisory_lock(key) provides application-level locking (mutexes) handled entirely by the database. Instead of locking a specific row/table, you lock a 64-bit integer conceptually tied to a resource.

5. Partitioning Strategies

Partitioning splits a massive logical table into smaller physical pieces (shards) across the disk to dramatically improve read/write performance.

  • Range Partitioning: By Date (e.g., logs_2024_01, logs_2024_02). Best for time-series data.
  • List Partitioning: By explicit category (e.g., users_na, users_eu).
  • Hash Partitioning: By hashing an ID to evenly distribute load across partitions.
Partition Pruning: Ensure queries always include the partitioning key in the WHERE clause. The planner automatically drops irrelevant partitions, scanning only fractional data sets. Do NOT partition if the table is under 10s of millions of rows or if queries span many partitions simultaneously without the partition key.

6. Materialized Views

Regular views execute their underlying query every time they are called. Materialized Views physically store the result set on disk like a real table.

REFRESH MATERIALIZED VIEW CONCURRENTLY

Materialized views eventually go stale as underlying tables change. A regular REFRESH locks the view entirely. Adding CONCURRENTLY requires a unique index on the view and refreshes it in the background without blocking read traffic.

7. JSONB Operations

PostgreSQL acts as an incredibly powerful Document DB natively through its binary JSON type (JSONB).

Querying and Indexing JSONB

Operators: ->> (get text), @> (contains operator), ? (key exists operator).

-- Find users whose "skills" array contains 'React'
SELECT * FROM users WHERE metadata @> '{"skills": ["React"]}';
Performance Note: B-Tree indexes do not work well on JSONB objects. You MUST wrap JSONB columns with a GIN (Generalized Inverted Index) to make containment operators (@>) lightning fast over millions of rows.

8. Upsert Patterns

Upsert (Insert or Update) safely guarantees concurrency resolution without throwing unique constraint violations.

INSERT INTO users (id, last_login)
VALUES (1, NOW())
ON CONFLICT (id) DO UPDATE 
SET last_login = EXCLUDED.last_login
RETURNING *;

EXCLUDED represents the hypothetical row that was proposed for insertion.

9. Execution Plans Deep Dive

Running EXPLAIN ANALYZE executes the query and provides the actual runtime vs estimated runtime.

Join Strategies

  • Nested Loop Join: Optimal for small datasets. For every row in table A, scan table B. Explodes quickly `O(N*M)`.
  • Hash Join: Optimal for large datasets without sorting. Builds a memory hash table of the smaller set, then scans the larger set probing the hash table `O(N+M)`.
  • Merge Join: Optimal for massive, already-sorted datasets. Sorts both tables based on join key, then steps through them sequentially.

10. Analyzing Live Performance

pg_stat_activity & pg_locks

pg_stat_activity provides a live feed of all running connections, the exact query they are currently executing, their state (active vs idle in transaction) and how long they have been executing.

If production freezes, run: SELECT pid, pg_blocking_pids(pid), query FROM pg_stat_activity WHERE wait_event_type = 'Lock'; to instantly reveal which exact PID is locking up the database queue.
