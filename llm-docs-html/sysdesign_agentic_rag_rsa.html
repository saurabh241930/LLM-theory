Designing a RSA NLP Dashboard

Agentic RAG Architecture for Enterprise Insurance Intelligence

The Challenge

🏗️ Scenario: Design an Agentic RAG platform for an RSA (Insurance) enterprise dashboard serving Sales, Policy, Customer, Compliance, and Financial Intelligence.

Requirements:
  • Structured SQL (10GB+) & Unstructured docs.
  • Excel generation, Email delivery, Real-time Reuters news.
  • Cost constraints: No 100B+ models, no large GPU clusters.
  • Handling both linguistic and precision queries.
Step 1: Problem Decomposition

Break the system into 5 logical concern layers to signal structured thinking:

  1. Query Understanding: Intent classification and routing.
  2. Data Retrieval: SQL vs. Document vs. News vs. Excel.
  3. Agent Orchestration: Managing specialized tool-use.
  4. Generation: Mid-size LLM for synthesis.
  5. Delivery: Multi-modal output (Charts, Excel, Email).

Step 3: The Intent Router (SLM Strategy)

Decision: Use a Small Language Model (SLM) like Phi-3-mini (3.8B) or Mistral-7B for routing.

  • Why? Routing is classification, not creative generation. SLMs are fast (<100ms) and cheap.
  • Optimization: Fine-tune on 500-1000 labeled RSA domain examples.

Step 4: Data Layer Design

4a. SQL Layer (10GB Structured)

Strategy: Two-pass Schema-aware retrieval.

  • Pass 1: SLM (SQLCoder-7B) generates SQL.
  • Pass 2: Validator dry-runs explain plan and enforces read-only constraints.

4b. Document RAG Layer

  • Chunking: Semantic chunking for legal clauses (never split mid-sentence).
  • Search: Hybrid (BM25 + Semantic) with CrossEncoder reranking.
  • Metadata: Pre-filter by doc_type, region, or product_line.

4c. Reuters News & Excel

  • News: Real-time tool-calling (Reuters API) cached for 15 mins.
  • Excel: Deterministic execution using pandas and openpyxl, triggered by the Excel Agent.
Step 5: 3-Layer Memory Architecture

Solve the "Long Conversation" problem where models lose track of constraints over time.

  • Working Memory: Most recent 3-5 turns in the LLM context.
  • Session Memory: Key entities (Region, Date range) stored in Redis.
  • Long-Term Memory: User preferences and past session summaries in Vector DB.
💡 Pro Tip: Every 5 turns, use an SLM to summarize the history, explicitly preserving "constraint tokens" (filters) that must never be lost.

Step 6: Model Selection Strategy

Layer Model Choice Reasoning
Intent Router Phi-3-mini (3.8B) Fast, low-cost classification.
SQL Gen SQLCoder-7B Purpose-built, beats 70B models at SQL.
Synthesis Llama-3-70B (4-bit) High-quality reasoning for final summary.
Predictions GAN / ARIMA LLMs are bad at math; use proper ML models.

Step 7: Implementation Guide (Step-by-Step)

Part 1: The Intent Router (Python)

We use a system prompt with few-shot examples to classify the request into specific tool categories. 

from litellm import completion

def route_query(user_query):
    system_prompt = """Classify the query into: [SQL, RAG, NEWS, EXCEL]. 
    Respond only with the category name."""
    
    response = completion(
        model="ollama/phi3", 
        messages=[{"role": "system", "content": system_prompt},
                  {"role": "user", "content": user_query}]
    )
    return response['choices'][0]['message']['content'].strip()

Decision Tradeoffs

Model Size: Tiered (SLM + Mid-LLM) wins over one giant model for cost/latency.
SQL Approach: Schema embeddings win over full schema in context to save tokens.
News: Real-time tool-calls win over embedding news into DB (too dynamic).
Excel: LLM orchestration with Python execution wins over LLM cell-generation (prevents hallucinations).
"I would never route a precision SQL query through a 70B model — that's waste. SLM handles SQL gen, 70B only activates for synthesis and linguistic narration where quality matters."
