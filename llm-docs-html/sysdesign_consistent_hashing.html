Consistent Hashing

The backbone of Memcached, Redis Cluster, and DynamoDB scaling

1. The Rehashing Problem (Why we need it)

If you have `N` cache servers, the standard way to balance load is mapping a key using modular hashing: serverIndex = hash(key) % N.

The Flaw of Standard Modulo

If you have 4 servers and one dies (N becomes 3), the modular equation fundamentally changes for almost every single key. Suddenly, a key that routed to Server 1 now mathematically routes to Server 2. This causes a massive cache miss storm, overwhelming backend databases as 75%+ of the keys are remapped instantly.

2. The Hash Ring Architecture

Consistent Hashing maps both the servers and the data keys onto the same abstract circle (Hash Ring), typically utilizing a space like SHA-1 (0 to 2^160 - 1).

  • Placement: Hash the Server IP using SHA-1 and place it on the ring.
  • Routing Flow: Hash the Data Key, place it on the ring. Then, travel CLOCKWISE along the ring until you find the first Server. Store/read the key there.

3. Node Addition and Removal Flow

The beauty of the Ring is the isolation of disruption.

  • Removing a Node: If Server 1 dies, only the keys that explicitly lived between Server 0 and Server 1 are remapped (they inherently fall forward clockwise to Server 2). The keys belonging to Server 0 and Server 2 remain completely untouched.
  • Adding a Node: If you add Server 3 between Server 1 and Server 2, it intercepts a logical slice of keys that previously routed to Server 2. Again, no other servers are affected. On average, only K/N keys are remapped (where K is total keys, N is total servers).

4. Edge Cases: Virtual Nodes (vNodes)

What if the hash function places Server 0 and Server 1 very close to each other on the ring, but Server 2 is far away? Server 2 would end up managing 80% of the ring's surface area, becoming a massive hot spot.

Solution: Virtual Nodes

Instead of mapping a physical server onto the ring once, map it 100 or 1,000 times using labels like Server1_001, Server1_002, etc.

This uniformly perfectly distributes the load across the ring. If Server 1 goes down, its 1,000 vNodes disappear, meaning its load is evenly subsidized across all the remaining physical servers in the cluster instead of catastrophically dumping onto a single clockwise neighbor.
