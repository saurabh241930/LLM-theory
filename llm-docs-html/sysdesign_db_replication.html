Database Replication: Leader/Follower Internals

Scaling Reads, High Availability, and Handling Lag

1. Architecture Concept

To scale a database that is read-heavy (like Twitter, where there are 1000 reads for every 1 write), we use Leader-Follower (Master/Slave) replication.

graph LR App[Web Server] -- "Writes (Insert, Update)" --> L[Leader DB] L -- Replicates WAL --> F1[Follower 1] L -- Replicates WAL --> F2[Follower 2] App -- "Reads (Select)" --> F1 App -- "Reads (Select)" --> F2
  • The Leader receives all Write traffic.
  • The Followers receive the Write-Ahead Log (WAL) or statement stream from the Leader, reconstruct the data, and serve Read traffic.

2. Sync vs Async vs Semi-Sync Replication

Asynchronous (Default in PG/MySQL)

The Leader writes data to its local disk and instantly acknowledges success to the client. In the background, it streams changes to Followers.

Pros: Fast writes. Cons: If the Leader dies immediately after ACK but before streaming, data is permanently lost.

Synchronous

The Leader writes to local disk, sends data to all Followers, waits for all of them to write to disk, and then ACKs the client.

Pros: Zero data loss. Cons: Excruciatingly slow. If one node in a different timezone goes offline, the entire database blocks writes entirely.

Semi-Synchronous (The Practical Choice)

The Leader requires exactly ONE Follower to acknowledge the write to disk before returning success to the client. This guarantees data survives a single-node failure without dragging write performance to a complete halt.

3. Edge Cases: Replication Lag Anomalies

Because Async replication is the standard, a Follower node might be milliseconds or even minutes behind the Leader. This creates severe UX anomalies.

Read-Your-Own-Writes Anomaly

Scenario: A user updates their profile picture (Write to Leader), then the page instantly refreshes (Read from Follower). The Follower hasn't received the write yet, so the user sees their old profile picture and assumes the save failed.

Solution: The system remembers the timestamp of the user's write. For the next X seconds, all reads specifically for that user's ID are forcefully routed to the Leader instead of the Follower.

Monotonic Reads Anomaly

Scenario: Follower 1 has zero lag. Follower 2 has 5 seconds of lag. A user refreshes their Twitter feed, hitting F1, and sees a new tweet. They refresh again, hitting F2, and the post vanishes!

Solution: Hash the User ID so that a specific user always consistently routes their read queries to the exact same Replica. Time "never moves backward" for them.
