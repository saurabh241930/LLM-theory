RAG Types

Types of Retrieval-Augmented Generation

1. Naive RAG (The Baseline)

The traditional "Index → Retrieve → Generate" linear pipeline.

Workflow: Documents are chunked, embedded into a Vector DB. At query time, the top-k most similar chunks are retrieved and sent to the LLM context.

graph LR A[Documents] --> B[Chunk & Embed] B --> C[(Vector DB)] D[Query] --> E[Retrieve Top-K] C --> E E --> F[LLM Generation]

Example: A simple "Chat with PDF" app where users ask questions about a singular local document.

Trap: Poor retrieval accuracy due to semantic shifts or bad chunking. LLM may hallucinate if provided irrelevant noise.

2. Advanced RAG (Optimization)

Solves Naive RAG issues by adding Pre-Retrieval and Post-Retrieval processing.
  • Pre-Retrieval: Query Rewriting, HyDE (Hypothetical Document Embeddings), and Multi-query expansion.
  • Post-Retrieval: Rerankers (Cross-encoders) to filter noise and Context Compression to fit long documents into tokens.
graph TD Q[User Query] --> PRE[Pre-Processing: HyDE/Rewrite] PRE --> RET[Retrieval: Vector/Hybrid] RET --> POST[Post-Processing: Rerank/Compress] POST --> GEN[LLM Generation]

Example: Enterprise search where the user's initial query "budget info" is expanded to "What is the 2024 marketing budget allocation?" for better vector matches.

3. Modular RAG (Flexibility)

Decouples components into specialized modules like "Routing", "Searching", and "Verifying".

Unlike Advanced RAG's fixed pipeline, Modular RAG can skip certain steps or loop back based on the query complexity.

graph LR R[Router] --> SRCH[Search Module] R --> MEM[Memory Module] SRCH --> VRFY[Verify/Filter Module] VRFY --> GEN[Generation]

Example: A system that first routes a query to "Memory" if it's a follow-up, or to "Search" if it's a new fact-based question.

4. Graph RAG (Deep Relationships)

Combines Vector search with Knowledge Graphs (Neo4j/LangChain Graph).

Why: Vector search is great for semantic similarity but fails at multi-hop reasoning (e.g., "Who is the CEO of the company that bought the company founded by X?").

graph TD NQ[Query] --> VEC[Vector Search: Entities] VEC --> KG[Graph Traversal: Relationships] KG --> CONT[Aggregated Context] CONT --> GEN[LLM Final Answer]

Example: Compliance auditing where you need to track relationships between financial entities across thousands of documents.

5. Agentic RAG (Orchestration)

LLM agents act as autonomous researchers, iteratively searching and reasoning.

Behavior: The agent can decide to search again if the first results are insufficient, or use different tools (Google Search vs. Internal DB) depending on the gap in its knowledge.

graph TD S[User Task] --> AG[Agentic Loop] AG --> TOOL{Tool Choice} TOOL --> DB[(Vector DB)] TOOL --> WEB[Web Search] DB --> EVAL[Evaluate Results] WEB --> EVAL EVAL -- "Incomplete" --> AG EVAL -- "Complete" --> ANS[Final Answer]

Example: An AI Market Researcher that needs to combine stale internal reports with live stock market data to provide a "Market Heatmap".

Senior Interview Deep-Dive

Q: When should you choose Graph RAG over Vector RAG?

Choose Graph RAG when the primary challenge is **relationship density** rather than semantic overlap. If the answer requires traversing nodes (Person -> Works At -> Company -> Owns -> Project), a vector search will get "lost" in similar-sounding chunks, while a graph traversal follows the deterministic edges.

Q: What is the biggest drawback of Agentic RAG in production?

Latency and Cost. Because agents are iterative, they might make 5-10 LLM calls for a single query. This creates high TTFT (Time to First Token) and consumes significant tokens. Scaling requires aggressive semantic caching and strict stop-conditions (recursion limits).
