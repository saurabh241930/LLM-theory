Q73: Compare Redis data structures and their use cases

String for caching, Hash for objects, List for queues, Set for unique membership, Sorted Set for leaderboards, Stream for event log.

Core Points

  •  String: simple key-value cache — SETEX key 3600 value for TTL-based caching
  •  Hash: store object fields — HSET user:1 name 'Saurabh' email '...' — efficient for partial updates
  •  Sorted Set: ZADD leaderboard score member — ZRANGE for top-N, ZRANK for rank lookup
flowchart LR Redis[("Redis")] --> STR["Strings (Cache, Counters)"] Redis --> HASH["Hashes (Object storage)"] Redis --> LIST["Lists (Message queues, Recent items)"] Redis --> SET["Sets (Unique tags, Intersections)"] Redis --> ZSET["Sorted Sets (Leaderboards)"]
Depth (Senior Signal):
  • Redis Stream: append-only log with consumer groups — lightweight Kafka alternative for event
  • sourcing
  • Lua scripts: execute multiple commands atomically — prevents race conditions without
  • MULTI/EXEC overhead
Trap / Watch Out: Storing large objects in Redis as serialized JSON blobs loses the granular update benefit of Hash — use Hash for structured objects

Q74: How do you handle slow queries in production?

Enable slow query log → EXPLAIN ANALYZE → add index or rewrite query → verify with load test.

Core Points

  •  Postgres: pg_stat_statements extension captures slow queries with cumulative stats — find top
  • offenders
  •  EXPLAIN ANALYZE: shows actual execution plan — look for Seq Scan on large tables as the first
  • sign of missing index
  •  Index creation: CREATE INDEX CONCURRENTLY — non-blocking in Postgres, won't lock table in
  • production
flowchart TD Monitor["APM / Slow Query Log"] --> Identify["Find Query taking > 1s"] Identify --> Explain["Run EXPLAIN ANALYZE"] Explain --> SeqScan{"Is it a Seq Scan on massive table?"} SeqScan -- "Yes" --> Fix["Add B-Tree or Hash Index"] SeqScan -- "No" --> Fix2["Vacuum analyze / Reindex"]
Depth (Senior Signal):
  • Query planner statistics: ANALYZE table to update column statistics — planner makes better
  • decisions with current stats
  • pg_stat_user_tables: track sequential scans per table — high seq scan count = missing index on
  • frequently filtered column
Trap / Watch Out: Adding an index blindly without EXPLAIN first is guesswork — always verify the planner uses the new index with EXPLAIN ANALYZE

Q75: What is a database transaction and what are ACID properties?

A transaction is a unit of work that is Atomic, Consistent, Isolated, and Durable — all or nothing, correct, concurrent-safe, and persistent.

Core Points

  •  Atomicity: all statements succeed or all are rolled back — no partial updates
  •  Consistency: transaction brings DB from one valid state to another — constraints are enforced
  •  Isolation: concurrent transactions don't see each other's intermediate state — isolation level controls
  • this
flowchart TD Tx["Transaction Begins"] --> A["Atomicity (All or Nothing)"] Tx --> C["Consistency (Valid state only)"] Tx --> I["Isolation (Concurrent Tx don't interfere)"] Tx --> D["Durability (Survives crashes once committed)"]
Depth (Senior Signal):
  • Durability: committed transactions survive crashes — WAL (Write-Ahead Log) ensures this in
  • Postgres
  • Savepoints: partial rollback within a transaction — rollback to savepoint without rolling back entire
  • transaction
Trap / Watch Out: Long transactions held open (waiting on external API calls) lock rows and cause contention — keep transactions short and tight

Q76: How do you implement optimistic vs pessimistic locking?

Optimistic: version field, fail on conflict (no lock held); pessimistic: SELECT FOR UPDATE holds row lock until commit.

Core Points

  •  Optimistic: UPDATE ... WHERE id=1 AND version=5 — if 0 rows updated, conflict detected; retry or
  • error
  •  Pessimistic: SELECT ... FOR UPDATE — row locked until transaction commits; other transactions
  • wait
  •  Optimistic is better for low-conflict scenarios (most reads, occasional writes) — no blocking
flowchart LR subgraph Pessimistic Locking T1["Tx1: SELECT FOR UPDATE"] --> Lock["Row Locked in DB"] T2["Tx2: SELECT FOR UPDATE"] -. "Blocks waiting for Tx1" .-> Lock end subgraph Optimistic Locking T3["Tx3: Read v=1"] --> Update1["UPDATE ... WHERE v=1 SET v=2"] T4["Tx4: Read v=1"] --> Update2["UPDATE ... WHERE v=1 SET v=2 (FAILS)"] end
Depth (Senior Signal):
  • Deadlock risk with pessimistic locking: two transactions lock rows in opposite order — Postgres
  • detects and rolls back one
  • Optimistic locking with Hibernate/SQLAlchemy: @Version annotation auto-manages version field
Trap / Watch Out: Optimistic locking under high contention (many concurrent writers on same row) causes too many retries — use pessimistic there

Q77: How do you implement full-text search in PostgreSQL?

tsvector + tsquery with GIN index — native Postgres full-text search, no external service needed for moderate scale.

Core Points

  •  tsvector: pre-processed searchable document — to_tsvector('english', 'The quick brown fox')
  •  tsquery: search expression — to_tsquery('quick & fox') matches documents with both words
  •  GIN index on tsvector column: fast full-text search without Elasticsearch for smaller datasets
flowchart TD Text["Raw Text: 'The quick brown fox'"] --> TSVector["to_tsvector() -> 'brown':3 'fox':4 'quick':2"] TSVector --> GIN["GIN Index (Inverted Index)"] Query["User Search: 'jumping foxes'"] --> TSQuery["to_tsquery('fox')"] TSQuery --> Match{"@@ matches GIN Index"} --> Result["Returns Row"]
Depth (Senior Signal):
  • Weighted search: assign different weights to title (A) vs body (D) — title matches rank higher
  • ts_rank() for relevance scoring — ORDER BY rank DESC gives most relevant results first
Trap / Watch Out: Postgres FTS is great up to tens of millions of rows — beyond that, or if you need fuzzy search and faceting, Elasticsearch is better

Core Database Management

Q63: How do you design database indexes for high-performance queries?

Index columns used in WHERE, JOIN, and ORDER BY clauses; avoid over-indexing which slows writes.

Core Points

  •  B-tree index: default in Postgres/MySQL — excellent for equality and range queries
  •  Composite index: column order matters — put equality columns before range columns for optimal
  • use
  •  Covering index: includes all columns the query needs — query served entirely from index, no table
  • lookup
flowchart TD Query["SELECT * FROM users WHERE id=123"] --> BTree["B-Tree Root Node"] BTree --> Branch["Branch Node (100-200)"] Branch --> Leaf["Leaf Node (123)"] Leaf -- "Stores Pointer/Row Data" --> Disk[("Disk Block")]
Depth (Senior Signal):
  • Partial index: index only a subset of rows — CREATE INDEX ON orders(user_id) WHERE status =
  • 'pending'
  • Index bloat: UPDATEs and DELETEs leave dead tuples; REINDEX periodically or use pg_repack
  • online
Trap / Watch Out: Indexes speed reads but slow writes — every write must update all indexes. Don't add an index for every column.

Q64: Explain the difference between SQL INNER JOIN, LEFT JOIN, and FULL

INNER: only matching rows; LEFT: all left rows + matched right; FULL OUTER: all rows from both sides.

Core Points

  •  INNER JOIN: returns rows where join condition is true in both tables — excludes non-matches
  •  LEFT JOIN: returns all rows from left table, NULL for right where no match — use for optional
  • relationships
  •  FULL OUTER JOIN: returns all rows from both tables, NULL where no match — use for
  • reconciliation queries
flowchart LR subgraph INNER JOIN A1((A)) --- B1((B)) end subgraph LEFT JOIN A2((A)) --> B2((B)) end subgraph FULL OUTER JOIN A3((( A + B ))) end
Depth (Senior Signal):
  • CROSS JOIN: Cartesian product — every row in A joined with every row in B; rarely intentional
  • Self JOIN: join a table to itself — useful for hierarchical data (employee-manager relationships)
Trap / Watch Out: OUTER JOIN LEFT JOIN with a WHERE clause on the right table converts it to an INNER JOIN — move the filter to the ON clause instead

Q65: What is database normalization and when do you denormalize?

Normalize to eliminate redundancy and ensure data integrity; denormalize for read performance when query complexity becomes a bottleneck.

Core Points

  •  1NF: atomic values, no repeating groups; 2NF: no partial dependencies on composite keys; 3NF: no
  • transitive dependencies
  •  Denormalization: duplicate data intentionally to avoid expensive JOINs — trade write overhead for
  • read speed
  •  BFSI apps: normalize for OLTP (transactions); denormalize or use a separate OLAP model for
  • reporting
flowchart TD Normalize["Normalization (1NF -> 3NF)"] --> Reduce["Reduces Data Redundancy"] Reduce --> complex["Creates complex multi-join queries"] Denormalize["Denormalization (adding redundant columns)"] --> ReadPerf["Increases Read Performance"] ReadPerf --> WriteRisk["Risk of data inconsistency on writes"]
Depth (Senior Signal):
  • Materialized views: precomputed denormalized query results, refreshed on schedule or on trigger
  • JSONB column in Postgres: store semi-structured data without full denormalization
Trap / Watch Out: Over-normalized schemas with too many joins are slow — 5+ table joins on hot paths warrant targeted denormalization

Q66: How does PostgreSQL MVCC work?

MVCC (Multi-Version Concurrency Control) gives each transaction a consistent snapshot without blocking reads with write locks.

Core Points

  •  Each row has xmin (created by) and xmax (deleted by) transaction IDs — transactions see rows
  • where xmin <= their snapshot
  •  Readers never block writers and writers never block readers — high concurrency for mixed
  • workloads
  •  Dead tuples: old versions of updated rows remain until VACUUM removes them — autovacuum runs
  • automatically
flowchart TD Update["UPDATE row SET val=2"] --> MVCC["MVCC Mechanism"] MVCC -- "Does not delete old row" --> OldRow["Old Row (xmin: 100) invisible to Tx > 105"] MVCC -- "Creates new copy" --> NewRow["New Row (xmin: 105) visible to Tx > 105"] Vacuum["Autovacuum"] -- "Cleans up dead tuples eventually" --> OldRow
Depth (Senior Signal):
  • Transaction isolation levels: Read Committed (default), Repeatable Read, Serializable — each
  • prevents different anomalies
  • Serializable Snapshot Isolation (SSI) in Postgres: full serializability without full locking — uses
  • conflict detection
Trap / Watch Out: Long-running transactions block VACUUM from reclaiming dead tuples — table bloat and query slowdown follows; monitor transaction age

Q67: How do you handle database migrations in production safely?

Apply backward-compatible migrations — expand then contract; never drop columns or rename in a single deploy.

Core Points

  •  Expand: add new column (nullable, no default) — old code still works
  •  Migrate: backfill data, update code to use new column
  •  Contract: once old code is gone, add NOT NULL constraint or drop old column
flowchart TD Start["V1: App uses name column"] --> Step1["Add first_name, last_name cols"] Step1 --> Step2["Deploy V2 App: Writes to both old and new."] Step2 --> Step3["Backfill old data into new columns"] Step3 --> Step4["Deploy V3 App: Reads/Writes ONLY to new columns"] Step4 --> Step5["Drop old name column"]
Depth (Senior Signal):
  • Online schema changes: pt-online-schema-change (MySQL) or pg_repack (Postgres) for ALTER
  • TABLE without locking
  • Flyway/Liquibase: migration version control — track which migrations have run per environment
Trap / Watch Out: Adding NOT NULL column without a DEFAULT in a single migration locks the entire table in Postgres until backfill completes
