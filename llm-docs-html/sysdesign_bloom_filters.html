Bloom Filters

Probabilistic space-efficient structures saving databases from disk I/O

1. The Core Concept

A Bloom Filter is a space-efficient probabilistic data structure that answers one specific question: "Is this element a member of a set?"

The Probabilistic Guarantee

A Bloom filter returns either:

  • "Possibly in set" (False Positives are possible).
  • "Definitely not in set" (False Negatives are mathematically impossible).

2. How it works (The Bit Array)

An empty Bloom filter is a massive bit array of m bits, all set to 0. There must also be k different hash functions defined.

  • Insertion: To add an element (e.g., "apple"), you feed "apple" to all k hash functions to get k array positions. Set the bits at all these positions to 1.
  • Querying: To check if "banana" is in the set, feed it to all k hash functions. If any of the resulting bit positions are 0, the element is absolutely, 100% not in the set. If all the resulting bit positions are 1, the element is probably in the set.
Why a False Positive? Different elements might hash to the exact same bit positions over time, causing collisions. A query might check 3 positions that happen to be exclusively set to 1 by other prior distinct element insertions. You can decrease the false positive probability massively by increasing the size of the bit array (m).

3. Critical Distributed Use Cases

Database Disk I/O Prevention (Cassandra / PostgreSQL)

Checking if a row exists in a 10TB table natively requires a painfully expensive disk read.

Modern databases keep a tiny RAM-based Bloom filter caching every single Primary Key inserted into an SSTable block. When a query searches for User ID 999, the DB hits the Bloom Filter first in microseconds.

Definitely NOT in set: The DB instantly returns `null` to the client without ever looking at the spinning disk. You just saved 100ms of latency.

Probably in set: The DB performs the expensive disk read. Even if it's a 1% false positive rate (the row doesn't exist on disk), you still drastically saved on 99% of useless read I/O queries.

CDN Caching Algorithms

Akamai notably uses Bloom filters for the "One-hit wonder" problem. 75%+ of URLs requested on the web are requested exactly one single time and never again. Caching these one-off URLs instantly fills up expensive RAM and forces out useful popular assets.

Instead, Akamai routes the first request through directly without caching it, but adds the URL to a Bloom Filter. When a request hits that URL a second time, it checks the filter, sees it's "probably in the set" (i.e., has been requested before), and ONLY THEN finally caches the object physically into RAM. This violently increases cache efficiency statistics.

Malicious URL Blocking

Browsers (like Safari/Chrome) download a small (a few Megabytes) Bloom filter of all known malicious phishing URLs. When a user navigates, the browser checks the local offline Bloom filter. If it hits, the browser then sends an API query to Google Safe Browsing to confirm the false-positive status. This protects user privacy (no API call on 99% of safe, normal web browsing).
