Mixed Precision Training

FP16, BF16, Loss Scaling, and PyTorch AMP

The Need for Speed (and VRAM)

Traditionally, neural networks were trained using standard 32-bit floating-point numbers (FP32). For modern LLMs with billions of parameters, calculating math in FP32 takes too long and takes exactly double the GPU memory compared to 16-bit formats. Mixed Precision Training is the standard practice of doing the heavy matrix multiplications in 16-bit, while keeping a "Master Copy" of the weights in 32-bit for safe updating.

The War of 16-Bit Formats: FP16 vs BF16

Not all 16 bits are created equal. A float is made of a Sign (positive/negative), an Exponent (how large the number is), and a Mantissa/Fraction (the decimal precision).

Standard FP16

Range: max ~65,504.

It has massive precision (10 fraction bits) but a tiny exponent (5 bits). Because neural networks don't care about extreme decimal precision, but do care about gradients vanishing to 0 or exploding past 65,000, FP16 is notoriously unstable and prone to overflowing (resulting in NaNs).

BF16 (Brain Float 16) 👑

Range: max ~3.4 x 1038 (Same as FP32).

Invented by Google Brain specifically for Deep Learning. It brutally sacrifices decimal precision (only 7 fraction bits) to maintain the exact same massive 8-bit exponent as FP32. Result: It never overflows. You can train masssive LLMs in BF16 without any of the headache of FP16.

Loss Scaling (The FP16 Hack)

If your GPU doesn't support BF16 (e.g., older Ampere or Turing cards) and you are forced to use FP16, you must use Loss Scaling to prevent gradients from underflowing to exactly 0.0 because of FP16's lack of precision for tiny decimal numbers.

  1. Compute the forward pass in FP16.
  2. Calculate the Loss.
  3. Multiply the Loss by a massive constant (e.g., 1024).
  4. Run Backpropagation. The heavily multiplied gradients are now large enough to fit inside FP16 without rounding down to 0!
  5. Divide the gradients by 1024 before updating the FP32 Master Weights.

Dynamic Loss Scaling handles this automatically, increasing the multiplier until an overflow is detected, scaling it back down, and trying again.

Implementation: PyTorch AMP

PyTorch handles all of this invisibly using Automatic Mixed Precision (AMP). It knows exactly which PyTorch ops should be cast to 16-bit (like Linear Matrix Multiplications) and which must safely remain in 32-bit (like LayerNorm and Softmax).

scaler = torch.cuda.amp.GradScaler() # Handles Loss Scaling for FP16 for data, target in dataloader: optimizer.zero_grad() # Autocast automatically downcasts valid ops to fp16/bf16 with torch.autocast(device_type='cuda', dtype=torch.bfloat16): output = model(data) loss = loss_fn(output, target) # Scaler handles the backwards pass gracefully scaler.scale(loss).backward() # Unscales the gradients and updates weights safely in FP32 scaler.step(optimizer) scaler.update()
