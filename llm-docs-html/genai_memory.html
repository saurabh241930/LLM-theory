Memory in Gen AI

How Large Language Models Remember Context Across Conversations
By default, Large Language Models are completely stateless. An LLM has no memory of the question you asked it five minutes ago. To create the illusion of a continuous conversation (like ChatGPT), developers must architect memory systems outside the model, explicitly passing historical context back into the LLM with every new request.

The Stateless Nature of LLMs

A bare LLM API endpoint operates mathematically as a pure function: f(input_text) = output_text. If you send "My name is Alice", it replies. If you send a separate request "What is my name?" one second later, it will hallucinate or apologize, because it did not retain state from the previous HTTP request.

Implementing Memory: Core Architectures

1. Full Conversation Buffer (Buffer Memory)

The simplest approach. The backend application stores every message exchanged in a database or session store. When the user sends a new message, the backend retrieves the entire history, concatenates it, and sends the massive payload to the LLM.

  • Pros: Perfect recall of the current session.
  • Cons: Quickly hits the token limit (Context Window). Costs scale linearly with the length of the conversation since you are paying for the history on every API call.

2. Sliding Window Memory

Instead of sending the entire history, the backend only sends the last K interactions (e.g., the last 4 user-AI message pairs). Older messages are truncated and forgotten.

3. Summary Memory

To preserve essential long-term context without blowing the token budget, the system recursively summarizes the conversation. When the history exceeds a token threshold, a background LLM call takes the oldest messages and condenses them into a dense summary string.

Input payload: [System Prompt] + [Rolling Summary of turns 1-10] + [Raw text of turns 11-12] + [Current User Message].

4. Entity Memory / Knowledge Graphs

Advanced systems extract structured facts from the conversation over time. If a user says, "I am allergic to peanuts," the backend runs an extraction chain: {"user_allergy": "peanuts"} and stores it in a SQL/NoSQL table or a Graph Database.

Whenever the user interacts in the future, the backend dynamically fetches their "Entity Profile" and injects it into the system prompt, regardless of whether they discussed peanuts 6 months ago.

5. Vector Database Memory (Semantic Memory)

All past messages (or summarized chunks of conversations) are embedded and stored in a Vector DB. When the user asks a new question, the system performs a semantic search against their past history, retrieving only the mathematically relevant historical turns.

  • Example: A user asks, "Can you remind me of that Python function we wrote last week?" The Vector DB retrieves the precise code block from 7 days ago and injects it into the current context window.
Production Trap - Multi-Tenant Memory Leaks

If you implement Semantic Memory using a Vector Database, you must apply strict hard-filtering tags (e.g., metadata: { "user_id": "U-123" }) on the queries. Without explicit isolation, an AI prompt could accidentally retrieve semantic matches from User B's private chat history and output it to User A.
