Concurrency & Load Balancing

Q68: Explain the CAP theorem with practical examples

Distributed systems can guarantee only 2 of 3: Consistency, Availability, Partition Tolerance — since partitions happen, you choose CP or AP.

Core Points

  •  CP (Consistency + Partition Tolerance): returns error if can't guarantee consistency — e.g., HBase,
  • Zookeeper
  •  AP (Availability + Partition Tolerance): returns best-available (possibly stale) answer — e.g.,
  • Cassandra, DynamoDB
  •  CA (no partition tolerance) is not realistic in distributed systems — network partitions always happen
  • eventually
flowchart TD Network["Network Partition (P) occurs"] --> Choice{"System must choose"} Choice -- "Consistency (CP)" --> CP["Reject writes/reads (e.g. Zookeeper)"] Choice -- "Availability (AP)" --> AP["Accept writes/reads risk staleness (e.g. DynamoDB)"]
Depth (Senior Signal):
  • PACELC extends CAP: even without partition, tradeoff exists between Latency and Consistency
  • DynamoDB: configurable consistency — strongly consistent read (CP) or eventually consistent read
  • (AP) per request
Trap / Watch Out: CAP is often misapplied — most systems are tunable between CP and AP per operation, not permanently one or the other

Q69: How do you implement database connection pooling?

Maintain a pool of pre-established DB connections; reuse rather than open new connection per request — critical for throughput.

Core Points

  •  DB connection setup takes 5-50ms — reusing pooled connection saves that overhead on every
  • request
  •  PgBouncer (Postgres): connection pooler with transaction mode — allows thousands of app
  • connections to few DB connections
  •  Pool sizing: (core_count * 2) + disk_spindles is a common starting heuristic; tune based on query
  • duration
flowchart TD App1["App Thread 1"] --> Pool["Connection Pooler (PgBouncer)"] App2["App Thread 2"] --> Pool AppN["App Thread N"] --> Pool Pool -- "Multiplexes 10k connections onto 100 DB connections" --> DB[("PostgreSQL")]
Depth (Senior Signal):
  • Connection pool exhaustion: all connections in use — new requests queue or fail; monitor with pool
  • metrics
  • SQLAlchemy pool settings: pool_size, max_overflow, pool_timeout — tune per service's
  • concurrency profile
Trap / Watch Out: Serverless functions (Lambda) create a new connection per invocation — use RDS Proxy or PgBouncer to avoid connection exhaustion

Q70: How do you design for time-series data storage?

Use a time-series DB (TimescaleDB, InfluxDB) or partition by time in Postgres — optimize for append-heavy, time-range queries.

Core Points

  •  Time-series data: high-volume append-only writes, queries always filter by time range
  •  TimescaleDB: Postgres extension — hypertables auto-partition by time; compression reduces
  • storage 90%+
  •  Retention policies: auto-drop data older than N days — don't store raw tick data forever
flowchart LR Sensor["IoT Sensor Gen"] --> TSDB["Time-Series DB (Influx/Timescale)"] TSDB -- "1. Append-only fast writes" --> Chunk["Time-partitioned Chunks"] Chunk -- "2. Columnar Compression" --> Storage[("Compressed Storage")] TSDB -- "3. Continuous Aggregates" --> Dash["Grafana Dashboard"]
Depth (Senior Signal):
  • Downsampling: aggregate raw data into lower resolution over time — keep 1-minute aggregates,
  • drop raw after 30 days
  • Write amplification: batch inserts instead of row-by-row — time-series DBs are optimized for batch
  • writes
Trap / Watch Out: Using a standard relational table for time-series data without partitioning causes full-table scans on every time-range query

Q71: What is the N+1 query problem and how do you fix it?

N+1: fetching a list of N items then making N individual queries for related data — fix with JOIN, eager loading, or DataLoader batching.

Core Points

  •  Example: fetch 100 orders, then loop and query each order's user separately = 101 queries
  •  Fix with JOIN: single query returns orders with user data — works for SQL ORMs
  •  DataLoader (GraphQL): batches all IDs from one render cycle, fetches in one query
flowchart TD N1["N+1 Query Issue"] --> Q1["SELECT * FROM users (1 query)"] Q1 --> Loop["For loops through N users"] Loop --> QN["SELECT * FROM posts WHERE user_id=? (N queries)"] Fix["Fix: Eager Loading"] --> FixedQ["SELECT * FROM posts WHERE user_id IN (1,2...N) (1 query)"]
Depth (Senior Signal):
  • SQLAlchemy: selectinload() or joinedload() for eager loading relationships
  • Django: select_related() for FK joins, prefetch_related() for M2M relationships
Trap / Watch Out: ORMs hide N+1 queries behind clean API — always log SQL in dev to catch them; they're invisible until production load reveals them

Q72: How do you implement database sharding?

Partition data horizontally across multiple DB instances by a shard key — each shard holds a subset of rows.

Core Points

  •  Shard key choice is critical — choose high-cardinality key that distributes evenly (user_id, tenant_id)
  •  Hash sharding: hash(shard_key) % num_shards — even distribution but resharding is expensive
  •  Range sharding: shard by date or ID range — easier resharding but hotspot risk (all recent writes on
  • one shard)
flowchart TD App["Application"] --> Router["Sharding Router / Proxy"] Router -- "Hash(user_id) % 3 == 0" --> ShardA[("DB Shard 1")] Router -- "Hash(user_id) % 3 == 1" --> ShardB[("DB Shard 2")] Router -- "Hash(user_id) % 3 == 2" --> ShardC[("DB Shard 3")]
Depth (Senior Signal):
  • Consistent hashing: adding/removing shards only remaps a fraction of keys — better than modulo
  • for dynamic resharding
  • Cross-shard queries: aggregations across shards require scatter-gather — expensive; design
  • queries to stay within one shard
Trap / Watch Out: Sharding before you need it adds massive complexity — exhaust vertical scaling, read replicas, and partitioning first
