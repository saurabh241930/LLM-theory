LLM Inference & GPU Optimization

Scaling models for production: Latency, Throughput, and Cost

1. KV Caching: The Performance Lifeline

During LLM generation, the model predicts one token at a time. Without KV (Key-Value) Caching, the model would re-calculate the Attention keys and values for all previous tokens in every step ($O(n^2)$ complexity).

Mechanism: We store the computed K and V vectors in GPU memory. For the next token, we only calculate K/V for the new token and concatenate it with the cache. This brings complexity down to $O(n)$.

The Cost: High VRAM usage. A Llama-3 70B model with a large context can consume gigabytes of VRAM just for the KV cache.

2. Speculative Decoding

How do we speed up a massive model? Use a tiny "Draft" model to guess the next 5-10 tokens, and then use the "Large" model to verify them in a single forward pass.

graph TD A[Tiny Model] -->|Speculates| B[Token 1, 2, 3] B --> C[Large Model] C -->|Single Pass| D{Validate} D -- Correct --> E[Accept all 3] D -- Wrong --> F[Reject & regenerate from error]

Benefit: 2x-3x speedup without any loss in accuracy.

3. Quantization: Fitting Giants on Small GPUs

Reducing the precision of model weights (e.g., from 16-bit to 4-bit) to save memory and increase speed.

Method Mechanism Best for
GPTQ Post-training quantization using second-order info. Static batching, generic usage.
AWQ Activation-aware Weight Quantization. Preserves "important" weights. Better accuracy at 4-bit than GPTQ.
GGUF CPU/GPU hybrid format (llama.cpp). Local/Consumer hardware.

4. Model Parallelism

When a model is too big for one GPU (e.g. 175B params), we split it.

  • Pipeline Parallelism: Splitting by layers. GPU 1 handles layers 1-10, GPU 2 handles 11-20. (Higher latency due to sequential flow).
  • Tensor Parallelism: Splitting the actual weight matrices. Each GPU calculates a part of the matrix multiplication simultaneously. (Requires high-speed NVLink).

5. Flash Attention: The IO-Aware Breakthrough

Standard attention is memory-bottlenecked, meaning the GPU spends more time waiting for data from VRAM than actually calculating. Flash Attention (Dao et al.) optimized this by rethinking the algorithm for GPU architecture.

Mechanism: Tiling & Recomputation

  • Tiling: It breaks the large Q, K, and V matrices into small "tiles" that fit inside the GPU's SRAM (which is 10x faster than VRAM).
  • Recomputation: Instead of storing the massive $O(N^2)$ attention matrix in VRAM, Flash Attention recalculates parts of it during the backward pass. This trade-off (more math, less memory) is actually faster because memory IO is the real bottleneck.

Result: Drastically reduces memory footprint and allows for massive 100k+ context windows.

6. Mixture of Experts (MoE): Sparse Activation

Why use a 1.8 trillion parameter model (like GPT-4) for every single query? MoE only activates a small fraction of the model for each token.

Architecture: The Router

Instead of one large Feed-Forward Network (FFN), we have multiple "Experts" (smaller FFNs).

graph TD A[Input Token] --> B{Gating/Router} B -->|Expert 1| C[FFN 1] B -->|Expert 2| D[FFN 2] B -->|Expert 3| E[FFN 3] C --> F[Weighted Sum] D --> F E --> F F --> G[Output]
  • Total Params: High (e.g., 8 experts x 7B each = 56B).
  • Active Params: Low (e.g., only 2 experts used = 12.9B).

Trade-off: You get the performance of a huge model with the inference cost of a smaller one. However, it requires massive VRAM to keep all experts loaded.
