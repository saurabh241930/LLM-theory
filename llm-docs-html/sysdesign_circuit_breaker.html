Circuit Breaker & Bulkhead Patterns

Preventing Cascading Failures in Distributed Systems

1. The Circuit Breaker Pattern

In a microservices architecture, if Service A synchronously calls Service B, and Service B experiences a massive 30-second latency spike or complete outage, Service A will exhaust all of its threads waiting for B to respond. This instantly crashes Service A, propagating the catastrophic failure across the entire system network.

The Three States

  • Closed: Standard operation. Traffic flows smoothly to the downstream service. The circuit is continuously monitoring error rates and execution times.
  • Open: If the error rate crosses a threshold (e.g., 50% failures in 10s), the circuit "trips". ALL subsequent requests from Service A to Service B immediately fail fast (throwing a defined fallback or cached response) without actually making the network call, relieving all pressure on Service B.
  • Half-Open: After a timeout, the circuit allows exactly ONE test request through to check if Service B has recovered.

2. The Bulkhead Pattern

If the hull of a ship is comprised of one massive compartment, a single puncture sinks the entire vessel. By isolating the hull into partitioned, watertight bulkheads, a breach in one section does not flood the entire ship.

Resource Isolation

In software, if a Web Server has a connection pool of 500 threads, and an obscure Payment API integration suddenly goes dark and hangs, 100% of the 500 threads will quickly get trapped polling that API. The server becomes completely unresponsive, dropping simple calls like loading the homepage.

Implementation: We partition our connection pools (Bulkheading). The Payment API integration is strictly allowed only 50 threads from the pool. The Homepage Rendering service is given 400 threads securely. If the Payment API hangs, exactly 50 threads lock up, and that internal feature crashes, but the core Homepage stays 100% perfectly available for other users.

3. Combining Strategies (Resiliency)

A senior system designer prevents downtime by inherently expecting it.

  • Retry with Exponential Backoff: Re-executing failed network calls aggressively floods a struggling downstream service (Denial of Service). A Retry must ALWAYS include mathematical backoff (wait 1s, then 2s, then 4s, 8s) and randomized Jitter (scattering the retries across time so 5,000 servers don't retry on the exact same millisecond mark).
  • Fallback Gracefully: If the Recommendations Microservice circuit breaker trips Open, the UI shouldn't render a 500 error block. The wrapper code should catch the circuit trip, and render a hard-coded static list of "Top Rated Items" as a safe fallback.
