Advanced Node.js & System Architecture

Express Internals, Auth Flows, ORMs, and Job Queues

1. Express Middleware Chain

Express relies on a strict sequential execution of functions called middleware. The order in which middleware is registered directly dictates the order of execution.

graph LR Req[HTTP Request] --> M1[Logger Middleware] M1 -- next() --> M2[Auth Middleware] M2 -- next() --> R[Route Handler] R -- response --> Res[HTTP Response] M2 -. next(err) .-> E[Error Middleware] R -. next(err) .-> E E --> Res

The 4-Argument Error Middleware

Error-handling middleware is defined just like regular middleware functions, except it MUST have four arguments instead of three: (err, req, res, next). If you only provide three arguments, Express interprets it as regular middleware, meaning it won't catch errors.

Trust Proxy

When running Express behind a reverse proxy (like Nginx or an AWS Load Balancer), Express will consider the connection's source IP to be the proxy's IP, not the client's. Setting app.set('trust proxy', 1) tells Express to trust the X-Forwarded-For header sent by the proxy, correctly populating req.ip and req.protocol.

2. JWT Refresh Token Rotation

Storing Access Tokens (AT) in LocalStorage exposes them to XSS attacks. A secure standard involves using two tokens: a short-lived Access Token kept in memory, and a long-lived Refresh Token (RT) explicitly stored in an httpOnly cookie.

Set-Cookie: RT (httpOnly), Body: AT (15m) Note over Client: Stores AT in JS memory variables Client->>API: 3. Request + Header: Bearer AT API-->>Client: 4. Data Note over Client, API: ...15 minutes later (AT expires)... Client->>API: 5. Request + Header: Bearer AT API-->>Client: 6. 401 Unauthorized Client->>API: 7. POST /refresh (Cookie RT automatically sent) Note over API: Validates RT, Revokes old RT, generates new Pair API-->>Client: 8. Set-Cookie: NEW RT (httpOnly), Body: NEW AT

Refresh Token Revocation & Rotation

Rotation: Every time the `/refresh` endpoint is hit, the old RT is invalidated and a completely new RT is issued. If a stolen RT is used, the system detects RT reuse (as the legitimate user already rotated it, or vice versa) and invalidates the entire token family, forcing re-authentication.

3. Passport.js Internals & OAuth2

Passport.js abstracts authentication strategies (Local, Google, GitHub). It relies heavily on session management using serializers.

Serialize / Deserialize User

serializeUser decides what data goes into the session cookie, while deserializeUser reconstructs the user object from that data.

Typically, you only serialize the user.id to keep the session payload small. On every subsequent request, deserializeUser takes that ID, fetches the fresh user record from the database, and attaches it to req.user.

4. Prisma vs TypeORM vs Mongoose

Choosing the right database interaction layer affects developer velocity and application performance.

  • Prisma: Next-generation Node.js and TypeScript ORM. Generates a fully type-safe client based on a unified schema.prisma file. Excellent developer experience. Uses a Rust engine under the hood.
  • TypeORM: Traditional Active Record and Data Mapper ORM influenced by Hibernate. Relies heavily on TypeScript decorators (@Entity, @Column). Better for complex legacy relational schemas but can be verbose.
  • Mongoose: The de-facto ODM (Object Data Modeling) library for MongoDB. Enforces schema validation at the application layer on top of a schemaless database.

The N+1 Problem & Eager vs Lazy Loading

The N+1 problem occurs when an ORM executes 1 query to fetch a list of N records, and then N additional queries to fetch a related entity for each record.

Solution (Eager Loading): Fetch all necessary associated data in the initial database query using JOINs. In Prisma, this is done using the include keyword. In TypeORM, this is done using relations. Alternatively, use GraphQL's DataLoader to batch and deduplicate queries.

5. Bull & BullMQ Job Queues

For operations that are too slow (e.g., sending emails, video processing) to reside in the HTTP request-response cycle, Node.js offloads tasks to Job Queues backed by Redis.

graph LR Producer[API Server] -- Adds Job --> Redis[(Redis Queue)] Redis -- Pulls Job --> W1[Worker Process 1] Redis -- Pulls Job --> W2[Worker Process 2] W1 -- Success/Fail --> Redis

Key Queue Concepts

  • Concurrency: The number of jobs a single worker process will concurrently execute. If a worker has concurrency 5, it will pull up to 5 jobs simultaneously from Redis.
  • Retries & Backoff: Automatically retry a failed job. Backoff strategies (like exponential backoff) ensure the system waits increasingly longer between retries, avoiding hammering a failing downstream service.
  • Priority: Assigning weights to jobs. E.g., 'Password Reset Email' (Priority 1) executes before 'Weekly Newsletter' (Priority 10).
  • Delayed Jobs: Jobs queued to be processed at a specific timestamp or timeframe in the future.

6. WebSockets & Socket.io Scale-Out

WebSockets provide full-duplex, bidirectional communication over a single TCP connection. Socket.io is a wrapper that provides auto-reconnection, fallback to HTTP long-polling, namespaces, and rooms.

Rooms & Namespaces

Namespaces: Act like different endpoints or paths (e.g., /chat vs /notifications), isolating connections entirely.

Rooms: Arbitrary channels within a namespace that sockets can join() and leave(). Useful for group chats or broadcasting to a specific tenant subset (e.g., io.to('room1').emit(...)).

The Redis Adapter Scale-Out Problem

When you scale a Node.js WebSocket server to multiple instances behind a load balancer, standard Socket.io breaks. A user connected to Instance A won't receive a broadcast emitted by Instance B.

The Fix: The Socket.io Redis Adapter. It uses Redis Pub/Sub as a backplane. When Instance B emits a message to a room, it publishes it to Redis. Instance A is subscribed to Redis, receives the message, and forwards it to its locally connected users.

7. OpenTelemetry in Node.js

OpenTelemetry (OTel) is the industry standard for generating and collecting telemetry data (Metrics, Logs, and Traces).

Auto-Instrumentation

OTel Node.js provides modules that dynamically monkey-patch standard libraries (like http, express, pg, redis). Without modifying your application business logic, OTel automatically wraps functions to intercept requests and database calls, generating spans with execution times.

Trace Context Propagation

To link a request passing through multiple microservices, the Trace ID must be propagated. In synchronous languages like Java, context is tied to the Thread. In Node.js, asynchronous context is notoriously difficult to track across callbacks and promises.

Senior Signal: OTel in Node uses the async_hooks API (specifically AsyncLocalStorage) to maintain Trace Context implicitly across asynchronous barriers, ensuring every database call knows which HTTP request triggered it.

8. Dependency Injection (tsyringe / InversifyJS)

Unlike standard CommonJS modules where dependencies are hardcoded via require(), Dependency Injection (DI) passes dependencies into a class, usually via its constructor. This enables loose coupling and drastically simplifies unit testing (via mocking).

graph TD DI[DI Container (InversifyJS/tsyringe)] DI -- Injects --> U[UserController] DI -- Injects --> M[EmailService] DI -- Injects --> D[DatabaseRepository] U -. Depends on .-> M U -. Depends on .-> D

Decorators and DI Containers

Modern TS/Node DI relies heavily on Decorators. Classes are marked with @injectable(). The framework's DI Container registers all implementations.

Singleton vs Transient: The Container manages instance lifecycles. A Singleton dependency is created once and shared across the entire app (e.g., DB Connection Pool). A Transient dependency gets instantiated freshly every time it is injected.

DI vs Factory: A Factory manually instantiates objects and their dependencies hierarchically. A DI Container automates this resolution tree entirely via reflection metadata.
