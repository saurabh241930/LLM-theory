CDN Architecture & Cache Invalidation

Global Content Delivery & Managing Stale Data

1. Content Delivery Networks (CDNs)

A CDN is a geographically distributed network of proxy servers that cache static assets (HTML, CSS, JS, Images, Videos) closer to requesting end users.

2. Push vs Pull CDNs

Strategy How it works Best Use Cases
Pull CDN The Edge node is initially empty. When a user requests a file, the CDN pulls it from the Origin server, caches it, and serves future users from the cache. Sites with massive, infrequently changing libraries holding millions of distinct images. (E.g. Pinterest, Wikipedia). Highly automated.
Push CDN Engineers actively upload content to the CDN edge nodes before users ever request it. Small footprints, highly predictable content, or massive scheduled media drops (E.g. Netflix uploading a new season at midnight).

3. Cache Invalidation (The Hard Part)

“There are only two hard things in Computer Science: cache invalidation and naming things.”

When the Origin Server updates an image, how do the CDN nodes know not to serve the old one?

Time To Live (TTL)

You set a header (Cache-Control: max-age=3600). After 1 hour, the CDN considers the file stale, discards it, and pulls the fresh version from the origin on the next request.

Problem: For 59 minutes, users might see completely outdated profiles or broken CSS files.

Object Versioning (Cache Busting)

The universally accepted industry standard for infinite, safe caching.

Instead of updating style.css, your build pipeline generates a uniquely fingerprinted filename: style.a7b93c.css. Because the filename is unique to its hashed contents, it can be cached by the CDN forever (TTL 1 Year).

When you deploy a new version (style.b8x12.css), the HTML requests the new file name. The CDN naturally registers a Cache Miss and fetches the new file natively without requiring explicit invalidation commands.

Active Purge APIs

Calling an explicit API (e.g., Cloudflare Purge Cache) to forcefully delete a URL across all global edge nodes instantly. This is extremely expensive and computationally heavy, reserved mostly for urgent legal takedowns or security fixes.

4. Database Caching Strategies (Redis / Memcached)

Similar to CDN edge logic, backend engineers cache raw database results in memory keys (Redis).

Cache Aside (Lazy Loading):
1. APP checks Redis.
2. If hit, return. If miss, fetch from DB, save to Redis, then return.
Best for: General read-heavy apps.

Write-Through Cache:
1. APP writes the new data to the DB and exactly at the same time writes it to Redis.
Best for: Avoiding stale Data anomalies. The cache is perfectly consistent, but write performance takes a slight hit.
