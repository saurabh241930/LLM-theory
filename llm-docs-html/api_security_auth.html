Q34: What is HATEOAS and is it practical?

HATEOAS embeds links to related actions in API responses — theoretically self-documenting but rarely fully implemented in practice.

Core Points

  •  Response includes _links with available next actions: { order: {...}, _links: { cancel: '/orders/1/cancel',
  • pay: '/orders/1/pay' } }
  •  Clients can discover API dynamically without hardcoding URLs — reduces coupling
  •  Most teams implement a pragmatic subset: include key related resource URLs without full HATEOAS
  • compliance
flowchart LR API -- "Payload + Links array" --> Client Client -- "Follows link 'rel': 'next'" --> NextAPI["Next Resource Endpoint"]
Depth (Senior Signal):
  • HAL (Hypertext Application Language) and JSON:API are popular HATEOAS-inspired standards
  • GraphQL solves the related-resources problem differently — clients query exactly what they need
Trap / Watch Out: Full HATEOAS is rarely worth the implementation cost for internal APIs — focus on consistent URL design and good documentation instead

Q35: How do you design APIs for long-running operations?

Async pattern: return 202 Accepted with a job ID immediately; client polls status endpoint or receives webhook on completion.

Core Points

  •  POST /reports → 202 Accepted { job_id: 'abc123' }
  •  GET /jobs/abc123 → { status: 'processing', progress: 45 } or { status: 'complete', result_url: '...' }
  •  Webhook alternative: client registers callback URL; server POSTs result when ready
flowchart TD Client -- "POST /jobs (huge operation)" --> API API -- "202 Accepted + Location: /jobs/123" --> Client API -- "Enqueue Task" --> Worker["Background Worker"] Client -- "GET /jobs/123 (Polling)" --> API Worker -- "Updates Status" --> DB
Depth (Senior Signal):
  • Server-Sent Events (SSE) for streaming progress updates without polling overhead
  • Job TTL: clean up completed job records after 24-48hr to avoid unbounded storage growth
Trap / Watch Out: Never block the HTTP connection for >30 seconds — client and proxy timeouts make it unreliable; use async pattern instead

Q36: Explain GraphQL vs REST — when to choose which?

REST is simple and cacheable; GraphQL eliminates over/under-fetching and is powerful for complex, flexible data requirements.

Core Points

  •  REST: fixed endpoints, fixed response shapes — easy to cache, monitor, and understand
  •  GraphQL: single endpoint, client defines exact query — no over-fetching (mobile gets only needed
  • fields)
  •  Choose GraphQL when: multiple client types (mobile/web/third-party), complex nested queries, rapid
  • frontend iteration
flowchart LR Client -- "POST { user { id, name, posts { title } } }" --> GraphQL["GraphQL Server"] GraphQL -- "Resolver: User" --> DB1[("User DB")] GraphQL -- "Resolver: Posts" --> DB2[("Posts DB")] DB1 --> GraphQL DB2 --> GraphQL GraphQL -- "Single JSON Response" --> Client
Depth (Senior Signal):
  • GraphQL N+1 problem: naive resolvers fire N DB queries for N items — solve with DataLoader
  • (batching)
  • REST is better for public APIs — simpler to document, easier for external developers to consume
Trap / Watch Out: GraphQL is not always better — the N+1 problem, complex caching, and query depth attacks require careful handling

Q37: How do you handle file uploads in REST APIs?

Multipart form-data for small files direct to server; pre-signed S3 URLs for large files uploaded directly from client.

Core Points

  •  Multipart/form-data: standard for file + metadata in same request — works for files up to ~10MB
  •  Pre-signed URL: API generates time-limited S3 URL → client uploads directly to S3 — server never
  • handles file bytes
  •  Return file URL or ID in response after successful upload
flowchart TD Client -- "1. Request Upload URL" --> API API -- "2. Pre-signed S3 URL" --> Client Client -- "3. PUT multipart/form-data" --> S3[("S3 Bucket")] S3 -- "4. Event Trigger" --> Lambda["Processor"]
Depth (Senior Signal):
  • Chunked upload for large files: split client-side, upload chunks in parallel, server reassembles
  • Validate file type server-side via magic bytes, not just extension — extensions can be spoofed
Trap / Watch Out: Don't stream large files through your API server to S3 — double bandwidth cost and blocks threads. Pre-signed URL is the correct pattern.

Q38: What are best practices for API documentation?

OpenAPI (Swagger) spec as the source of truth — auto-generate docs, client SDKs, and mock servers from it.

Core Points

  •  OpenAPI 3.0 spec: documents endpoints, request/response schemas, auth methods, and error
  • codes
  •  Interactive docs (Swagger UI, Redoc) let developers test endpoints without writing code
  •  Include real-world examples in docs — not just schema, but actual request/response payloads
flowchart LR Code["Annotations/Code"] --> SwaggerGen["Swagger/OpenAPI Generator"] SwaggerGen --> JSON["openapi.json"] JSON --> UI["Swagger UI / Redoc"] UI --> Dev["Developer (Client)"]
Depth (Senior Signal):
  • Contract testing (Pact): consumers write tests against API contract — breaks if provider changes
  • break the contract
  • Changelog in docs: document every breaking and non-breaking change with version and date
Trap / Watch Out: Docs that lag behind implementation are worse than no docs — auto-generate from code annotations or spec-first development

Q39: How do you implement request tracing across microservices?

Inject a Trace ID at API Gateway entry; propagate via HTTP headers through all service calls; collect spans in Jaeger or Zipkin.

Core Points

  •  W3C Trace Context standard: traceparent header carries trace ID + span ID — propagate on every
  • outbound call
  •  Each service creates a child span with its own work duration — builds the full request tree
  •  Correlate logs, metrics, and traces by trace ID — enables root cause analysis across services
flowchart LR Client -- "Request" --> API_GW["API Gateway (Generates Trace-ID)"] API_GW -- "X-B3-TraceId: 123" --> ServA["Service A"] ServA -- "X-B3-TraceId: 123" --> ServB["Service B"] ServA -. "Send Span" .-> Jaeger["Jaeger/Zipkin"] ServB -. "Send Span" .-> Jaeger
Depth (Senior Signal):
  • OpenTelemetry (OTel): vendor-neutral SDK for traces + metrics + logs — export to Jaeger,
  • Datadog, or Grafana Tempo
  • Sampling: trace 100% in dev, 1-10% in prod — full tracing at scale is cost-prohibitive
Trap / Watch Out: Logging without trace IDs makes debugging distributed issues nearly impossible — always include trace ID in every log line

Q40: What is the difference between authentication and authorization?

Authentication verifies identity (who are you?); authorization verifies permission (what can you do?).

Core Points

  •  AuthN: validating credentials — JWT signature check, session lookup, OAuth token introspection
  •  AuthZ: checking permissions — RBAC (role-based), ABAC (attribute-based), or policy-based (OPA)
  •  Always do AuthN first, then AuthZ — no point checking permissions for an unverified identity
flowchart TD User["User"] --> AuthN{"Authentication"} AuthN -- "Verifies Identity (Who are you?)" --> IDP["IdP (Okta / Auth0)"] IDP --> AuthZ{"Authorization"} AuthZ -- "Checks Permissions (What can you do?)" --> Roles["RBAC/ABAC Policy"]
Depth (Senior Signal):
  • RBAC: user has role (admin, editor, viewer); roles have permissions — simple but inflexible for
  • complex rules
  • ABAC: permissions based on attributes (user department, resource owner, time of day) — more
  • flexible, more complex
  • OPA (Open Policy Agent): policy as code — externalize authorization rules, query from any service
Trap / Watch Out: Confusing 401 (unauthenticated — not logged in) with 403 (unauthorized — logged in but not permitted) is a very common mistake

Q41: How do you test REST APIs effectively?

Layer tests: unit tests for business logic, integration tests for API contracts, and e2e tests for critical flows.

Core Points

  •  Unit tests: test service/repository layer in isolation with mocked dependencies — fast, run on every
  • commit
  •  Integration tests: spin up real DB + app in Docker, hit API endpoints — validate full request/response
  • cycle
  •  Contract tests (Pact): ensure API changes don't break consumers — run in CI before deployment
flowchart LR Dev["Developer"] --> Unit["Unit Tests (Mocked DB)"] Unit --> Integration["Integration Tests (Testcontainers)"] Integration --> E2E["E2E Tests (Cypress/Postman)"] E2E --> CD["Deployment"]
Depth (Senior Signal):
  • Postman collections + Newman for automated API regression testing in CI pipeline
  • Mutation testing: inject intentional bugs to verify tests catch them — validates test quality, not just
  • coverage
Trap / Watch Out: 100% code coverage doesn't mean correct behavior — test behavior and edge cases, not just line coverage
