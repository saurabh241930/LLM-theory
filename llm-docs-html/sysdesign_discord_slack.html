System Design: Collaborative Platform (Discord / Slack)

Architecting Real-Time Messaging at 50M+ DAU Scale

1. Requirements & Scale Estimation

Functional Requirements

  • 1-on-1 Chats: Direct real-time messaging between two users.
  • Group Chats / Channels: Small groups (Slack) and massive servers (Discord - up to 500,000 members).
  • Presence: Online/Offline/Idle status indicators.
  • Message History: Infinite scrolling of historical messages.
  • Notifications: Push notifications for @mentions or offline messages.

Non-Functional Requirements

  • Low Latency: Messages must appear exactly as they are sent (< 200ms).
  • High Availability: The system cannot go down. Eventual consistency is perfectly acceptable over strict ACID for messages.
  • Massive Concurrency: Sustaining millions of persistent, open TCP connections simultaneously.

Back-of-the-Envelope Math (Scale Strategy)

User Base: 50 Million Daily Active Users (DAU)

Traffic (QPS): Assume 100 messages sent per user/day.
50M * 100 = 5 Billion messages/day.
5 Billion / 86400 seconds = ~60,000 Writes/sec average. At peak hours, this hits 300,000 Writes/sec.

Storage: 5B messages * ~150 Bytes (ID, timestamp, text, metadata) = 750 GB / Day. Roughly 270 TB / Year of pure text metadata dynamically generated.

Connections: We must continuously maintain 50 Million active WebSocket threads concurrently.

Edge Case Considerations to Tackle Later:

  • The Hot Channel Problem: What happens when 500,000 users are in the "Midjourney" server and 5,000 users all send a message within the exact same second? (The Fan-Out problem).
  • Connection Drops: Mobile client disconnecting in tunnels.
  • Out-of-Order Messages: Ensuring message sequence accuracy mathematically.

2. Step-by-Step Component Design

Step A: The Connection Manager (WebSocket Gateway)

Traditional HTTP request-response architecture completely fails for real-time chat natively. Users cannot "poll" the server every second (50M * 1 request/sec = 50M QPS, which crashes infrastructure instantly). We need WebSockets for a persistent, bi-directional TCP pipe.

We deploy heavily stripped-down Node.js, Go, or Erlang/Elixir nodes whose solely proprietary job natively is to hold exactly 1,000,000 idle TCP connections efficiently in RAM without doing heavy CPU crunching natively.

Step C: Storage & Presence

Database: Discord uses ScyllaDB (Wide-column NoSQL). Partition key is `channel_id`, sorting key is `message_id`. This allows O(1) reads for channel history.

Presence: WebSockets send heartbeats every 5s. Gateway updates Redis (`Key: User_ID | Value: Online | TTL: 15s`). If heartbeats stop, TTL expires automatically, marking them offline.
