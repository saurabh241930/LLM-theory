ColBERT & Late Interaction

When single-vector dense retrieval falls short, and the genius of MaxSim

The Flaw of Standard Embeddings (Early Interaction)

In standard RAG, you take a 1,000-word document and run it through OpenAI's text-embedding-3. The model squashes the entire semantic meaning of all 1,000 words into a single dense vector (e.g., [0.4, -0.1, 0.9, ...]).

This "Early Interaction" is computationally brilliant; you only store 1 vector per document. But it is semantically disastrous for complex queries. The nuance of exactly *how* a specific entity was mentioned in the document is destroyed and blended into the average meaning of the paragraph.

ColBERT: Contextualized Late Interaction

ColBERT (Contextualized Late Interaction over BERT) completely rewrites the rules. Instead of squashing the document into one vector, it saves one vector for every single token in the document.

If a document has 500 tokens, ColBERT saves a 500x128 matrix into the database. When the user asks a 10-token query, ColBERT creates a 10x128 matrix. The search operation is no longer comparing two single vectors; it is comparing a Matrix vs a Matrix.

The MaxSim Scoring Function

How do we compare two matrices efficiently without a massive Neural Network? MaxSim (Maximum Similarity).

The MaxSim Math

For every token in the Query, we find the single token in the Document that is most mathematically similar to it (the Maximum Cosine Similarity). We sum those maximums up.

Score = Σ Max(CosSim(Qi, Dj))

Why it's genius: The "Neo" query token instantly latches onto the "Neo" document token, ignoring the rest of the document. The "Who" token latches onto "the One". By delaying the interaction between the Query and Document until after they are both embedded into matrices, ColBERT achieves near Cross-Encoder accuracy at 100x the speed.

The Massive Drawback

Because ColBERT stores a vector for every single token instead of every chunk, the VRAM and database storage required absolutely explodes. A 100GB text corpus might require 1-2 Terabytes of database storage to hold the ColBERT multi-vectors. This is why it is often heavily compressed (like ColBERTv2's 2-bit residual compression) before deployment.
