Q73: Conceptually, how does an Event Loop work?

A single-threaded loop that offloads blocked I/O operations to the system kernel and continues processing other events.

When the kernel signals an I/O operation is complete, the loop picks up the callback and resumes processing.

Q74: Multi-threading vs Multi-processing for Backend Scaling.

Multi-threading for I/O efficiency (shared memory); Multi-processing for parallelism (bypassing GIL/Isolation).
Model Strengths Resource Cost
Threading Low overhead, shared memory state. Memory per thread is small.
Processing Parallel computation, crash isolation. High overhead (IPC required).

Q75: When do you use Celery vs BullMQ?

Use Celery for Python-heavy heavy-lift tasks (ML, PDF gen); use BullMQ (Redis-based) for high-performance Node.js async jobs.

Key trade-off: Celery is robust but can be slow to configure; BullMQ is extremely fast and utilizes Redis lua scripts for atomicity.

Q76: gRPC vs REST vs GraphQL.

gRPC for high-performance internal microservices; REST for external APIs; GraphQL for complex frontend data requirements.
  • gRPC: Uses Protocol Buffers (Binary) and HTTP/2. Lower latency.
  • REST: Standard JSON/HTTP. Highest compatibility.
  • GraphQL: Single endpoint. Client decides data shape.

Q77: What is the N+1 Query Problem and how to fix it?

Executing one query to get a list, then N additional queries for related data (e.g., getting users, then a query for each user's orders).
Fix: Use Eager Loading (JOINs or SELECT IN (...)) to fetch all related data in 1 or 2 queries instead of N+1.

Q78: Why do we need Connection Pooling?

Establishing a new DB connection is expensive (TCP handshake + Auth). Pools keep a set of "warm" connections open for reuse.
Trap: Exhausting the pool (too many concurrent requests) causes the app to hang while waiting for a connection to free up.
