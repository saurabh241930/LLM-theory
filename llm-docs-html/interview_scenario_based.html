1. Database Performance & Architecture (SQL / NoSQL)

Your Node.js API has a report endpoint that runs a 3-table JOIN returning 50,000 rows. It times out under load. P99 latency is 12 seconds. Fix it.

Node.jsSQLHard

Diagnosis: Pulling 50k rows in a single HTTP request exhausts server RAM (large JSON serialization in Node's single thread) and completely blocks the V8 event loop. A 12-second P99 usually denotes a Missing Index triggering sequential table scans across three tables during the JOIN operation.

The Fix:

  1. Database Level (Indexes): Run EXPLAIN ANALYZE on the query. Ensure the Foreign Keys forming the `JOIN` clauses have Composite Covering Indexes so the DB engine doesn't hit the disk.
  2. Backend Level (Pagination & Streaming): Stop fetching 50k rows at once. Implement Keyset Pagination (Cursor-based) yielding 100 rows per API call. If the user absolutely needs all 50k rows (e.g., a CSV Export), you must use Node.js Streams (`pg-query-stream`). You pipe the Database Read stream directly into the HTTP Response stream, keeping Node memory usage to ~30MB instead of crashing with an OutOfMemory error.
  3. Architecture Level (Materialized Views): If this report is analytical and drawn hourly, offload the computational cost. Precalculate the JOINs into a MATERIALIZED VIEW refreshed async nightly. The SELECT instantly becomes single-table O(1).

You cannot compute the balance on the fly on the Read path. You must pre-compute it on the Write path.

  • Use a Postgres Database Trigger: After an INSERT to `transactions`, the trigger instantly executes an UPDATE user_balances SET current_balance = current_balance + NEW.amount WHERE user_id = NEW.user_id.
  • Alternative (Message Queues): Have the transactional system publish a "TransactionCreated" event to Kafka. An async worker processes it and updates a Redis Key containing the running balance. The frontend page load simply executes an instantaneous O(1) Redis GET balance:123.

Two concurrent transactions both try to transfer money between the same two accounts at the same time. Walk me through exactly what can go wrong and how Postgres prevents it.

SQLHard

What Goes Wrong: The classic Lost Update anomaly or a Deadlock.
- Transaction 1 (T1) reads Account A ($100). Transaction 2 (T2) reads Account A ($100).
- T1 subtracts $50 and writes $50. T2 subtracts $20 and writes $80. T2 overwrites T1, and $50 simply vanishes into thin air.

The Fix: `SELECT FOR UPDATE` & Lock Ordering

graph TD T1[T1: Transfer A -> B] T2[T2: Transfer B -> A] T1 -- Locks A --> A[(Account A)] T2 -- Locks B --> B[(Account B)] T1 -. Waing for B .-x B T2 -. Waiting for A .-x A classDef error stroke:#f00,stroke-width:2px;

Postgres handles this via explicitly locking the rows on read: SELECT * FROM accounts WHERE id = A FOR UPDATE;. This prevents T2 from reading the row until T1 completes.

The Deadlock Trap: If T1 transfers A -> B, it locks A, then B. If T2 concurrently transfers B -> A, it locks B, then A. They both block each other permanently waiting for the other row to unlock.
Solution: The app logic MUST fundamentally sort the IDs before executing. Always lock the lower ID first. Both T1 and T2 structurally sort [A, B], subsequently locking A first. T2 hits the lock on A precisely and queues safely behind T1. Problem permanently averted.

  4. Finally, execute ALTER TABLE users ALTER COLUMN age SET DEFAULT 0; to secure all future rows securely.
