1. Modern LLM Serving Stacks

Serving LLMs in production is fundamentally different from standard web serving due to GPU memory constraints and varying completion lengths.

Stack Core Innovation Best For
vLLM PagedAttention (Operating System style memory management) Highest throughput demand.
NVIDIA Triton Model Analyzer & Multi-framework support Enterprise pipelines needing multi-model serving.
TGI (HuggingFace) Optimized for Transformers library integration Fast prototyping to production.

2. Key Performance Metrics (SLAs)

TTFT (Time to First Token)

The latency from request to seeing the first word. Crucial for perceived user experience.

ITL (Inter-Token Latency)

The time between subsequent tokens. High ITL leads to "choppy" reading experiences.

Tokens Per Second (TPS)

The overall speed. p99 TPS is a key reliability metric.

3. Serving Optimizations

Continuous Batching

Unlike static batching, continuous batching allows the server to insert new requests as soon as an existing request finishes a token, rather than waiting for the whole batch to complete.

graph LR A[Req 1: End] --> B[Insert Req 3] C[Req 2: Middle] --> D[Keep Processing]

Speculative Decoding

Use a smaller "draft" model to predict the next few tokens, then have the large "target" model verify them in a single forward pass. Can result in 2x-3x speedup.

4. Interview: Scaling to 1M QPS

Q: How do you serve 1 million queries per second for an LLM?

This is a trick question. 1M QPS for LLMs would require thousands of H100 GPU clusters. The answer lies in:

  • Aggressive Semantic Caching: Using Redis/Vector DB to cache common math/logic queries.
  • Router Orchestration: Routing simple tasks to 8B models and complex ones to 70B+ models.
  • Multi-Region Clusters: Distributing GPU load across data centers to bypass local GPU quotas.
