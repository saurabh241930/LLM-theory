Small-to-Big Retrieval: Optimizing both Search and Generation

The Chunking Paradox

Standard RAG forces you into a brutal trade-off when deciding your chunk size (e.g., 200 tokens vs 1000 tokens):

  • Tiny chunks (200 tokens): Phenomenal for vector search. The embedding perfectly captures the dense meaning of those 3 sentences. But terrible for generation; the LLM lacks the surrounding context to give a full answer.
  • Large chunks (1000 tokens): Perfect for generation; the LLM has all the context. But terrible for vector search. A 1000-word paragraph has too much noise, watering down the embedding and destroying retrieval accuracy.

The Solution: Small-to-Big Retrieval

We decouple the retrieval chunk from the generation chunk. We slice a massive document into a hierarchy: 1 large "Parent" chunk, split into multiple tiny "Child" chunks.

Under the Hood

In a database like Chroma or Qdrant, you only embed and search against the tiny Child chunks (e.g., 100 tokens). They yield highly precise cosine similarity search results.

However, when a Child chunk ranks #1, you do not send the Child text to the LLM. Every Child chunk is tagged with a parent_id. The orchestrator (Langchain/LlamaIndex) uses that ID to fetch the massive Parent Document from a standard Key-Value store (or document store) and injects the entire Parent into the prompt.

The ultimate result: The ultra-high precision of tiny-chunk vector search, combined with the incredibly rich context of large-chunk generation.
