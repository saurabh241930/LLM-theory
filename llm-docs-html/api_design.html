Q27: How do you handle API versioning and backward compatibility?

URI versioning (/v1/, /v2/) for major breaking changes; header versioning for fine-grained control; additive changes are non-breaking.

Core Points

  •  URI versioning: /api/v1/users — explicit, easy to route, cache-friendly; most common approach
  •  Non-breaking changes: adding new fields, new endpoints, new optional query params — these don't
  • need version bump
  •  Breaking changes: removing fields, changing field types, changing URL structure — always version
  • bump
flowchart TD Client --> Router["API Router"] Router -- "/v1/users" --> Handler1["V1 Handler (Legacy)"] Router -- "/v2/users" --> Handler2["V2 Handler (Modern)"]
Depth (Senior Signal):
  • Sunset header: tell clients when old version is being deprecated — give minimum 6 months notice
  • API changelog + semantic versioning in documentation — consumers need to know what changed
  • and why
Trap / Watch Out: Don't version every minor change — only version when you break existing consumers. Over-versioning creates maintenance hell.

Q28: JWT vs OAuth2 vs API Keys — when to use which?

API Keys for service-to-service, JWT for stateless user sessions, OAuth2 for delegated third-party access.

Core Points

  •  API Key: simple, long-lived, no expiry by default — good for server-to-server where key is stored
  • securely
  •  JWT: self-contained token with claims, expiry, and signature — no DB lookup on every request
  • (stateless)
  •  OAuth2: authorization framework for third-party delegation — 'Login with Google' is OAuth2 + OIDC
flowchart LR JWT["JWT: Stateless, self-contained"] OAuth["OAuth2: Scoped authorization delegation"] APIKeys["API Keys: Server-to-server auth"] JWT --- APIKeys
Depth (Senior Signal):
  • JWT weakness: can't invalidate before expiry — use short TTL (15min) + refresh token pattern
  • PKCE (OAuth2 extension): for public clients (mobile/SPA) where client secret can't be stored safely
Trap / Watch Out: JWT is not encryption — the payload is base64 encoded, not encrypted. Don't put sensitive data in JWT claims.

Q29: How do you design a RESTful resource hierarchy?

Model resources as nouns, use HTTP verbs for actions, keep hierarchy max 2-3 levels deep.

Core Points

  •  Nouns not verbs: /orders not /getOrders; /orders/{id}/cancel not /cancelOrder
  •  Hierarchy: /projects/{id}/tasks/{id} — nested when child can't exist without parent
  •  Flat when possible: /tasks?project_id=X is often better than deep nesting — easier to cache and
  • document
flowchart LR Root["/users"] --> UserID["/users/{id}"] UserID --> Orders["/users/{id}/orders"] Orders --> OrderID["/users/{id}/orders/{order_id}"]
Depth (Senior Signal):
  • Sub-resources for actions that don't map to CRUD: POST /orders/{id}/cancel, POST
  • /accounts/{id}/freeze
  • Use plural nouns consistently: /users not /user, /products not /product
Trap / Watch Out: Avoid going deeper than 3 levels: /companies/{id}/departments/{id}/employees/{id}/tasks — becomes unwieldy to maintain

Q30: How do you implement pagination in REST APIs?

Cursor-based pagination for large/real-time datasets; offset-based for simple use cases with known total counts.

Core Points

  •  Offset pagination: ?page=2&limit=20 — simple but slow on high offsets (DB scans all skipped rows)
  •  Cursor pagination: ?after=cursor_token — cursor encodes last seen ID/timestamp; O(1) regardless
  • of page depth
  •  Return next_cursor, has_more, and total_count in response envelope
flowchart LR Client -- "GET /items?limit=50&offset=100" --> API API -- "Select ... Offset 100 Limit 50" --> DB DB --> API API -- "Items + Metadata (Next/Prev links)" --> Client
Depth (Senior Signal):
  • Keyset pagination: WHERE id > last_id ORDER BY id LIMIT 20 — uses index efficiently
  • Cursor should be opaque to client — base64 encode so clients don't try to manipulate it
Trap / Watch Out: Offset pagination breaks on real-time feeds — new items shift offsets, causing duplicates or skips. Use cursor there.

Q31: How do you handle API error responses consistently?

Standardized error envelope with HTTP status code, machine-readable error code, human message, and request trace ID.

Core Points

  •  Error envelope: { error: { code: 'VALIDATION_FAILED', message: '...', details: [...], trace_id: '...' } }
  •  HTTP status codes: 400 (bad input), 401 (unauthenticated), 403 (unauthorized), 404 (not found), 429
  • (rate limited), 500 (server error)
  •  Machine-readable codes (INSUFFICIENT_FUNDS, USER_NOT_FOUND) allow clients to handle
  • programmatically
flowchart TD Error["Exception Thrown"] --> Middleware["Global Error Handler"] Middleware --> Log["Log to DataDog/Sentry"] Middleware --> Format["Format RFC 7807 Standard Error"] Format --> Client["4xx / 5xx Standard JSON Schema"]
Depth (Senior Signal):
  • RFC 7807 (Problem Details) is a standardized format for HTTP error responses — worth adopting
  • Include docs URL in error response: 'documentation':
  • 'https://api.example.com/errors/VALIDATION_FAILED'
Trap / Watch Out: Never return 200 OK with an error in the body — it breaks HTTP semantics and makes monitoring impossible

Q32: Explain content negotiation in REST APIs

Client declares preferred response format via Accept header; server responds with matched Content-Type or 406.

Core Points

  •  Accept: application/json → server returns JSON; Accept: application/xml → XML
  •  Content-Type header on request body tells server how to parse incoming data
  •  406 Not Acceptable if server can't satisfy client's Accept header
flowchart TD Client -- "Accept: application/xml" --> API API -- "Parses Header" --> Logic["Business Logic"] Logic --> Formatter{"Content Negotiator"} Formatter -- "application/xml" --> XML["XML Payload"] Formatter -- "application/json" --> JSON["JSON Payload"]
Depth (Senior Signal):
  • API versioning via Accept header: Accept: application/vnd.myapi.v2+json — cleaner URL but harder
  • to test in browser
  • Compression: Accept-Encoding: gzip — server compresses response, saves bandwidth
Trap / Watch Out: Don't ignore the Accept header and always return JSON — some clients (B2B integrations) legitimately need XML

Q33: How do you secure REST APIs against common attacks?

Defense in depth: input validation, rate limiting, auth on every endpoint, HTTPS, and OWASP API Security Top 10 awareness.

Core Points

  •  Always validate and sanitize input — SQL injection, XSS, and injection attacks all come from
  • untrusted input
  •  Rate limiting per client IP and API key — prevents brute force and DDoS
  •  Auth on every endpoint — no security through obscurity; even 'internal' endpoints need auth
flowchart TD Web["Internet"] --> WAF["WAF (AWS WAF / Cloudflare)"] WAF --> RL["Rate Limiter (Redis)"] RL --> Auth["AuthZ & AuthN Gateway"] Auth --> API["Internal Microservice API"]
Depth (Senior Signal):
  • OWASP API Security Top 10: Broken Object Level Authorization (BOLA) is #1 — always verify
  • ownership of resource
  • JWT algorithm confusion attack: explicitly specify alg: RS256 in verification, never accept alg: none
Trap / Watch Out: BOLA (returning other users' data by manipulating IDs) is the most common API vulnerability — always check ownership, not just authentication

Q83: How do you implement Idempotency Keys in Payment APIs?

The client sends a unique Idempotency-Key header; the server stores the first response in Redis with that key and returns it for subsequent duplicate requests.
Senior Tip: Always use a distributed lock (like Redlock) during the first processing attempt to prevent "Double Spend" if two identical requests hit the server at the exact same millisecond.

Trap: Setting Access-Control-Allow-Origin: * in production is a massive security risk. It allows any site to make requests to your API while carrying user cookies.

Senior Approach: Explicitly whitelist Origins in an environment variable. For Preflight requests (OPTIONS), ensure your server returns 204 No Content without executing expensive auth logic.
