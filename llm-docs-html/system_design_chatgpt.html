System Design: ChatGPT Clone

Personal Assistant Chatbot for 10M Active Users

Q83: Design a ChatGPT Clone (Personal Assistant Chatbot) for 10M Users

A massively scalable streaming architecture utilizing SSE/WebSockets for realtime token delivery, conversational memory management via NoSQL, and a heavily optimized LLM Inference cluster.

When giving this interview, treat the problem not just as a standard web application, but as a specialized ML Serving pipeline. The main challenges are Streaming Latency (Time-To-First-Token), Context Window Management, and GPU Compute Cost.

2. Key Components & Approach

A. API Gateway & Connection Management
  • Protocol: Use Server-Sent Events (SSE) for the core LLM output. Unlike WebSockets (which are bi-directional), SSE is incredibly efficient for uni-directional streaming from server to client, perfectly matching the token-by-token generation of LLMs.
  • Rate Limiting: Use a Token Bucket algorithm via Redis to prevent abusive usage (GPUs are expensive). Limit both "requests per minute" and "input tokens per request".
B. Chat Session & Memory Management
  • An LLM is inherently stateless. True "conversational memory" requires the backend to fetch the previous chat history and append it to the current user prompt before sending it to the GPU.
  • Storage: Use a NoSQL database like Cassandra/ScyllaDB for chat history. It handles massive write-heavy workloads perfectly since every single turn of a conversation is an INSERT.
  • Cache: Keep the active conversation window in Redis. When a user sends a message, fetch the last N messages from Redis (e.g., last 4000 tokens) instantly instead of querying disk.
C. The Inference Cluster (The Bottleneck)

Standard load balancers (Round Robin) are terrible for LLMs because request lengths vary wildly (a "Hello" vs "Write a book").

  • Continuous Batching: Use an engine like vLLM. Instead of waiting for a batch of 16 queries to all finish, vLLM dynamically injects new queries into the GPU as soon as older ones complete.
  • PagedAttention: Explain to the interviewer that memory fragmentation in the KV Cache (the GPU memory storing the context of the conversation) limits batch size. Mentioning PagedAttention (memory management like virtual memory in OS) is a massive senior signal.
  • Orchestrator: Routes requests to GPUs based on KV Cache Locality. If GPU-2 already holds the context for User A's conversation in memory, route User A's next message to GPU-2 to avoid re-computing the prompt.

Depth (Senior Signal) - Edge Cases & Optimizations:
  • Context Overflow: What if the conversation exceeds 128k tokens? Instead of dropping old messages, run a background summarization agent. "Turn the first 50 messages into a 500-token summary", keeping the core facts but saving token space.
  • Cold Starts vs Hot Instances: Generating the first token (Time-To-First-Token / Prefill Phase) is compute-heavy. Generating subsequent tokens (Decode Phase) is memory-bandwidth heavy. Split the cluster—have dedicated GPUs that ONLY process the initial prompt, and transfer the intermediate state (KV Cache) to Decode GPUs that stream the tokens back.
Trap / Watch Out: Don't design it like a standard CRUD app. If the interviewer asks "How do you scale the database?", clarify that the DB isn't the bottleneck—the GPU memory (KV cache) is. Storing text is cheap; executing transformer math is expensive. Focus heavily on streaming protocols and GPU utilization.
