Pinecone, pgvector, Cost Tradeoffs, and Hybrid Search

1. Architecting the Choice: Serverless vs Native

The first decision in RAG production is where your vectors live. It's a battle between Specialization and Consolidation.

Feature Pinecone (Serverless) pgvector (Integrated)
Scale Hyperscale (Billions of vectors) High (Millions of vectors)
Query Latency Low (Highly optimized) Medium (Shared DB resources)
Metadata Pre-filter / Post-filter limited Infinite (full SQL power)
Operational Cost Pay-per-query/storage Static (Your existing DB cost)

2. Hybrid Search: The Industry Standard

90% of failures in RAG are due to "Keyword Mismatch." Hybrid Search combines **Dense** (Vectors) and **Sparse** (BM25) results using **RRF (Reciprocal Rank Fusion)**.

Python (Pinecone Hybrid)
from pinecone import Pinecone

pc = Pinecone(api_key="...")
index = pc.Index("customer-support")

# Upsert with sparse and dense values
index.upsert(
    vectors=[
        {
          "id": "doc1", 
          "values": [0.1, 0.2, ...], 
          "sparse_values": {"indices": [1, 5], "values": [0.3, 0.8]}
        }
    ]
)

3. Metadata Filtering: The ACL Gap

In Enterprise RAG, you can't just retrieve "Similar Docs." You must retrieve "Similar Docs that USER_A has permission to see."

Pre-Filtering vs Post-Filtering

Post-Filtering: Retrieve top 100, then filter by permission. RISK: If user only has access to 2 docs, they might get 0 results.

Pre-Filtering: The Vector DB applies the filter *during* the graph traversal. pgvector excels here because it uses your actual users and permissions tables.

4. Multi-Tenancy & Namespaces

Never mix customer data in the same "flat" index. Use Namespaces (Pinecone) or Partitioned Tables (Postgres) to create hard silos.

// JS Namespace Query (LangChain.js)
const docs = await vectorStore.similaritySearch(
  "How to reset password?", 
  3, 
  { namespace: "client_id_99" }
);
