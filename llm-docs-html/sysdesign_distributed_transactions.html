Distributed Transactions

Two-Phase Commit vs The Saga Pattern

1. The Microservices Nightmare

In a monolith, transferring money from one account to another uses a strict ACID relational Database Transaction. If either the debit or the credit code fails, the database natively rolls back everything synchronously.

In a microservices architecture, the Order Service has an isolated database, and the Payment Service has an isolated database. If the Order Service deducts inventory, and a subsequent HTTP call to the Payment Service encounters a 500 Server Error to charge the card, your inventory is definitively corrupted. How do you roll back changes across separate autonomous databases?

2. The Two-Phase Commit (2PC)

A strict, synchronous protocol ensuring ACID properties distributed over the network via a central Coordinator.

Phase 1: Prepare

The Coordinator asks all participating nodes if they are fully prepared to commit the transaction. All nodes lock their local resources (e.g., locking the row in MySQL) and reply "Yes" or "No".

Phase 2: Commit

If ALL nodes voted "Yes", the Coordinator sends a global "Commit" command. If a single node voted "No" (or timed out), the Coordinator sends a global "Abort" command, and all nodes release their locks immediately.

Why is 2PC inherently bad? It is synonymous with performance suicide. It creates network-level blocking locks across multiple disjoint databases simultaneously. If the central Coordinator node crashes halfway through Phase 2, the participating nodes helplessly freeze indefinitely, permanently holding database locks that degrade the entire site until human intervention.

3. The Saga Pattern

Sagas abandon strict ACID consistency in favor of distributed Base (Basically Available, Soft State, Eventual Consistency). Instead of blocking distributed locks, a Saga splits a large transaction into a sequence of small, localized database transactions.

Compensating Transactions

Since each local step genuinely commits its data permanently to its localized database immediately, you cannot simply "rollback" if a later step fails.

If Step 3 logically fails, the system must sequentially execute Compensating Transactions for Step 2 and Step 1. A Compensating Transaction is explicit application code written to conceptually negate the original action (e.g., if Step 1 was chargeCard($50), the Compensating step is distinctly refundCard($50).

Sagas are predominantly driven using one of two core models:

Choreography Saga (Event-Driven)

No central brain. Services publish Domain Events onto an Event Stream (Kafka). Other services listen and react.

  • OrderService: Commits Order (Pending). Publishes OrderCreated Event.
  • PaymentService: Listens to Event. Commits Charge. Publishes PaymentSucceeded Event.
  • InventoryService: Listens to Event. Commits Deduct Stock.
Pros: Totally decoupled. No single point of failure. Cons: Complex to mentally trace the logical flow or debug effectively across 10 microservices interacting purely via scattered event listeners.

Orchestration Saga (Command-Driven)

A central "Orchestrator" service explicitly tells everyone what to do. It acts like an air-traffic controller holding the state machine structure.

  • Orchestrator: Sends Create Order Command to Order Service. Waits for reply.
  • Orchestrator: Sends Charge Card Command to Payment Service. Waits for reply.
  • If Payment fails, the Orchestrator: actively fires a Cancel Order Compensating Command back to the Order Service to trigger the conceptual rollback.
Pros: The entire business flow is centralized, readable, and perfectly traceable. Cons: Creates a localized single point of failure/bottleneck within the orchestrator node itself.
