RAG Pipeline at Scale

Design a production RAG system that handles 10,000 queries/day. Walk me through every component.

Answer: 10,000 queries/day is mathematically about ~0.11 queries per second (QPS) on average, but we must architect for peak loads (e.g., ~5-10 QPS). Here is the component walkthrough:

  1. API Gateway (Kong/Nginx): Handles authentication, rate limiting, and SSL termination. Routes requests to the backend.
  2. Backend Service (Node.js/FastAPI): Validates the incoming payload, hashes the query, and orchestrates the AI workflow.
  3. Semantic Cache (Redis): Checks if the user's query intent matches a previously cached query (using dense vectors or LLM-based caching). If Cache Hit -> Return immediately.
  4. Query Rewriter/Router: An LLM or Semantic Router decides if this query needs the Vector DB. It might expand the query (e.g., "how much?" -> "how much does auto insurance cost?").
  5. Vector Database (Pinecone/Milvus): Stores document embeddings. Performs the ANN (Approximate Nearest Neighbor) search to retrieve the top-K chunks.
  6. Reranker (Cross-Encoder): Re-scores the top-K chunks retrieved to ensure extremely high precision context.
  7. LLM Generation (OpenAI/Anthropic): The backend builds the final prompt encapsulating the reranked context and user query. Calls the LLM API.
  8. Telemetry & Observability (Datadog/LangSmith): Logs latency, token usage, and the retrieved context ID for future evaluation/debugging.

How do you handle document updates in a RAG system? If a source document changes, how do you re-index without downtime?

Answer: We handle updates using deterministic Document IDs and a decoupled asynchronous pipeline:

  • Hashing: When a document is ingested, we generate a hash (e.g., SHA-256) of its contents. If the hash matches the database, we skip processing.
  • Zero Downtime Re-indexing: When a change is detected, we do not overwrite right away. Instead, we chunk and embed the new document in an async background queue (BullMQ). We assign a `version_id` to these new chunks.
  • Atomic Swap/Delete: Once safely embedded into the Vector DB, we execute a metadata delete command `DELETE FROM vectordb WHERE document_id = X AND version_id = OLD_VERSION`. This swaps the old data for the new data atomically, ensuring read queries during the process are never interrupted or left with partial data.

How would you chunk documents for a legal/medical domain vs a general FAQ? What chunking strategy and why?

Answer: Chunking is incredibly domain-dependent:

  • General FAQ: Fixed-size chunking (e.g., 500 tokens with 50-token overlap) works fine. FAQs belong to single-turn concepts, making naive algorithms adequate.
  • Legal/Medical: Fixed chunks frequently shear context. We use Structural Chunking. For legal, we chunk by specific markdown headers (e.g., `## Section 1`, `Clause A`) using tools like LangChain's Markdown text splitter. For medical documents where context spans multiple pages (e.g., patient history leading to diagnosis), we use Parent-Child (Small-to-Big) RAG. We retrieve smaller, highly precise chunks (sentences), but inject the adjoining broader parent chunk (the whole paragraph/page) into the LLM.

Your RAG system is returning irrelevant answers. How do you debug it systematically?

Answer: I isolate the pipeline into three primary failure modes:

  1. Is it a Retrieval Failure? (Check Vector DB): I dump the prompt trace (via LangSmith) and look at the retrieved context. If the requested answer is not in the context, the vector pipeline failed. Fix: Adjust embedding model, enable hybrid search, add a reranker step, or rewrite the user's query.
  2. Is it a Grounding/LLM Failure? (Check Prompt/Model): If the answer is in the retrieved context but the LLM hallucinates or ignores it, the LLM failed. Fix: Adjust the prompt to be stricter ("Answer ONLY from the text provided"), lower the temperature to 0, or switch to a more capable model.
  3. Is it a Semantic Failure? (Check Chunking): If the context contains the answer but lacks the surrounding meaning (e.g., a pronoun like "He did it" without knowing who "He" is), chunk sizes are too small. Fix: Increase overlap or use Parent-Child indexing.

How do you handle multi-language documents in a RAG pipeline?

Answer: The most robust approach relies on high-quality multilingual models rather than naive translation:

  • Ingestion: Embed the documents in their native languages using a model trained on multi-language data (e.g., `text-embedding-3-large` or `multilingual-e5`). Store them as-is.
  • Retrieval: When a user queries in Language A, the multilingual embedding model inherently maps it to a similar semantic vector space as Document B (written in Language B). Dense search natively crosses the language barrier without an explicit translation step.
  • Generation: Retrieve the native chunks, inject them into the LLM, and prompt: "Using the provided context, answer the question in the language the query was asked in." LLMs (like GPT-4) are excellent at cross-lingual synthesis. Note: For Keyword search (BM25), you must translate the query to the document's language first.

Design a hybrid search system combining keyword (BM25) and semantic (vector) search. How do you merge and rerank results?

Answer: Hybrid search leverages the strengths of both dense (meaning) and sparse (exact match) search.

  1. Dual Path: The user query is sent simultaneously to a Vector DB (dense vectors) and an Inverted Index like Elasticsearch (BM25 sparse vectors for keywords/acronyms).
  2. Retrieval: We pull the top `K=50` from both sources.
  3. Merging via RRF: Given that vector scores (cosine) and BM25 scores (TF-IDF variants) are on completely different mathematical scales, we cannot simply add them. We use Reciprocal Rank Fusion (RRF). This ranks documents based on their positional rank in both lists: `Score = 1 / (k + rank)`.
  4. Reranking: We take the top merged 15 results and pass them through a Cross-Encoder (e.g., `bge-reranker-large`). The Cross-Encoder evaluates the exact relationship between the query and each chunk simultaneously, yielding the absolute best top 5 chunks for the LLM.

Agentic RAG Architectures

What is Agentic RAG, and how does it differ from Standard (Naive) RAG?

Answer: Standard RAG is a linear, one-pass pipeline: Query -> Embed -> Retrieve -> Generate. It fails if the retrieval is poor or the query is complex.

Agentic RAG transforms the pipeline into an iterative, reasoning-based loop. An LLM "agent" orchestrates the process, deciding whether it has enough information, if it needs to re-query, or if it should use different tools. It introduces autonomy and self-correction into the retrieval process.

Describe the 4 main Agentic RAG patterns and when to apply each.

Pattern Mechanism Best Use Case
Router RAG An LLM router selects the best data source (e.g., Vector DB for docs, SQL for numbers, Search API for news). Multi-Silo Data: When information is spread across different types of databases or APIs.
Corrective RAG (CRAG) A self-grader evaluates retrieval relevance. If "Irrelevant", it triggers a fallback (e.g., Web Search) to find better context. Accuracy-Critical: When the Vector DB might not contain the answer, and you need a "safety net" to prevent hallucinations.
Self-RAG / Reflection The model generates segments and "critiques" them using reflection tokens (Is it relevant? Is it supported by facts? Is it useful?). Complex Synthesized Answers: When the answer requires high factual density and strict adherence to retrieved context.
Multi-Hop / Iterative Breaks a complex query into sub-steps. It retrieves info for Step 1, uses that to formulate Step 2, and so on. Comparative Reasoning: "How does the profit margin of Product A compare to the industry average in 2023?" (Requires two separate lookups).

How do you implement a "Self-Correction" loop in an Agentic RAG system?

Answer: We implement a Multi-agent or State-Machine flow (often using LangGraph):

  1. Retrieve: Fetch top-K documents.
  2. Grade: A separate "Grader LLM" (or prompt) scores each document: [Relevant, Partially Relevant, Irrelevant].
  3. Decision Branch:
    • If all are Relevant -> Proceed to Generation.
    • If some are Irrelevant -> Trigger Query Transformation (rewrite the user query to be more specific) and re-retrieve.
    • If all are Irrelevant -> Fallback to a secondary tool (e.g., Google Search API).
  4. Generate & Verify: After generation, a final check ensures the answer is grounded in the context.

When would you choose a "Plan-and-Execute" Agent for RAG over a simple ReAct Agent?

Answer:

  • ReAct Agent: Thinks and acts one step at a time. It's great for short-tail reasoning or when the next step depends entirely on the previous tool's output. However, it can lose track of the original goal in long sequences.
  • Plan-and-Execute: First creates a multi-step "To-Do" list and then executes each step. It's superior for long-form, complex research tasks because it maintains a high-level roadmap, reducing the risk of the agent getting "distracted" or caught in infinite loops.

LLM in Production

How do you reduce LLM API costs in a high-traffic production system?

Answer: Cost reduction is a multi-layered approach:

  • Semantic Caching: Cache frequent queries using Redis so identical/highly-similar consecutive queries bypass the LLM entirely API.
  • Model Routing algorithms: Not all queries need GPT-4. We use an LLM-based router or semantic router to direct simple summarization queries to cheaper/faster models (GPT-4o-mini / Haiku) and route complex reasoning tasks to OpenAI Opus/Pro tier.
  • Prompt Optimization: Exclude irrelevant chat history (e.g., truncate memory to the last 4 turns) and use smaller context limits to drastically reduce input tokens.
  • Fine-Tuning: For hyper-specific repetitive tasks (e.g., classifying support tickets), we fine-tune a smaller open-weights model (like Llama-3 8B) which costs fractions of a cent compared to leading proprietary models.

When would you cache LLM responses and how? What are the risks?

Answer:

When: We cache when there is an uneven distribution of queries (a "fat head"), like generic FAQs ("What's the refund policy?"), or expensive generation tasks that don't change frequently.

How: We embed the incoming user query and check a Redis Vector Cache. If the cosine similarity matches an existing key with `>0.95` confidence, we return the cached string.

Risks: 1) Context Contamination: A user asking "What is my balance?" should never be cached globally; caching must respect `tenant_id` or `user_id`. 2) Staleness: If the underlying RAG document changes, the cache serving the old policy will persist unless explicitly invalidated via event-driven mechanisms. 3) Security: PII injection into shared caching pools.

How do you handle LLM rate limits gracefully in a backend system?

Answer: You must protect the API layer to prevent cascading downstream failures:

  • Exponential Backoff + Jitter: We configure the LLM client (e.g., LiteLLM/LangChain) with backoff. If we get a 429 response, we wait 1s, 2s, 4s, 8s plus a random sub-second jitter to prevent the "thundering herd" problem pressing the API all at once.
  • Async Queues: For long-running batch jobs, push LLM generation into a BullMQ queue. If rate limited, the worker pauses processing for the exact seconds specified in the `Retry-After` HTTP header.
  • UX Grace: For real-time chats, we stream responses. If delayed, the UI shows intermediate loader states ("Thinking...", "Analyzing documents..."). If an absolute timeout hits, we fall back to a predefined generic message instructing the user to try again.

Design a fallback strategy when your primary LLM provider (OpenAI/Anthropic) goes down.

Answer: An enterprise system uses an abstraction proxy (like LiteLLM or an internal Provider/Adapter pattern):

  1. The system attempts `gpt-4o`.
  2. If a 5xx error or prolonged Timeout is caught, the orchestrator triggers the Fallback Policy.
  3. The orchestrator dynamically re-maps the prompt to Anthropic Claude 3.5 Sonnet payload format and issues the request.
  4. If that fails, it falls back to a self-hosted `vLLM` container running Llama-3 in a regional VPC.
  5. To prevent thrashing, we implement a Circuit Breaker pattern. If OpenAI fails 5 times in a row, the circuit "opens" and routes all traffic to the fallback provider immediately for the next 5 minutes before attempting to "half-open" and test the primary provider again.

How do you version prompts in production? What happens when a prompt change breaks downstream behavior?

Answer: Prompts are volatile code. We treat them exactly like source code:

  • Storage: Prompts are stored in Git (e.g., generic JSON or `.yaml` files) decoupled from application logic, or via a Prompt Management Dashboard (LangSmith/Vellum).
  • Versioning: Every modification creates a semantic version (e.g., `prompt_v1.2.0`).
  • Prevention: Before a prompt is merged, it must pass an automated CI/CD evaluation pipeline. It checks the prompt against a "Golden Dataset" of 50-100 regression queries. If accuracy drops below an 85% threshold, the PR fails.
  • Rollback: If a prompt change sneaks into production and breaks down-stream systems (e.g., outputs malformed JSON), we can instantly rollback to the previous Git commit/version tag via the environment variables to restore operation without refactoring application code.

How do you stream LLM responses to the frontend efficiently?

Answer: To minimize Time-To-First-Token (TTFT), we use Server-Sent Events (SSE) or WebSockets.

  1. The client opens a persistent SSE connection (`text/event-stream`).
  2. The Node.js/Python backend invokes the LLM API with `stream=True`.
  3. As the provider sends byte chunks back to the server, the backend iterates over the async generator and instantly flushes these tokens directly to the SSE stream.
  4. The React/Vue frontend consumes the SSE endpoint using libraries like `@microsoft/fetch-event-source` and reactively updates state, appending text to the user's view word-by-word.

Hallucination & Reliability

How do you detect and mitigate hallucinations in a production RAG system?

Answer:

Mitigation: 1) Stricter prompting: "Answer only using the provided text. Cite your source." 2) Top-K tuning: Provide highly relevant chunks (via reranking) to minimize noise taking up context. 3) Reduce Temperature to 0.

Detection: We use an LLM-as-a-judge system running asynchronously on logged queries. We parse the generated answer and ask an evaluator LLM: "Does the answer perfectly align with this source context?" If the score is low, we flag it in a dashboard for human review.

What is faithfulness vs answer relevance vs context recall — and how do you measure each?

Answer: These are core RAG metrics (often measured via RAGAS):

  • Faithfulness (Hallucination metric): Checks if the generated answer is entirely entailed by the retrieved context. (Measured by LLM-as-a-judge evaluating premise vs hypothesis).
  • Answer Relevance: Checks if the generated answer directly addresses the user's initial query, avoiding evasive or tangential answers. (Measured by checking semantic similarity between generated answers and the question intent).
  • Context Recall: Checks if the retriever actually fetched all the necessary information required to answer the query, validating search quality. (Measured against an annotated ground-truth answer).

How would you build an automated evaluation pipeline for your LLM outputs?

Answer:

  1. Create a Golden Dataset: 100 manually curated rows of `[Query, Expected Context, Expected Output]`.
  2. Integrate a framework like RAGAS or TruEra into the CI/CD pipeline (GitHub Actions).
  3. On pull request (e.g., when a chunk size changes or a prompt is updated), a script runs the entire dataset through the dev environment.
  4. Calculate aggregate scores for Faithfulness, Relevance, and BLEU/ROUGE (if absolute string match is needed).
  5. If the new pipeline scores `< 0.9` (or substantially lower than the previous baseline), the pipeline halts and blocks the merge.

A customer reports the AI gave wrong information. Walk me through your incident response.

Answer:

  1. Replicate & Isolate: Locate the specific query via a unique `trace_id` in our observability tool (LangSmith). Re-run the exact query in staging.
  2. Debug Pipeline: Did the Vector DB retrieve the wrong chunk? (Context Recall issue). Did the LLM ignore the correct chunk and make things up? (Faithfulness issue).
  3. Patch: If chunking failed, improve metadata or reranking. If the LLM failed, adjust the system prompt to explicitly restrict that hallucination pattern.
  4. Regression Testing: Add the customer's exact query to our Golden Dataset to ensure this particular edge-case never breaks in future updates.
  5. Deploy & Communicate: Push the patch and inform the customer, showing the system is adaptable and improving.

How do you ground LLM responses strictly to retrieved context and prevent it from using parametric knowledge?

Answer: This requires aggressive prompt engineering and parameter tuning:

  • Parameters: Set Temperature to 0.0 and Top_P to a low threshold.
  • Prompting Schema:
    You are a strict extraction system. NEVER use prior knowledge. Answer ONLY using the text below. If the text does not contain the answer, your MUST reply "Information not found in context".
  • Chain of Thought logic: Enforce the model to extract quotes first before answering: `Step 1. Quote the exact text confirming the answer. Step 2. Summarize the quoted text.` This forces grounding validation before generation.

Multi-Tenant Architecture

You're building a GenAI SaaS with 500 tenants. How do you isolate their data in a shared vector DB?

Answer: We use Metadata Filtering, an industry standard for logical partition isolation without the infrastructure cost of 500 separate databases.

When a document is ingested, every chunk receives a metadata tag: `{ "tenant_id": "T-123", "dept": "HR" }`. At query time, the API Gateway authenticates the user, securely injecting `T-123` into the backend. The backend hardcodes a pre-filter (`$eq: { tenant_id: "T-123" }`) into every vector search. The Vector DB executes the filter before calculating distances, guaranteeing Tenant A mathematically cannot retrieve Tenant B's vectors.

How do you implement per-tenant rate limiting for LLM API calls?

Answer: We use an API Gateway (like Kong) paired with a Redis distributed cache.

We deploy a Token Bucket algorithm. Each tenant has a max Tokens-Per-Minute (TPM) allocation based on their subscription tier. Before dispatching a request to OpenAI, the backend checks Redis: `INCR tenant:123:tokens_used`. If the accumulated tokens exceed their limit, the backend rejects the request immediately returning a `HTTP 429 Too Many Requests` back to the tenant, shielding the underlying LLM provider from the load.

How do you handle per-tenant prompt customization without managing 500 different prompts?

Answer: We use a Compositional Prompt Architecture stored in a relational DB (e.g., PostgreSQL).

We maintain a single "Global Core Prompt" (e.g., "You are an AI assistant processing documents..."). We allow tenants to define override rules in a Settings UI (e.g., "Always respond in Spanish"). In the DB, the tenant's record holds a `custom_instructions` text field. At generation time, the backend fetches both and concatenates them dynamically using variables: ` {tenant.custom_instructions} User Query: {query}`. This avoids managing 500 distinct files.

Design cost attribution — how do you know which tenant is consuming how many tokens?

Answer: LLM APIs (OpenAI/Anthropic) natively return a `usage` payload inside every response containing `prompt_tokens` and `completion_tokens`.

The backend intercepts this payload, maps those tokens to current provider pricing (e.g., $5.00/1M prompt tokens), and emits a structured log or telemetry event tagged with the authenticated `tenant_id`. These events are piped into a Data Warehouse (Snowflake/Clickhouse) or Time-Series database, rolling up via materialized views to provide internal billing dashboards or customer-facing usage metrics.

A tenant's data must be deleted immediately (GDPR). How do you handle this in a vector DB?

Answer: Vector databases do not have native sub-table level drops, so we rely on explicit mapping.

We maintain a canonical relational database (Postgres) representing our tenant schema. When a tenant triggers an account deletion, we collect their internal `tenant_id` alongside all their `document_ids`. We then issue a batch metadata-based delete command to the Vector database: `DELETE from collection WHERE metadata.tenant_id == "T-123"`. The Vector DB invalidates the physical embeddings immediately. This is backed up by TTLs (Time to Live) on associated cloud storage buckets.

Agents & Tool Use

How does a ReAct agent work internally? What is the think-act-observe loop?

Answer: ReAct interleaves reasoning capabilities with environment actions.

  1. Think: The LLM analyzes the user prompt and breaks down the task. It writes a "Thought" string explaining its rationale (e.g., "I need to find the customer's ID before I can look up their orders").
  2. Act: Based on the thought, it outputs a strict JSON payload predicting the name of a registered tool and the arguments required (`call: { name: 'get_user_id', args: { email: 'x@x.com' } }`). Execution pauses.
  3. Observe: The backend takes this JSON, executes the actual code (SQL/API call), and appends the raw result back to the LLM's memory trace as an "Observation".

It loops back to "Think" indefinitely until it decides it has enough context to issue a "Final Answer".

When do agents fail in production and how do you guard against it?

Answer: Agents fail spectacularly due to non-deterministic execution paths:

  • Failures: Hallucinating non-existent tools, passing malformed JSON parameters, or getting trapped in an infinite retry loop (calling the same failed tool 100 times).
  • Guards: Use `gpt-4o` or models specifically trained on Structured Outputs (JSON mode) to enforce strict schema adherence. Implement a hard `max_iterations = 5` threshold. If an agent loops 5 times without a `Final Answer`, the backend terminates it with an Exception back to the user interface indicating "Failed to complete task."

How do you design tool schemas for an LLM agent that interacts with your legacy MySQL + MongoDB stack?

Answer: The Golden Rule is Bounded Context, No Raw Execution.

Never expose a generic "run_query(sql_string)" tool. Instead, build specific, atomic wrapper functions.
For MySQL: `tool: get_order_status(order_id: int, user_id: string)`
For Mongo: `tool: get_profile_preferences(user_email: string)`
Provide hyper-detailed schema descriptions to the LLM so it knows exactly what the tool does and what format the parameters must take. The backend code encapsulates the complex joins and sanitization, keeping the LLM strictly as a routing and extraction engine.

How do you prevent an agent from getting stuck in an infinite tool-calling loop?

Answer: Aside from a hard `max_iterations`, we implement Iterative Memory Awareness.

If an agent calls Tool X with Argument Y, and it receives `Error: Data Not Found`, it might just try it again due to lack of diverse reasoning. We log the history trace. If our backend orchestrator detects that the exact same Tool+Argument combo was executed twice consecutively yielding the exact same error, we intercept and forcibly prompt the LLM: "System Observation: Tool X has failed twice. Try an alternative strategy or exit and ask the user for clarification."

Design an agent that can answer questions by querying both a SQL database and a vector store. How do you decide which tool to call?

Answer: You expose both systems as modular tools to the Agent.

Tool 1: `query_structured_sales_db(SQL_query)` - Decription: "Use this to answer questions about exact sales numbers, dates, aggregations, counts, and revenue figures."
Tool 2: `query_unstructured_policy_docs(search_string)` - Description: "Use this to answer conceptual questions about company policies, return rules, definitions, and subjective summaries."

The LLM's reasoning loop evaluates the intent. If it sees "What was the revenue in Q3?", it routes to structured SQL. If it sees "Why was the Q3 contract rejected?", it routes to the Vector Store. For hybrid queries, it will call both sequentially and synthesize the observations.

How do you add human-in-the-loop approval for high-stakes agent actions?

Answer: Through asynchronous decoupling states.

When the Agent selects a high stakes tool (e.g., `process_customer_refund`), the backend intercepts execution before running the underlying function. The Agent's process tree transitions into a `PAUSED_WAITING_APPROVAL` state. The backend sends a WebSocket/API event to the frontend UI: "Agent wants to run a refund for $500. [Approve] / [Deny]". When the human clicks Approve, the backend wakes the paused executing state, finishes the function call, and resumes the Agent loop with an "Action Approved and Executed" observation.

Async & Queue Architecture for LLMs

Why would you use an async job queue for LLM tasks instead of a synchronous API call?

Answer: Synchronous logic fails due to network unreliability and generation latency.

LLM streams can take anywhere from 2 seconds to 4 minutes depending on output length and provider congestion. If 1,000 users upload PDFs synchronously, the web server's HTTP threads will lock up completely, resulting in 504 Gateway Timeouts. An async queue (BullMQ, Celery) allows the backend to return a 202 Accepted instantly. Dedicated background workers process the heavy LLM tasks at their own decoupled pace, ensuring the primary web server is always highly available.

Design a BullMQ-based pipeline for processing 1000 document ingestion jobs with retry logic and progress tracking.

Answer: We use BullMQ Flows (Parent-Child chaining).

  1. A user uploads 1000 PDFs. The system creates a single `Batch_Parent` job tracking the status of 1000 individual `Ingest_Child` jobs mapped to each PDF.
  2. Worker Nodes pick off `Ingest_Child` jobs one by one. They extract text, create chunks, hit the embedding API, and upsert to the Vector Store.
  3. As each `Ingest_Child` completes, it updates the `Batch_Parent` progress (e.g., 250/1000 complete) which the frontend polls.
  4. If a chunk embedding fails due to a network error, BullMQ automatically retries the specific child job using an Exponential Backoff policy (e.g., 5 retry maximums), ensuring one bad PDF doesn't break the entire 1000 document batch.

How do you prioritize urgent LLM jobs over batch jobs in the same queue?

Answer: In queuing systems like BullMQ, this is handled via Integer Priorities.

When queuing an asynchronous chat generation request (where a user is waiting Real-time at screen), we enqueue the job with `Priority = 1`. When we kick off a massive hourly background scraping and text classification job, we enqueue it with `Priority = 10`. The worker infrastructure inherently sorts the Redis active list by priority value, ensuring that any real-time user request dynamically skips ahead of thousands of background batch tasks.

A long-running LLM job fails midway. How do you implement resumable processing?

Answer: We implement Idempotency with State Tracking.

If we are extracting structured data from a 500-page lease document, we do not treat the 500 pages as a single monolithic block. Instead, we chunk it. After successfully processing every 10 pages, the worker updates a database record: `Progress Cursor = 40`. If the worker container crashes midway, the job fails and re-enters the active queue. When the new worker picks up the job, it polls the database, reads `Progress Cursor = 40`, and gracefully resumes analysis starting precisely from page 41, saving massive token costs.

How do you handle backpressure when LLM API rate limits are hit mid-queue?

Answer: With a queue-level Rate Limiter plugin reacting directly to API headers.

When an LLM provider returns a 429 error, it typically includes a `Retry-After: 30` header indicating a cool-off window. Instead of aggressively retrying and triggering a ban, the worker catches the 429 error and explicitly tells the BullMQ global queue manager to "PAUSE". The manager halts the entire execution queue for exactly 30 seconds. This applies systemic backpressure, stopping all workers instantly preventing further cascading api violations. Once 30 seconds expires, the queue unpauses automatically.

NL-to-SQL Specifically

What are the failure modes of NL-to-SQL in production?

Answer: There are four primary failure states:

  1. Hallucination: Creating tables or column names that simply do not exist in your database schema.
  2. Syntax Errors: Combining incompatible functions, missing GROUP BY statements, or invalid SQL dialect translation (e.g., assuming Postgres syntax within a Snowflake environment).
  3. Business Logic Misinterpretation: The query is syntactically valid but functionally wrong. (e.g., The user asks for "Revenue," and the LLM sums `total_cart_value` instead of `completed_purchases`.).
  4. Security: Prompt injection causing destructive outputs (`DROP TABLE`).

How do you handle ambiguous queries where multiple SQL interpretations are valid?

Answer: This is solved at the Prompt and UX levels.

First, prompt the LLM to identify ambiguity: "If the query request is vague and has multiple conflicting SQL interpretations, output a clarifying question to the user instead of generating SQL."

If generated anyway, the UI must visualize the assumptions. We display the executed data table but provide a natural language rationale above it: "I assumed 'active users' means users who logged in within the last 30 days. To change this assumption, refine your question."

How do you prevent SQL injection via LLM-generated queries?

Answer: 100% Zero-Trust execution infrastructure.

No LLM code should ever have mutation privileges. Execute the generated SQL via a dedicated IAM Database Role that is completely Read-Only (`SELECT` permissions only). Drop, Truncate, Insert, and Update must be explicitly revoked. Additionally, pass the generated string through a strict AST (Abstract Syntax Tree) SQL parser/linter in the backend to ensure it is bounded to a single SELECT query without malicious chained statement injections (`;`).

How do you give the LLM enough schema context without blowing the context window on a 200-table database?

Answer: We implement Schema Retrieval (RAG for Schema).

We do not dump 200 schemas into the prompt. Instead, we chunk the database structure table-by-table (combining DDL, column descriptions, and sample rows) and embed them into a Vector Database. When the user asks, "Show regional sales", we dynamically search the Vector Store to retrieve the top 3-5 most mathematically relevant schemas (e.g., `sales_records`, `region_mapping`). We inject only those 5 relevant tables into the LLM context, ensuring high accuracy while saving thousands of tokens.

Curated Architecture Questions

Walk me through your multi-tenant architecture decision — why did you choose that over schema-per-tenant or DB-per-tenant?

Answer: DB-per-tenant creates massive operational drag. When deploying an LLM stack, managing 500 individual Postgres instances, scaling 500 redundant CPU allocations, and orchestrating massive migration scripts for schema updates is agonizing. Schema-per-tenant limits DB connection pooling limits. I opted for a Shared DB with Row-Level/Metadata Filtering. It provides cost efficiency, instant tenant onboarding, and we enforce strict data isolation inside the API access logic and Vector DB metadata queries, which is computationally trivial and highly secure when implemented correctly via ORMs/Frameworks.

Why BullMQ over alternatives like Celery, SQS, or RabbitMQ for your use case?

Answer: The platform was rooted deeply bridging TypeScript applications and LLM logic. BullMQ heavily leverages Redis (which we already required for Semantic Caching and Rate Limiting). This drastically streamlined infrastructure, unlike SQS or RabbitMQ which required onboarding new AWS/AMQP topologies. BullMQ excels at advanced Flows (Parent-child job tree dependencies), which are vital for asynchronous Agent loops, and has native logic for API Rate Limit backpressure plugins directly mapped to its Redis core.

In your RAG pipeline, how did you decide on chunk size and overlap?

Answer: We avoided guessing and ran an empirical evaluation benchmark via RAGAS.

We created a Golden Dataset and ran the ingestion pipeline testing configurations: `[256-token, 512-token, 1024-token]`. We found that 256 generated too much noise fragmenting clauses, leading to low Context Recall. 1024 was too bloated, increasing API costs and confusing the LLM due to the "Lost in the Middle" phenomenon. 512 tokens with a standard ~50-token semantic overlap provided the golden mean between embedding-level precision and generation-phase coherence.

How does your API Gateway handle authentication across microservices?

Answer: Microservices should be decoupled from identity providers. We use the API Gateway (e.g., Kong) as the primary security checkpoint. The client sends a JWT token. The Gateway validates the JWT signature, claims, and expiry. Once validated, the Gateway rips off the JWT token and appends explicitly trusted internal HTTP Headers (e.g., `X-Internal-Tenant-ID`, `X-Internal-User-Role`) before passing the request to downstream backend components like the Vector DB or Generation Engine. The microservices inherently trust these specific headers since the internal VPC network prevents direct public bypasses.

If you were to scale your current architecture 10x, what breaks first?

Answer: The two biggest bottlenecks would be Vector Search Latency under heavy concurrent load and Provider API Rate Limits.

At 10x load, performing approximate nearest neighbor search via brute force or un-optimized Index configurations (HNSW) without memory allocation tweaking will drastically slow down retrieval speed. The other failure is TPM (Tokens per Minute). Rapid traffic spikes will instantly exhaust OpenAI bounds causing a massive backup in the BullMQ waiting queues. Moving to dedicated API infrastructure (Microsoft Azure Provisioned Throughput Units - PTUs) and Sharding the Vector Database horizontally across clusters would be the immediate resolution roadmaps.

Latency & Performance Optimization

How do you reduce the Time-To-First-Token (TTFT) and increase tokens per second (TPS) in production?

Answer: Optimization requires addressing different parts of the inference stack:

  • Reducing TTFT (Wait time):
    • Prompt Caching: Cache the KV (Key-Value) states of static system prompts or common prefixes (like document contexts) to avoid re-computing them.
    • Flash Attention: Use kernels like FlashAttention-2 or Flash-Decoding to accelerate attention computation on high-memory bandwidth GPUs.
    • Speculative Decoding: Use a tiny "draft model" to predict several tokens ahead, which the main "target model" verifies in a single parallel step.
  • Increasing TPS (Throughput):
    • Continuous Batching: Unlike static batching, it schedules new requests as soon as any request in the batch finishing, maximizing GPU utilization (used in vLLM/TGI).
    • Quantization: Convert weights from FP16 to INT8, FP8, or 4-bit (AWQ/GPTQ) to reduce memory bandwidth bottlenecks and allow larger batch sizes.
    • KV Cache Management: Use PagedAttention to manage memory fragmentation, allowing more concurrent requests to share the same VRAM efficiently.

What is Speculative Decoding, and in what real-world scenarios is it most effective?

Answer: Speculative Decoding is an optimization technique where a smaller, faster "draft" model (e.g., Llama-3 8B) predicts the next 3-5 tokens. The larger "target" model (e.g., Llama-3 70B) then validates these tokens in a single forward pass.

Effectiveness: It is most effective when:

  1. High Latency/Large Model: The target model is significantly slower than the draft model.
  2. High Predictability: The task involves repetitive or highly structured text (e.g., code generation, standard legal formatting) where the small model can guess correctly ~70-80% of the time.
  3. Low Concurrency: In scenarios with high QPS, the overhead of the draft model might actually reduce total system throughput; it's best for improving per-user latency.

How does KV Caching improve latency, and what are its memory implications?

Answer: During autoregressive generation, Transformers re-calculate the attention for all previous tokens to predict the next one. KV Caching stores the Key and Value tensors of previous tokens in VRAM so they don't have to be re-computed.

  • Latency Benefit: Converts an $O(N^2)$ computation (where $N$ is sequence length) into an $O(N)$ operation for each new token generated, drastically speeding up response times.
  • Memory Implications (The "VRAM Tax"): As the conversation grows, the KV cache grows linearly. For a 70B model with a 32K context window, the KV cache can consume tens of gigabytes of VRAM. This is why PageAttention (vLLM) is used to prevent fragmentation and Multi-Query Attention (MQA) or Grouped-Query Attention (GQA) were designed to reduce the size of the KV cache by sharing keys/values across multiple heads.

Advanced Fine-Tuning & Hyperparameters

How do you choose between different cost (loss) functions based on the problem type during fine-tuning?

Answer: The loss function defines what the model "learns" from the data:

  • Supervised Fine-Tuning (SFT): Uses Cross-Entropy Loss. This is standard for teaching a model to follow instructions or imitate a specific writing style by predicting the next token in a "Golden" dataset.
  • Direct Preference Optimization (DPO): Uses a specialized Binary Cross-Entropy variant on pair-wise preferences (Chosen vs Rejected). It implicitly learns a reward model without needing a separate RLHF step. Best for alignment (e.g., "be more helpful, less harmful").
  • Reinforcement Learning from Human Feedback (RLHF): Uses PPO (Proximal Policy Optimization) with a KL-Divergence penalty. The KL penalty ensures the model doesn't drift too far from the original base model ("reward hacking") while trying to maximize the reward score.
  • Embedding Fine-Tuning: Uses Contrastive Loss (Triplet/InfoNCE). It pulls similar concepts together in vector space and pushes dissimilar ones apart. Essential for domain-specific RAG retrieval optimization.

How do you decide on hyperparameters like LoRA Rank ($r$), Alpha ($\alpha$), and Learning Rate?

Answer: While there is no "one-size-fits-all", industry rules of thumb include:

  • Rank ($r$): Represents the "expressive power" of the adapter.
    • $r=8 \text{ or } 16$ is usually sufficient for simple style changes.
    • $r=64 \text{ or } 128$ for complex domain knowledge or task-specific logic (like coding).
  • Alpha ($\alpha$): A scaling factor for the LoRA weights. A common heuristic is $\alpha = 2 \times r$. This keeps the signal-to-noise ratio stable across different ranks.
  • Learning Rate (LR): For LoRA, we use higher rates than full fine-tuning. Typically $1 \times 10^{-4}$ to $2 \times 10^{-4}$. We use a Cosine Scheduler with Warmup (e.g., 5-10% of total steps) to prevent the gradients from exploding at the start.
  • Batch Size: Usually 4-16 depends on VRAM. We use Gradient Accumulation to simulate larger batches (e.g., $BS=4$, $Acc=4 \rightarrow \text{Effective Batch Size}=16$) for better stability.

How do you identify and prevent "Catastrophic Forgetting" during the fine-tuning process?

Answer: Catastrophic forgetting occurs when a model loses its general reasoning or broad knowledge (e.g., forgets how to do math while being trained on medical docs).

  1. Identification: We use "Benchmarks as a Guardrail". During FT, we evaluate the model not just on the new task, but also on a held-out set on general benchmarks (MMLU, GSM8K). A sharp drop in GSM8K while medical accuracy rises is a red flag.
  2. Prevention - Parameter Efficient FT (PEFT): Techniques like LoRA or Q-LoRA drastically reduce forgetting because they freeze the original weights and only train a tiny fraction of new parameters.
  3. Prevention - Weight Decay & LR: Lower the learning rate and increase weight decay to keep weight updates subtle.
  4. Prevention - Data Packing/Replay: Include a small percentage (e.g., 5-10%) of the original pre-training data or a general instructional dataset (like Open-Orca) in your fine-tuning mix to "remind" the model of its foundational skills.

Memory in Gen AI

Why do LLMs forget the beginning of a long conversation, and how do you systematically fix it?

Answer: LLMs are inherently stateless. "Memory" only exists if you explicitly pass the conversation history in the context window. They forget the beginning because of Context Limits and the Sliding Window mitigation strategy.

When a conversation exceeds the model's token limit (e.g., 8K tokens), developers are forced to drop the oldest messages (Sliding Window). To fix this systematically, we use Summary Memory. A parallel background LLM call takes the oldest messages dropping out of the window and condenses them into a dense paragraph. The system prompt is then dynamically updated: [System Prompt] + [Rolling Summary of Turns 1-50] + [Raw Text of Turns 51-55], preserving the historical narrative without blowing the token budget.

Design a conversational memory system for a therapy AI that needs to remember patient context from 6 months ago.

Answer: A simple buffer or sliding window fails here. We need Persistent Semantic Memory paired with Entity Extraction.

  1. Semantic Memory (Vector DB): After every session, we chunk and embed the transcript. When the patient returns 6 months later and says, "My brother is acting up again," we embed that query and hit the Vector DB. It retrieves the context from Month 1 explaining the brother's specific behavioral issues and injects it into the LLM.
  2. Entity Memory (Knowledge Graph): Real-time Vector search can miss explicit facts (like names). During the chat, we run an asynchronous extraction pipeline to populate a NoSQL user profile: {"allergies": ["peanuts"], "siblings": [{"name": "John", "relation": "brother"}]}. This structured JSON is injected into the System Prompt on every single turn, ensuring foundational facts are never "forgotten" or buried by search algorithms.

What are the critical security risks of Semantic Memory in a multi-tenant Chatbot, and how do you mitigate them?

Answer: The critical risk is Cross-Tenant Data Leakage. If you embed all user conversations into a single Vector Database collection for conversational memory, an ambiguous query from User A (e.g., "What is the secret API key we discussed?") might have a high cosine similarity to a chunk from User B's private chat history.

The LLM will ingest User B's retrieved chunk and expose it to User A. Mitigation: You must enforce Hard Metadata Filtering at the database level. Every embedded memory chunk must contain a user_id or tenant_id tag. The Backend/API Gateway strictly enforces $eq: { user_id: "User-A-UUID" } on every vector search, guaranteeing mathematical isolation before the ANN search even begins. You cannot rely on prompt-level instructions to maintain privacy.
