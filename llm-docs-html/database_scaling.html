Q78: What is event-driven architecture and when do you use it?

Services communicate via events (Kafka, RabbitMQ) rather than direct API calls — loose coupling, async processing, and natural audit trail.

Core Points

  •  Producer publishes event to topic; multiple consumers independently react — no direct coupling
  • between producer and consumer
  •  At-least-once delivery: consumers must be idempotent — process duplicate events safely
  •  Event store: Kafka retains events for days/weeks — consumers can replay from any point
flowchart LR Sender["Order Service"] -- "Event: OrderCreated" --> Broker[("Message Broker (Kafka)")] Broker --> Consumer1["Inventory Service"] Broker --> Consumer2["Shipping Service"] Broker --> Consumer3["Email Service"]
Depth (Senior Signal):
  • Event schema evolution: use Avro + Schema Registry — enforce backward/forward compatibility
  • Consumer lag monitoring: if consumer falls behind, it needs more instances or optimized processing
Trap / Watch Out: Event-driven adds debugging complexity — distributed tracing and correlation IDs are essential to trace a request across events

Q79: How do you implement rate limiting at the database level?

Track request counts in Redis per key with TTL — check and increment atomically before allowing the operation.

Core Points

  •  Redis INCR + EXPIRE: increment counter, set TTL on first request — atomic with Lua script
  •  Sliding window: ZADD with timestamp scores, ZREMRANGEBYSCORE to remove old entries,
  • ZCARD for count
  •  Apply at middleware layer: check rate limit before business logic runs
flowchart TD API["API Server"] --> Redis[("Redis")] Redis -- "INCR user:123:minute" --> Count{"Count > Limit?"} Count -- "No" --> Proceed["Proceed to DB"] Count -- "Yes" --> Block["Return 429 Too Many Requests"]
Depth (Senior Signal):
  • Distributed rate limiting across multiple API servers: Redis as shared state ensures global limit is
  • respected
  • Token bucket in Redis: store remaining tokens as value, replenish on schedule via background job
Trap / Watch Out: In-memory rate limiting (per server instance) doesn't work with multiple API server instances — must use shared Redis store

Q80: Explain write-ahead logging (WAL) in databases

WAL writes changes to a log before applying to data files — enables crash recovery, replication, and PITR.

Core Points

  •  Changes logged to WAL first — if crash occurs, DB replays WAL to recover committed transactions
  •  Physical replication: replica streams WAL from primary and applies it — stays in sync
  •  PITR (Point-in-Time Recovery): replay WAL from base backup to any point — essential for disaster
  • recovery
flowchart TD App["App"] -- "Commit Tx" --> RAM["DB Buffer Pool"] RAM -- "1. Append to WAL" --> DiskWAL[("WAL File on Disk")] DiskWAL -- "2. Acknowledge Commit" --> App RAM -- "3. Async Checkpoint" --> DiskData[("Main Data Files")]
Depth (Senior Signal):
  • Logical replication: sends row-level changes (not physical blocks) — allows replication to different
  • schema versions
  • WAL archiving: ship WAL segments to S3 — enables PITR without storing full DB snapshots
Trap / Watch Out: WAL accumulates until checkpointed — if checkpoint is delayed (heavy writes), WAL directory grows and can fill disk

Q81: How do you implement soft deletes?

Add deleted_at timestamp column — filter WHERE deleted_at IS NULL in queries; retain data for audit and recovery.

Core Points

  •  Soft delete: UPDATE records SET deleted_at = NOW() instead of DELETE
  •  All queries must filter deleted_at IS NULL — use DB views or ORM global scope to enforce
  •  Partial index: CREATE INDEX ON records(id) WHERE deleted_at IS NULL — index only active
  • records
flowchart LR Delete["User clicks Delete"] --> DB[("Database")] DB -- "Does NOT run DELETE" --> Update["UPDATE users SET deleted_at = NOW()"] AppQuery["SELECT * FROM users"] -- "Must include" --> Clause["WHERE deleted_at IS NULL"]
Depth (Senior Signal):
  • Unique constraints: unique on (email, deleted_at IS NULL) — allows re-registration after soft delete
  • Scheduled hard delete: after 90 days, permanently delete soft-deleted records for compliance and
  • performance
Trap / Watch Out: Forgetting to add WHERE deleted_at IS NULL in a query returns deleted records — use a default scope or view to avoid this

Q82: How do you handle database backups and disaster recovery?

Automated daily snapshots + continuous WAL archiving → test restore regularly → RTO/RPO define the strategy.

Core Points

  •  RTO (Recovery Time Objective): max acceptable downtime — drives multi-AZ/active-standby
  • decisions
  •  RPO (Recovery Point Objective): max acceptable data loss — drives backup frequency and WAL
  • shipping
  •  AWS RDS: automated backups to S3 with PITR up to 35 days retention — test restore quarterly
flowchart TD Primary[("Primary DB")] -- "Daily" --> FullBackup[("S3 Full Snapshot")] Primary -- "Continuous (Streaming)" --> WALArchive[("S3 WAL Archive")] Crash{"Disaster Strikes"} --> Recover["Restore Snapshot + Replay WALs"]
Depth (Senior Signal):
  • Multi-AZ RDS: synchronous standby in another AZ — automatic failover in 60-120 seconds
  • Cross-region read replica: disaster recovery if entire region goes down — promote to primary in DR
  • scenario
Trap / Watch Out: Backups you've never restored are untested backups — schedule quarterly restore drills to a test environment
