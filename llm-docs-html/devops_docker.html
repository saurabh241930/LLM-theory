Beyond the Basics: Layering, Caching, and Multi-Stage Optimization

1. Image Layering (Union File System)

Every instruction in a Dockerfile (FROM, RUN, COPY) creates a new read-only layer. When you start a container, Docker adds a thin writable layer on top.

Why Order Matters?

Docker uses a cache to speed up builds. If a layer changes, all subsequent layers are invalidated and must be rebuilt. This is why you should always copy your package.json or requirements.txt before copying the rest of your source code.

2. Multi-Stage Builds (The Production Standard)

Standard builds include compilers, source code, and development tools in the final image, making it huge and vulnerable. Multi-stage builds solve this by using multiple FROM statements.

# Stage 1: Build FROM node:18-alpine AS builder WORKDIR /app COPY . . RUN npm install && npm run build # Stage 2: Production FROM nginx:alpine COPY --from=builder /app/dist /usr/share/nginx/html

Result: Image size drops from ~800MB to ~20MB, and the attack surface is minimized.
