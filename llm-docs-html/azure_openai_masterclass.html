Masterclass: Azure OpenAI & Security

PTU vs PAYG, Content Safety, and Managed Identity RBAC

1. Capacity Planning: PTU vs. PAYG

Deploying OpenAI in an enterprise requires more than an API key. You must choose your Throughput Model.

PAYG Pay-As-You-Go

Standard global availability. Shared resources. Rates are metered per 1K tokens.

Best for: Development, Testing, and Low-traffic apps.

PTU Provisioned Throughput

Reserved capacity with predictable latency. No "noisy neighbor" effects.

Best for: 24/7 Production, High-throughput QA agents.

2. The "No Key" Policy: Managed Identities

Storing AZURE_OPENAI_API_KEY in environment variables is a high risk. Modern Azure apps use Managed Identities to authenticate via RBAC.

Crucial: Assign the Cognitive Services OpenAI User role to your App Service or VM identity to enable token-based access. 
# Secure Client Initialization (Python)
from azure.identity import DefaultAzureCredential
from openai import AzureOpenAI

# 1. Zero-Key Auth: Uses the logged-in Identity
credential = DefaultAzureCredential()
token_provider = get_bearer_token_provider(
    credential, "https://cognitiveservices.azure.com/.default"
)

# 2. Initialize Client
client = AzureOpenAI(
    azure_ad_token_provider=token_provider,
    api_version="2024-02-15-preview",
    azure_endpoint="https://my-aoai.openai.azure.com/"
)

3. Content Safety & Jailbreak Filters

Azure OpenAI includes a built-in Content Safety layer. It analyzes both Input (Prompt) and Output (Completion) for 4 categories: Hate, Self-Harm, Sexual, and Violence.

  • Async Mode: Low latency, filter results returned after result.
  • Sync Mode: High security, blocks the completion if a threshold is hit.

4. Multi-Region Global Load Balancing

Because PTU is region-locked, large enterprises often use Azure API Management (APIM) as a gateway to load balance between East US, Sweden Central, and West Europe deployments.
