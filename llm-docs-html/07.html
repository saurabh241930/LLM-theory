The Problem with Cosine

Vector similarity is fast but noisy. It ranks "How to return" higher than "Defect warranty" because of keyword overlap, ignoring the query-chunk relationship.

Bi-Encoder vs. Cross-Encoder

Bi-Encoder (Vector Search): Separate embeddings. Fast (O(1)). Coarse.

Cross-Encoder (Reranker): Reads [Query, Chunk] together. Slow (O(n)). Precise.

Two-Stage Workflow

Stage 1: Embedding search retrieves Top-100 fast.

Stage 2: Reranker scores the Top-100, returns Top-5.

Why? Reranking 1M docs is too slow/expensive. Reranking 100 is cheap (~$0.001).

The Funnel
Stage 1: All Docs (Vector Search)
Stage 2: Top-100 (Reranker)
Stage 3: Top-5 (To LLM)

Trade-offs

Vector search is fast but inaccurate. Reranking is accurate but slow.

Recommendation: Always use a two-stage pipeline. Don't rerank the whole database.

When to Add Reranking

  • Latency Critical? No (Reranking adds ~500ms).
  • Accuracy Critical? Yes (Legal/Medical).
  • Data Noisy? Yes (Cosine fails often).
  • Budget? Yes (API costs ~$0.001/q).
Decision Tree

Key Takeaways

✅Cosine is Coarse: Fast but misses subtle relevance.
✅Rerankers are Precise: Cross-encoders read query+chunk together.
✅Two-Stage is Key: Vector (1M→100) → Rerank (100→5).
✅Cost Effective: High quality for minimal latency/cost overhead.
