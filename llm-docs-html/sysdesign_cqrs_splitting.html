Read/Write Splitting & CQRS

Command Query Responsibility Segregation & Event Sourcing

1. Native Read/Write Splitting

When database traffic overwhelmingly skews towards reading (95% read vs 5% write), scaling vertically (buying larger single node databases) is economically abysmal. Instead, traffic is split across the network boundary.

  • Architecture: Single primary node serving all INSERT/UPDATE statements natively streams binary logs asynchronously to multiple Read-Only replicas serving all SELECT statements.
  • Complexity: The backend application ORM layer intelligently routes queries based on the HTTP verb. It requires explicit logic to prevent the Read-Your-Own-Writes anomaly stemming from replication lag.

2. CQRS Deep Dive

Read/Write Splitting uses the exact same database schema natively for both operations. CQRS (Command Query Responsibility Segregation) posits that the Data Model utilized for Writes/Validation should be fundamentally, physically different from the Data Model utilized for Reads/Display.

Why different models?

The Write Side (Commands): Demands absolute data integrity, locking, strict normalizations, constraints, and relational consistency. It executes business logic (e.g., verifying funds before withdrawal) to mutate state safely in a PostgreSQL engine.

The Read Side (Queries): Demands blazing fast multi-view performance. Complex heavily-joined relational data is transformed via a streamed message queue into a highly denormalized document inside MongoDB or an Elasticsearch cluster explicitly tailored for instant search UI rendering.

3. Event Sourcing (The Core Companion)

CQRS is almost entirely permanently coupled with Event Sourcing. Instead of maintaining the current state of a database entity (relational rows replacing old values)...

Storing Events Instead of State

Event Sourcing records every state mutation as an immutable chronological sequence of events.

Instead of storing the exact current Account Balance ($100), the system strictly stores the ledger: [AccountCreated, Deposit($50), Deposit($100), Withdrawal($50)]. The current actual state is resolved mathematically by replaying the events.

Benefit: A completely flawless audit trail. An absolute "undo" structure perfectly tracking historical anomaly states. If you introduce a new Read reporting database next year, you just replay the Event Log from the beginning of time sequentially to hydrate the entire new database immediately.
