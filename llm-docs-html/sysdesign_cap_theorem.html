CAP Theorem — The Practical Version

Consistency, Availability, and the Inevitability of Partitions

1. Core Concept

The CAP Theorem states that a distributed data store can only simultaneously provide two out of the following three guarantees:

  • Consistency (C): Every read receives the most recent write or an error. (All nodes see the exact same data at the same time).
  • Availability (A): Every request receives a non-error response, without guaranteed that it contains the most recent write.
  • Partition Tolerance (P): The system continues to operate despite an arbitrary number of messages being dropped or delayed by the network between nodes.

2. The Practical Reality: 'P' is NOT a Choice

Why CA Systems Don't Exist over the WAN

In the real world, networks fail. Switches reboot, fiber cables get cut, routers experience congestion. Because Partition Tolerance (P) is inevitable in distributed systems, the true choice during a network failure is strictly between CP and AP.

When the network link between Node 1 and Node 2 crashes:

  • If Node 1 accepts a write, Node 2 cannot receive it.
  • To maintain Consistency (CP): Node 2 must refuse to serve reads to Client 2 because it knows its data is stale. It sacrifices Availability.
  • To maintain Availability (AP): Node 2 must serve Client 2 with stale data. It sacrifices Consistency.

3. Architecting for CP vs AP

Type Database Examples Use Case Resolution during a Network Tie
CP (Consistency/Partition) MongoDB, HBase, Redis, PostgreSQL (strict sync) Financial transactions, Inventory management, Billing The system locks down. It returns errors rather than serving inaccurate data. (E.g. A MongoDB replica set loses its primary and blocks writes until a new one is elected).
AP (Availability/Partition) Cassandra, DynamoDB, CouchDB, Riak Social media timelines, Shopping cart items, Product reviews The system keeps serving traffic on both sides of the partitioned network. Uses "Eventual Consistency" to heal the data discrepancy once the network is restored.

4. Edge Cases: PACELC Theorem

CAP only describes what happens during a network partition. But what happens during normal operation when the network is perfectly fine? The PACELC Theorem expands on CAP:

PACELC: If Partitioned, choose Availability or Consistency. Else (normal operation), choose Latency or Consistency.
  • DynamoDB / Cassandra (PA/EL): When partitioned, choose Availability. Else, choose Latency (serve reads from the closest node immediately, even if slightly stale).
  • MongoDB (PC/EC): When partitioned, choose Consistency (blocks writes). Else, choose Consistency (reads happen from the Primary node by default, incurring higher latency).
