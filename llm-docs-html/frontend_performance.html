Q53: How do you handle error boundaries in React?

Error boundaries are class components that catch rendering errors in their subtree and show a fallback UI instead of crashing the app.

Core Points

  •  componentDidCatch and getDerivedStateFromError: catch errors in child component tree
  •  Granular boundaries: wrap individual sections (sidebar, feed, card) so one failure doesn't crash the
  • whole app
  •  Log errors to Sentry or similar in componentDidCatch for production observability
flowchart TD App["Main App"] --> EB["Error Boundary Class Component"] EB -- "try-catch equivalent for rendering" --> ComponentTree["Child Component Tree"] ComponentTree -- "Throws JS Error" --> EB EB -- "Catches Error" --> Fallback["Render Fallback UI avoiding white screen"]
Depth (Senior Signal):
  • react-error-boundary library: functional API wrapper around class-based error boundaries
  • Error boundaries don't catch: async errors, event handler errors, or errors in the boundary itself
Trap / Watch Out: One global error boundary at the root catches errors but shows blank app — granular boundaries provide much better UX

Q54: How do you implement authentication flow in a React SPA?

Store JWT in memory (not localStorage), use refresh token in httpOnly cookie, protect routes with auth guard component.

Core Points

  •  Access token in memory (useState/Context): XSS can't steal it unlike localStorage
  •  Refresh token in httpOnly cookie: browser sends it automatically, JS can't access it
  •  ProtectedRoute component: checks auth state, redirects to /login if unauthenticated
flowchart LR Login["Login Form"] --> Context["Auth Provider"] Context -- "API Call" --> Backend["Auth Backend"] Backend -- "Returns JWT" --> Storage["HttpOnly Cookie"] Storage --> Router["Protected Routes"]
Depth (Senior Signal):
  • Silent refresh: use refresh token to get new access token before expiry — transparent to user
  • PKCE flow for SPAs: OAuth2 extension that doesn't require client secret — safe for public clients
Trap / Watch Out: Storing JWT in localStorage is a common mistake — any XSS vulnerability can steal it. Memory storage is safer.

Q55: What is the difference between controlled and uncontrolled components?

Controlled: React state drives the input value; uncontrolled: DOM manages its own state, accessed via ref.

Core Points

  •  Controlled: value={state} onChange={setState} — React is the single source of truth; enables
  • validation on every keystroke
  •  Uncontrolled: no value prop, use ref to read value on submit — simpler for file inputs and legacy
  • integrations
  •  Controlled is preferred for most forms — predictable, easier to test, works with form libraries
flowchart LR subgraph Controlled Component ReactState["React State (useState)"] <--> Input["Input value={state} onChange={update}"] end subgraph Uncontrolled Component DOMNode["Raw DOM Node"] --> Ref["React useRef()"] end
Depth (Senior Signal):
  • Mixing controlled and uncontrolled: providing value then removing it switches mode — React warns
  • about this
  • React Hook Form uses uncontrolled inputs with refs for performance — avoids re-render on every
  • keystroke
Trap / Watch Out: Providing value={undefined} is the same as uncontrolled — use value={''} as initial state to keep it controlled

Q56: How do you implement micro-frontends?

Split large frontend apps into independently deployable modules, integrated at runtime via Module Federation or iframes.

Core Points

  •  Webpack Module Federation: host app loads remote modules at runtime from different deployment
  • URLs
  •  Each team owns their module end-to-end: build, test, deploy independently
  •  Shared dependencies (React, design system) defined as singletons — loaded once, not per module
flowchart TD AppShell["App Shell / Container"] --> Fed1["Module Federation: Header (React)"] AppShell --> Fed2["Module Federation: Product List (Vue)"] AppShell --> Fed3["Module Federation: Cart (Angular)"]
Depth (Senior Signal):
  • Communication between modules: custom events on window or shared state store — avoid direct
  • imports across boundaries
  • iframe approach: maximum isolation but poor UX (separate scroll, URL, auth per frame)
Trap / Watch Out: Micro-frontends add significant complexity — only justified for large teams where deployment coupling is a real problem

Q57: What is Event Delegation and how does React handle events?

Event delegation attaches a single listener to a parent element to manage events for all descendants, improving performance.

Core Points

  • Instead of putting onclick on 1,000 list items, you put one listener on the ul and use event.target
  • React implements this through its SyntheticEvent system—a cross-browser wrapper around native events
  • React attaches a single event listener to the root container for almost all events
flowchart TD ReactRoot["Root DOM Node (id=root)"] -- "Attaches Single Listener" --> EventQueue["React SyntheticEvent"] Button1["Button A Click"] --> ReactRoot Button2["Button B Click"] --> ReactRoot EventQueue --> Handler["Dispatches to specific onClick"]
Depth (Senior Signal):
  • React 17 changed event delegation from attaching to the document object to the root DOM container
  • This change allowed multiple React versions to safely coexist on the same page
Trap / Watch Out: Calling e.stopPropagation() in React stops the React SyntheticEvent propagation, not necessarily the native DOM propagation outside the React tree.

Q58: How do you implement SSR and what are its tradeoffs?

SSR renders HTML on server per request — faster first contentful paint and SEO; tradeoff is server load and complexity.

Core Points

  •  Next.js (React): getServerSideProps renders per-request; getStaticProps renders at build time (SSG)
  •  Angular Universal: server-side rendering for Angular apps — hydration re-attaches event listeners on
  • client
  •  Time to First Byte (TTFB) is higher for SSR vs static — server must render before responding
flowchart LR Server["Node/Next.js Server"] -- "Renders fully formed HTML" --> Browser["Browser"] Browser -- "Displays FCP instantly" --> Paint["User Sees Page"] Browser -- "Downloads JS bundles" --> Hydrate["React Hydration"] Hydrate -- "Interactive" --> App["Responsive SPA"]
Depth (Senior Signal):
  • Streaming SSR (Next.js 13+ App Router): stream HTML chunks as they're ready — user sees
  • content faster
  • Islands architecture: most of the page is static HTML; only interactive islands are hydrated
Trap / Watch Out: SSR doesn't automatically improve performance — if the server is slow, TTFB suffers. Measure before assuming SSR is better.

Q59: What is hydration and why does it matter in SSR?

Hydration is attaching React/Angular event handlers to server-rendered HTML so the static page becomes interactive.

Core Points

  •  Without hydration: HTML is visible but clicks, inputs, and state don't work
  •  Full hydration: React/Angular re-renders the entire component tree client-side to attach event
  • listeners
  •  Hydration mismatch: if server and client render different HTML, React logs a warning and re-renders
  • from scratch
flowchart TD StaticHTML["Static HTML from Server"] --> Attach["React attaches event listeners"] Attach --> Match{"HTML match initial client render?"} Match -- "No" --> Error["Hydration Mismatch Error"] Match -- "Yes" --> Interactive["Page is interactive"]
Depth (Senior Signal):
  • Partial hydration: only hydrate interactive components — static parts stay as HTML forever (Astro,
  • Qwik)
  • Resumability (Qwik): serialize component state at render time, resume on client with zero
  • re-execution
Trap / Watch Out: Accessing browser APIs (window, document, localStorage) during SSR will crash — always guard with typeof window !== 'undefined'

Q60: How do you implement accessibility (a11y) in web apps?

Semantic HTML + ARIA roles where needed + keyboard navigation + screen reader testing — accessibility is not an add-on.

Core Points

  •  Semantic HTML: use
  • keyboard-accessible
  •  ARIA labels: aria-label, aria-describedby for elements that lack text — bridges gap for screen readers
  •  Focus management: trap focus in modals, restore focus on modal close
flowchart LR DOM["Semantic HTML DOM"] --> AccessibilityTree["a11y Tree"] AccessibilityTree --> ScreenReader["Screen Reader / Assistive Tech"] Styles["CSS / Visuals"] -. "Ignored by" .-> ScreenReader
Depth (Senior Signal):
  • WCAG 2.1 AA: minimum standard for most products — covers color contrast, keyboard nav, and
  • screen reader compatibility
  • Test with screen readers: VoiceOver (Mac), NVDA (Windows), and automated tools like axe-core or
  • Lighthouse
Trap / Watch Out: ARIA overuse is harmful — 'no ARIA is better than bad ARIA'; semantic HTML elements already have correct roles built in

Q61: How do you handle internationalization (i18n) in modern SPAs?

Externalize all strings to locale files, use a library for interpolation/pluralization, and handle RTL layouts for Arabic/Hebrew.

Core Points

  •  React: react-i18next — useTranslation hook, t('key') for string lookup, locale JSON files per language
  •  Angular: built-in i18n with @angular/localize or ngx-translate for runtime switching
  •  ICU message format: handles pluralization and gender in translations correctly
flowchart LR App["React App (i18next)"] --> Key["Text Node: {t('welcome_msg')}"] Key --> Lookup{"Current Lang Context"} Lookup -- "en" --> EN[("en.json")] Lookup -- "es" --> ES[("es.json")]
Depth (Senior Signal):
  • RTL support: CSS logical properties (margin-inline-start vs margin-left) work for both LTR and RTL
  • Number, date, and currency formatting: use Intl API (built into browsers) — don't hardcode formats
Trap / Watch Out: Hardcoding strings or date formats anywhere in component code makes i18n impossible later — centralize from day one

Q62: What are Web Vitals and how do you optimize for them?

Core Web Vitals: LCP (load), FID/INP (interactivity), CLS (layout shift) — directly measured by Google and affect SEO.

Core Points

  •  LCP (Largest Contentful Paint): optimize with image preloading, CDN, fast server response — target
  • <2.5s< /li>
  •  INP (Interaction to Next Paint): minimize main thread blocking — defer non-critical JS, break up long
  • tasks
  •  CLS (Cumulative Layout Shift): reserve space for images/ads with aspect-ratio CSS — target <0.1< /li>
flowchart TD UX["User Experience Metrics"] --> LCP["LCP (Largest Contentful Paint) - Loading speed"] UX --> FID["INP/FID (Interaction to Next Paint) - Interactivity"] UX --> CLS["CLS (Cumulative Layout Shift) - Visual stability"]
Depth (Senior Signal):
  • Lighthouse in CI: automated Web Vitals scoring on every deploy — fail PR if score drops below
  • threshold
  • Real User Monitoring (RUM) with web-vitals library: measure actual user experience, not just lab
  • scores
Trap / Watch Out: Lighthouse score in Chrome DevTools is a lab measurement — optimize for real-user CrUX data from Google Search Console too
