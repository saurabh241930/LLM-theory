In the development of Large Language Model (LLM) applications, chunking is the process of partitioning large bodies of text into smaller, more manageable segments. This is a critical technical requirement because the internal "memory" of an LLM—known as the context window—is finite. By breaking documents into pieces that fit these constraints, developers ensure that relevant information can be effectively retrieved and processed during inference.

Standard Chunking Strategies

Choosing the right chunking strategy involves Balancing semantic integrity (keeping related concepts together) against computational efficiency.

1. Fixed-Size Chunking

The simplest approach, where text is split into chunks of a constant length (e.g., 500 characters or tokens). While predictable and fast, this method often fails because it ignores grammatical structures, potentially splitting sentences mid-word.

2. Sliding Window (Overlap)

Similar to fixed-size, but nearby chunks overlap (e.g., a 500-token chunk with a 100-token overlap). This ensures that semantic context isn't lost at the boundaries, though it increases storage requirements in the vector database.

3. Recursive Character Splitting

This strategy attempts to split text at natural boundaries: first by paragraphs, then sentences, and finally by words. It only proceeds to the next level of granularity if the current fragment still exceeds the target size.

4. Semantic Chunking

An advanced technique where sentences are embedded into a vector space and similarity scores are calculated between neighbors. A "break" is created whenever there is a significant drop in similarity, ensuring that each chunk represents a single coherent topic.

Advanced Indexing Strategies

As dataset sizes grow from thousands to millions of documents, standard flat chunking often results in a "needle in a haystack" problem. Advanced systems use multi-tiered indexing to decouple the Retrieved unit from the Reasoning unit.

5. Multi-Representation Indexing (Parent-Child)

Instead of search and generation using the same chunk, this method stores Small Chunks (e.g., 100 tokens) for high-precision retrieval but returns the Parent Document (e.g., 1000 tokens) to the LLM. This provides the "Precision of a laser" with the "Context of a floodlight."

# Pseudo-flow: Parent Document Retrieval
for child_chunk in retrieved_top_k:
  parent_id = child_chunk.metadata["parent_id"]
  full_context = vectordb.get_by_id(parent_id)
  llm_prompt.append(full_context)
6. Hierarchical Indexing (RAPTOR)

RAPTOR (Recursive Abstractive Processing for Tree-Organized Retrieval) clusters chunks and generates summaries for each cluster. This processes repeats recursively to create a tree of summaries. Retrieval can then traverse the tree, allowing the LLM to answer "Big Picture" questions that span the entire document set.

Decision Framework

The following logic helps determine the optimal strategy for a given dataset:

Is your document structured (headers, sections)?
├─ YES → Use Document-Aware Chunking
└─ NO → Continue...

Is semantic accuracy critical (law, medicine)?
├─ YES → Use Semantic Chunking
└─ NO → Continue...

Budget for embedding all sentences?
├─ YES → Semantic Chunking (Best Quality)
└─ NO → Recursive or Fixed + Overlap

Real-World Scenario: Enterprise Documentation

Imagine a company with 1,000,000 technical manuals. A user asks: "What is the general safety philosophy across all battery products?"

  • Naive RAG: Might find 5 specific chunks about "Lithium-ion temperature limits." (Too specific).
  • Hierarchical RAG: Retrieves high-level category summaries about "Battery Safety Protocols," providing a comprehensive answer.

Common Pitfalls

  • Chunks Too Small: Leading to fragmented context and poor reasoning (less than 200 tokens).
  • Chunks Too Large: Exceeding the model's comfortable "reasoning" density within the context window (more than 1000 tokens).
  • Index Bloat: Storing child, parent, and summary representations can triple your storage costs.
  • Missing Metadata: Failing to store the source filename or page number with the chunk, making the answer impossible to verify.

Key Takeaways

✅ Targeted chunk size ≈ Context Window / 3 (for RAG).
✅ Fixed-size is for speed; Semantic is for accuracy.
✅ Always include metadata (doc_id, page) with every chunk.
✅ Recursive splitting is the most reliable "middle ground" for general documentation.
