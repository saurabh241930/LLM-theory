System Design: Company Policy AI Assistant

Architecting a multi-domain internal AI for HR, IT, and Finance.

Goal: Design an Internal Company AI Assistant capable of answering multi-domain questions regarding HR employee policies, leave & attendance records, expense reimbursement statuses, and IT security compliance guidelines. This assistant must operate securely within the enterprise firewall, protect sensitive employee data, and guarantee high accuracy (zero hallucinations on policy facts).

1. The Core Challenge: Why Not Send All Documents at Once?

A common misconception when building an internal AI is attempting to load the entire employee handbook, IT manual, and SQL database schema directly into the LLM's context window. This approach fails catastrophically for three reasons:

  • Context Limits & Cost: Even models with massive 128K+ token windows charge per token. Sending a 500-page PDF on every query is financially unsustainable.
  • Attention Degradation: LLMs suffer from "Lost in the Middle" syndrome. When crammed with too much noise, they struggle to retrieve specific facts (e.g., "What is the per-diem limit in London?"), often hallucinating or ignoring the correct paragraph buried in the middle of the prompt.
  • Data Privacy & ACL: You cannot send User A's un-redacted employee records to the LLM when User B is asking a generic HR question. Data must be filtered before it hits the LLM.

Solution: We use an Agentic Integration & Retrieval-Augmented Generation (RAG) pattern. The system first categorizes the user's intent, retrieves only the highly relevant data chunks or API responses, and constructs a narrow, highly targeted factual prompt for the LLM to synthesize.

3. Intelligent Routing with LangChain / LangGraph

The core of this system is not a single static chain, but a dynamic routing agent. An employee asking "What is the maternity leave policy?" requires document retrieval (RAG). An employee asking "How many PTO days do I have left?" requires executing an SQL query or API call against a transactional database.

We use LangGraph (or LangChain's AgentExecutor) to handle this. The router LLM is equipped with Function Calling (Tools):

  • Query_HR_VectorDB(query: str)
  • Query_IT_Security_DB(query: str)
  • Get_User_PTO_Balance(employee_id: str)
  • Check_Expense_Status(employee_id: str, expense_id: str)

How it works: The router analyzes the user request. If the required data is dynamic (PTO), it generates a JSON payload to call the Get_User_PTO_Balance tool, hitting the Workday API. If the data is static text (IT Policy), it routes the query down the RAG path.

4. Advanced RAG: Chunking & Hybrid Search

Chunking Strategies

Treating all documents equally leads to poor retrieval. We apply domain-specific chunking:

  • IT Security Policies (Semantic Chunking): IT guides are highly prose-based. We use Semantic Chunking (breaking text at sentence or paragraph boundaries when the semantic embedding drifts) so full concepts hold together.
  • HR Reimbursement Tables (Structural / Markdown Chunking): Tables regarding per-diem rates by city fail completely under standard splitting. We pre-process these docs using tools like Unstructured.io, converting tables into Markdown or HTML schemas, maintaining the exact row/column relationships within a single chunk.

Why Hybrid Search?

If an employee asks, "What is policy SEC-809 regarding YubiKeys?", a pure Vector (Semantic) Search might fail. "SEC-809" is an exact keyword string; its dense mathematical vector representation might inadvertently map closer to documents discussing "general hardware security" rather than the exact required document.

The Solution: Hybrid Search (BM25 + Dense Vectors)

  1. Dense Search (Semantic): Uses embeddings to find meaning ("I lost my password token" -> retrieves YubiKey docs).
  2. Sparse Search (BM25): Uses exact keyword matching (TF-IDF variants) to surgically strike exact IDs like "SEC-809".

We execute both searches in parallel, retrieving the Top-K from each. We then use a fusion algorithm like Reciprocal Rank Fusion (RRF) to merge the lists, before passing the combined list to a Cross-Encoder Reranker (e.g., Cohere Rerank) to sort the absolute specific passages for the LLM to read.

5. Hallucination Prevention & Measurement (RAGAS)

In a corporate environment, the AI declaring "You get 6 months of paid leave" when the policy is actually 6 weeks is a critical incident. We measure accuracy using frameworks like RAGAS (Retrieval Augmented Generation Assessment) in CI/CD, and enforce real-time guardrails.

RAGAS Metrics in Pipeline Testing

  • Faithfulness (Groundedness): Measures if the LLM's final answer can be completely derived from the retrieved chunks. If the LLM generates facts not present in the chunks, this score plummets.
  • Answer Relevance: Measures if the generated answer directly addresses the user's initial question, avoiding tangential rambling.
  • Context Precision: Measures if the Retriever put the most relevant chunks at the very top of the list.

Real-Time Guardrails

To prevent hallucinations at run-time, we append rigorous System Prompts forcing citation:

System: You are a strict corporate assistant. 
Answer the user's query ONLY using the provided Context chunks. 
You must cite the [Document ID] next to every factual claim you make.
If the Context does not contain the answer, you must output EXACTLY: 
"I do not have access to that policy document." Do not guess.
