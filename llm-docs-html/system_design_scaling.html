Q14: Design a scalable file upload and processing system

Pre-signed S3 URLs for direct uploads, SQS trigger for async processing, status polling or WebSocket for feedback.

Core Points

  •  Client requests pre-signed S3 URL from API → uploads directly to S3 (bypasses backend, saves
  • bandwidth)
  •  S3 event triggers SQS message → Lambda or worker picks up and processes (resize, parse, index)
  •  Processing status stored in DB; client polls or receives push notification via WebSocket/SNS
flowchart TD Client -- "1. Request Upload URL" --> API API -- "2. Returns Pre-signed URL" --> Client Client -- "3. Direct Upload" --> S3[("S3 Bucket")] S3 -- "4. S3 Event Trigger" --> SQS[("SQS Queue")] SQS --> Worker["Processing Worker"]
Depth (Senior Signal):
  • Chunked multipart upload for large files — S3 supports up to 5TB via multipart API
  • Virus scanning integration: run ClamAV or AWS Macie before making file available to downstream
Trap / Watch Out: Never stream large file uploads through your API server — it blocks threads and adds unnecessary latency

Q15: How do you approach capacity planning for a new service?

Start with SLA targets → estimate peak RPS → derive resource needs → add headroom → validate with load testing.

Core Points

  •  Define p99 latency SLA and availability target first — these drive all other decisions
  •  Back-of-envelope: peak RPS × avg response time = concurrency needed (Little's Law)
  •  Load test at 2x expected peak before launch — verify auto-scaling kicks in correctly
flowchart TD Req["Expected Traffic"] --> Little["Little's Law: N = CPU * T"] Little --> Node["Calc Required Nodes"] Node --> Test["Load Test at Peak x2"] Test --> ASG["Configure Auto-Scaling Group"]
Depth (Senior Signal):
  • Track CPU, memory, connection pool, and DB query time as the four leading indicators of
  • saturation
  • USE method (Utilization, Saturation, Errors) for systematic bottleneck identification
Trap / Watch Out: Don't capacity plan for average load — design for peak + 50% headroom; averages are misleading for bursty workloads

Q16: What is eventual consistency and when is it acceptable?

Eventual consistency means replicas converge to the same state given no new updates — acceptable when staleness is tolerable.

Core Points

  •  Examples where OK: social media feed, product recommendations, analytics dashboards
  •  Examples where NOT OK: financial balances, inventory deduction, authentication tokens
  •  BASE (Basically Available, Soft state, Eventually consistent) vs ACID — different guarantees for
  • different needs
flowchart LR ClientA["Client A (US)"] --> Replica1[("Replica US")] ClientB["Client B (EU)"] --> Replica2[("Replica EU")] Replica1 -- "Async Sync" --> Replica2 Replica2 -- "Async Sync" --> Replica1
Depth (Senior Signal):
  • Conflict resolution strategies: last-write-wins (LWW), vector clocks, CRDTs for automatic merging
  • Read-your-own-writes consistency: route user's reads to the same replica they just wrote to
Trap / Watch Out: Don't say 'eventual consistency is fine' without qualifying what staleness window is acceptable to the business

Q17: How would you design a real-time leaderboard?

Redis Sorted Sets for O(log N) rank updates and range queries — scores are atomic, reads are instant.

Core Points

  •  ZADD for score updates, ZRANK for rank lookup, ZRANGE for top-N — all O(log N)
  •  Partition by time window (daily, weekly, all-time) using separate sorted sets
  •  Persistent storage in Postgres for historical data; Redis is the live cache
flowchart TD Client["Game Client"] --> API["Score API"] API -- "ZADD score" --> Redis[("Redis Sorted Set")] API -- "ZRANGE 0 10" --> Redis Redis -- "Top 10 List" --> API
Depth (Senior Signal):
  • At massive scale, partition by score range across Redis shards — shard 1 handles scores 0-1M,
  • shard 2 handles 1M-2M etc.
  • Lua script in Redis for atomic score-update + rank-check in one round trip
Trap / Watch Out: Don't use SQL ORDER BY + COUNT for real-time rank — full table scan at scale is unacceptable

Q18: Explain idempotency in APIs and how to implement it

An idempotent operation produces the same result if called once or N times — critical for payment and retry-safe APIs.

Core Points

  •  Client sends an Idempotency-Key header (UUID) with each request
  •  Server stores key + response in Redis with TTL; on duplicate request, returns cached response
  • without reprocessing
  •  Especially critical for payment APIs — prevents double-charging on network retry
flowchart TD Client -- "POST /charge Header: Idem-Key=123" --> API API --> Check{"Key in Redis?"} Check -- "Yes" --> Return["Return Cached 200 OK"] Check -- "No" --> Process["Process Payment"] Process --> Cache["Store Result in Redis"]
Depth (Senior Signal):
  • Idempotency key scope: per-user or global — design based on whether two users can send same
  • key
  • Database-level: use UPSERT with unique constraint on idempotency key for persistence
Trap / Watch Out: GET and DELETE are naturally idempotent; POST is not — you must implement it explicitly for POST/PATCH

Q19: How do you design for multi-tenancy in a SaaS application?

Three models: shared DB + shared schema (row-level), shared DB + separate schema, or separate DB per tenant — tradeoffs in cost vs isolation.

Core Points

  •  Row-level: tenant_id column on every table + Row Level Security (Postgres RLS) — cost-efficient
  • but blast radius risk
  •  Schema per tenant: strong isolation, easier backup/restore per tenant — moderate cost
  •  DB per tenant: maximum isolation for enterprise/regulated — highest cost and ops overhead
flowchart TD subgraph Multi DB - Physical Isolation DB_A[("Tenant A DB")] DB_B[("Tenant B DB")] end subgraph Single DB - Logical Isolation Table["Table: Data with tenant_id"] end
Depth (Senior Signal):
  • Hybrid: shared DB for small tenants, dedicated DB for enterprise tenants (tiered isolation)
  • Connection pooling (PgBouncer) is critical in shared models — one pool per tenant schema
Trap / Watch Out: Forgetting tenant_id in a query is a data leak — use middleware/ORM hooks to enforce tenant scope automatically

Q20: How do you implement distributed caching and avoid cache stampede?

Cache-aside pattern with Redis; prevent stampede using probabilistic early expiration or a lock-based single-fill approach.

Core Points

  •  Cache-aside: app checks Redis, on miss fetches from DB and populates cache
  •  Cache stampede: many requests hit DB simultaneously on cache expiry — solved with mutex lock or
  • probabilistic refresh before expiry
  •  Cache invalidation strategies: TTL-based, write-through, write-behind, or event-driven invalidation
flowchart TD App["Application"] --> Cache[("Redis")] Cache -- "Miss (Lock acquired)" --> DB[("Primary DB")] DB --> Populate["Populate Cache"] Cache -- "Hit" --> App
Depth (Senior Signal):
  • Redis cluster for horizontal scaling; consistent hashing determines which shard owns each key
  • Bloom filter to avoid caching 'negative' lookups (non-existent keys) — prevents cache pollution
Trap / Watch Out: Cache invalidation is the hardest problem — 'just use TTL' is acceptable but stale data window must be explicitly accepted

Q21: Design a webhook delivery system

Async queue-based delivery with retries, exponential backoff, signature verification, and delivery log.

Core Points

  •  On event: write to outbox → relay publishes to queue → worker makes HTTP POST to customer
  • endpoint
  •  Retry with exponential backoff (1s, 2s, 4s, ... up to 24hr) on non-2xx responses
  •  HMAC-SHA256 signature on payload — customer verifies to prevent spoofing
flowchart LR App["Internal System"] --> Queue[("Webhook Queue")] Queue --> Worker["Worker Node"] Worker -- "HMAC Post" --> Customer["Customer Endpoint"] Customer -- "500 Error" --> Retry["Exp Backoff + Retry"]
Depth (Senior Signal):
  • Dead letter queue after max retries — alert customer and allow manual replay
  • Circuit breaker per endpoint: if customer endpoint fails consistently, pause delivery and notify
Trap / Watch Out: Don't make webhook delivery synchronous — customer endpoints can be slow or down; always decouple with a queue

Q22: How do you handle secrets management in a microservices environment?

Centralized secrets store (AWS Secrets Manager, HashiCorp Vault) with short-lived credentials and automatic rotation.

Core Points

  •  Never hardcode secrets in code or environment files committed to version control
  •  AWS Secrets Manager: store DB passwords, API keys — inject at runtime via SDK or ECS task role
  •  Vault: dynamic secrets — generates short-lived DB credentials on demand, auto-revokes after TTL
flowchart TD Service["Microservice"] --> Vault["HashiCorp Vault"] Vault -- "Dynamic Creds (TTL=1hr)" --> Service Service --> DB[("Database")]
Depth (Senior Signal):
  • IAM roles for service-to-service auth on AWS — no static keys needed between AWS services
  • Secret zero problem: how does Vault client authenticate to Vault? — solved via trusted platform
  • identity (EC2 instance profile)
Trap / Watch Out: Storing secrets in env vars in Docker images is a common mistake — they appear in image layers and process lists

Q23: Explain the Sidecar pattern in microservices

A helper container deployed alongside the main service container, handling cross-cutting concerns like logging, proxy, and auth.

Core Points

  •  Sidecar runs in the same pod (Kubernetes) — shares network and storage with main container
  •  Common sidecars: Envoy proxy (Istio), log shipper (Fluentd), secret injector (Vault agent)
  •  Decouples infrastructure concerns from business logic — main service stays clean
flowchart LR subgraph Kubernetes Pod Main["Main App Container"] Sidecar["Envoy / Fluentd Sidecar"] Main <--> Sidecar end Sidecar <--> External["External World / Logs"]
Depth (Senior Signal):
  • Service mesh uses sidecars at scale — every pod gets an Envoy sidecar for mTLS, circuit breaking,
  • and observability
  • Ambassador pattern: a variant where sidecar handles outbound traffic specifically
Trap / Watch Out: Sidecar adds resource overhead per pod — weigh the benefit of decoupling vs cost at high pod density

Q24: How would you implement full-text search in a large application?

Elasticsearch or OpenSearch as dedicated search index — sync from primary DB via CDC or event stream.

Core Points

  •  Inverted index: Elasticsearch maps terms to document IDs — O(1) lookup for any term
  •  Sync strategy: Debezium CDC from Postgres → Kafka → Elasticsearch consumer maintains index
  •  Relevance tuning: BM25 scoring, field boosting (title > body), and synonym dictionaries
flowchart TD App["App Server"] --> Postgres[("Postgres Primary")] Postgres -- "CDC (Debezium)" --> Kafka[("Kafka")] Kafka --> ES[("Elasticsearch")] App -- "Search Queries" --> ES
Depth (Senior Signal):
  • Fuzzy search, autocomplete (edge n-gram tokenizer), and multi-language analysis built into ES
  • Index lifecycle management: hot-warm-cold tiers for cost optimization on time-series indices
Trap / Watch Out: Don't use LIKE '%term%' in SQL for full-text search — it's a full table scan and kills performance at scale

Q25: What is the difference between horizontal and vertical scaling?

Vertical scaling adds resources to one machine; horizontal scaling adds more machines — horizontal is preferred for cloud-native systems.

Core Points

  •  Vertical: bigger instance (CPU, RAM) — simple but has hardware ceiling and requires downtime
  •  Horizontal: more instances behind a load balancer — no ceiling, zero-downtime scaling, but requires
  • stateless services
  •  Stateless services scale horizontally easily; stateful services need sticky sessions or external
  • session store (Redis)
flowchart LR subgraph Horizontal S1["Server 1"] LB["Load Balancer"] --> S1 LB --> S2["Server 2"] LB --> S3["Server 3"] end
Depth (Senior Signal):
  • Database horizontal scaling: read replicas (scale reads), sharding (scale writes) — different
  • techniques
  • Auto Scaling Groups on AWS: scale out on CPU > 70%, scale in on CPU < 30% — define cooldown
  • periods
Trap / Watch Out: Horizontal scaling doesn't automatically help if the bottleneck is the DB — scale the DB tier separately

Q26: How do you design an audit logging system?

Immutable append-only event log capturing who did what, when, and what changed — stored separately from business data.

Core Points

  •  Capture: actor, action, target entity ID, before/after state, timestamp, request ID
  •  Write to append-only store — never update or delete audit logs (compliance requirement in BFSI)
  •  Separate storage from business DB — high write volume shouldn't impact transactional performance
flowchart LR Service["Any Service"] -- "Log Event" --> Kafka[("Kafka (Immutable)")] Kafka --> S3[("S3 WORM Storage")] Kafka --> ES[("Search Index (Dashboards)")]
Depth (Senior Signal):
  • WORM storage (S3 Object Lock) for tamper-proof compliance logs
  • Kafka as audit log backbone — immutable, ordered, replayable; consumers build various views
Trap / Watch Out: Don't log sensitive data (PII, passwords) in audit logs — log entity IDs and action types, not raw values
