Core Rendering & Layout

Q42: How does React's reconciliation algorithm work?

React diffs the virtual DOM tree against the previous snapshot; only nodes that changed are updated in the real DOM.

Core Points

  •  Virtual DOM: lightweight JS object tree representing UI — diffing it is O(n) with React's heuristics
  •  Key prop: tells React which list items are stable across renders — missing keys cause unnecessary
  • re-mounts
  •  Fiber architecture (React 16+): reconciliation is interruptible — high-priority updates (user input) can
  • preempt renders
flowchart TD StateChange["State / Props Change"] --> VDOM_New["New Virtual DOM"] VDOM_Old["Old Virtual DOM"] --> Diffing{"Reconciliation (Diffing)"} VDOM_New --> Diffing Diffing -- "Computes Minimal Mutation" --> DOM["Real DOM"]
Depth (Senior Signal):
  • React assumes different component types produce different trees — swapping component type
  • triggers full subtree re-mount
  • Concurrent mode: time-slicing breaks rendering into chunks, keeping UI responsive during heavy
  • renders
Trap / Watch Out: Using array index as key is dangerous in dynamic lists — index doesn't identify items, only position; items can mismatch on reorder

Q43: When should you use useState vs useReducer?

useState for simple independent values; useReducer when state logic is complex, has multiple sub-values, or transitions depend on previous state.

Core Points

  •  useState: single value or simple object — straightforward, minimal boilerplate
  •  useReducer: complex state machine — actions make transitions explicit and testable
  •  useReducer enables dispatching the same action from many places without prop drilling
flowchart LR subgraph useState Primitive["Primitive Value"] --> SetState["setState()"] end subgraph useReducer Complex["Complex State Object"] --> Dispatch["dispatch({type: ACT})"] Dispatch --> Red["Reducer Function"] end
Depth (Senior Signal):
  • Combine useReducer + useContext for app-wide state without Redux — simpler than Redux for
  • mid-size apps
  • Immer.js with useReducer: write mutations that produce immutable state updates — cleaner
  • reducer code
Trap / Watch Out: useState with an object doesn't merge updates like setState in class components — you must spread previous state manually

Q44: Explain React.memo, useMemo, and useCallback — when to use each

All three prevent unnecessary work — memo for component re-renders, useMemo for expensive computations, useCallback for stable function references.

Core Points

  •  React.memo: wraps component, skips re-render if props haven't changed (shallow comparison)
  •  useMemo: caches computed value, recomputes only when dependencies change — use for
  • expensive derivations
  •  useCallback: caches function reference — prevents child re-renders when function is passed as prop
  • to memo'd child
flowchart TD Parent["Parent Re-renders"] --> Child{"Child Component Wrapped in React.memo?"} Child -- "Yes" --> Props{"Props Changed? (uses useCallback)"} Child -- "No" --> Rerender["Child Re-renders"] Props -- "Yes" --> Rerender Props -- "No" --> Skip["Skip Render"]
Depth (Senior Signal):
  • Premature memoization adds complexity — measure first with React DevTools Profiler before
  • adding memo
  • useMemo and useCallback have their own cost (comparison) — only worthwhile if the saved work is
  • heavier
Trap / Watch Out: React.memo doesn't help if parent passes new object/array literals as props on every render — memoize those objects too

Q45: How do you manage global state in React? Compare options.

Context + useReducer for simple cases; Zustand/Jotai for lightweight atomic state; Redux Toolkit for complex, large-scale apps.

Core Points

  •  Context API: built-in, no dependency — but all consumers re-render when any context value
  • changes (use splitting)
  •  Zustand: minimal boilerplate, selective subscriptions — components only re-render when their slice
  • changes
  •  Redux Toolkit: best for large teams — devtools, time-travel debugging, strong conventions
flowchart TD Context["React Context (Low frequency)"] --> Redux["Redux (High structure / Boilerplate)"] Redux --> Zustand["Zustand (Minimalist / No providers)"] Zustand --> Jotai["Jotai (Atomic / Recoil-like)"]
Depth (Senior Signal):
  • Server state (API data) is different from client state — use React Query or SWR instead of stuffing it
  • in Redux
  • Jotai/Recoil: atom-based state — fine-grained reactivity, each component subscribes to only what it
  • needs
Trap / Watch Out: Putting everything in global state is an anti-pattern — server cache (React Query), URL state (router), and UI state should be separate

Q46: What are React hooks rules and why do they exist?

Only call hooks at the top level (not in loops/conditions) and only in React functions — ensures hook call order is stable across renders.

Core Points

  •  React tracks hooks by call order — if a hook is conditionally called, order changes and state
  • mismatches on next render
  •  This is why ESLint plugin (eslint-plugin-react-hooks) enforces the rules automatically
  •  Custom hooks encapsulate stateful logic — reusable across components without changing
  • component hierarchy
flowchart TD Comp["Function Component"] --> TopLevel["Hooks must be at Top Level"] Comp --> NoLoop["Cannot be inside loops, conditions, or nested functions"] TopLevel --> Ensure{"Ensures Call Order Independence"} NoLoop --> Ensure
Depth (Senior Signal):
  • Under the hood: hooks are stored in a linked list per component fiber; order is the only index React
  • has
  • Custom hook naming convention: must start with 'use' so React and linters can identify and validate
  • them
Trap / Watch Out: Calling a hook inside a conditional or loop looks harmless but silently corrupts state — React gives a cryptic error, not a clear one

Q47: Explain the Browser Critical Rendering Path and how to optimize it

The sequence of steps the browser takes to convert HTML, CSS, and JS into pixels on the screen, optimized by minimizing render-blocking resources.

Core Points

  • Steps: Parse HTML -> Build DOM -> Parse CSS -> Build CSSOM -> Render Tree -> Layout -> Paint -> Composite
  • CSS is render-blocking: the browser won't render processed content until CSSOM is built
  • JS is parser-blocking: when the parser encounters a script tag, it pauses HTML parsing to fetch and execute it
flowchart LR HTML["HTML"] --> DOM["DOM Tree"] CSS["CSS"] --> CSSOM["CSSOM Tree"] DOM --> RenderTree["Render Tree"] CSSOM --> RenderTree RenderTree --> Layout["Layout"] Layout --> Paint["Paint"]
Depth (Senior Signal):
  • Use `async` (executes when downloaded, without blocking) or `defer` (executes after DOM parsing) for scripts
  • Preload critical assets using <link rel="preload"> to fetch them early in the lifecycle
Trap / Watch Out: Inlining all CSS/JS avoids network requests but ruins caching and balloons the initial HTML payload.

Q48: What are Web Workers and Service Workers? How do they differ?

Web Workers offload heavy CPU tasks to background threads; Service Workers act as network proxies for caching and offline support.

Core Points

  • JS is single-threaded. Heavy calculations block the main thread, causing UI freezes
  • Web Workers execute scripts in a separate background thread without DOM access, communicating via postMessage
  • Service Workers intercept network requests, enabling offline capabilities (PWA), background sync, and push notifications
flowchart TD Browser["Browser Main Thread (UI)"] -- ".postMessage()" --> WebWorker["Web Worker (Heavy compute)"] Browser -- "Registers" --> ServiceWorker["Service Worker (Network Proxy / Offline)"] ServiceWorker <--> Cache[("Browser Cache API")]
Depth (Senior Signal):
  • Service workers have a distinct lifecycle: Install -> Wait -> Activate. An activated SW requires a reload to control clients
  • SharedWorkers can be accessed by multiple scripts across different windows/iframes of the same origin
Trap / Watch Out: Trying to manipulate the DOM or accessing `window` inside any Worker will throw an error since they run in a different global context.

Q49: Explain CSS Flexbox vs CSS Grid and when to use each

Flexbox is for 1D layouts (rows or columns), while CSS Grid is for 2D layouts (rows and columns simultaneously).

Core Points

  • Flexbox flows items in a single direction. Best for alignment, distributing space, and wrapping items
  • CSS Grid defines a structured grid template, allowing items to span multiple tracks
  • Flexbox is content-out (size driven by content), Grid is layout-in (content conforms to grid areas)
flowchart LR Flex["Flexbox: 1-Dimensional"] --> Row["Rows OR Columns"] Grid["CSS Grid: 2-Dimensional"] --> GridCells["Rows AND Columns simultaneously"]
Depth (Senior Signal):
  • You can easily mix them: A grid item can be a flex container. Use Grid for page-level structure and Flexbox for components
  • minmax() and auto-fit/auto-fill in Grid allow for fully responsive layouts without media queries
Trap / Watch Out: Overusing Grid for simple single-axis alignments adds unnecessary complexity; default to Flexbox until you need a second dimension.

Q50: Explain RxJS operators you use most and why

switchMap for latest-only (search), mergeMap for parallel, concatMap for ordered, forkJoin for parallel completion, debounceTime for input throttling.

Core Points

  •  switchMap: cancels previous inner Observable on new emission — perfect for search-as-you-type
  • (discard stale results)
  •  debounceTime: waits for pause in emissions before continuing — prevents flooding API on keystroke
  •  forkJoin: waits for all Observables to complete — use for parallel API calls that all must succeed
flowchart LR ClickStream["Click Event Stream"] --> Map["map()"] Map --> Debounce["debounceTime(300ms)"] Debounce --> SwitchMap["switchMap(API Call)"] SwitchMap --> Result["Subscribe (Update UI)"]
Depth (Senior Signal):
  • takeUntilDestroyed (Angular 16+) or takeUntil(destroy$): auto-unsubscribe on component destroy to
  • prevent memory leaks
  • catchError inside switchMap: handle individual request errors without killing the outer Observable
Trap / Watch Out: Subscribing inside subscribe (nested subscriptions) is an anti-pattern — always flatten with switchMap/mergeMap/concatMap

Q51: How do you implement code splitting and bundle optimization in React?

React.lazy + Suspense for component-level code splitting; analyze bundle with webpack-bundle-analyzer to find heavy dependencies.

Core Points

  •  React.lazy(() => import('./HeavyComponent')): splits HeavyComponent into separate chunk, loaded
  • on demand
  •  Suspense fallback: shows loading UI while chunk downloads — prevents blank screen
  •  Tree shaking: import { specificFunction } from 'library' instead of import library — removes unused
  • code
flowchart TD Webpack["Bundler (Webpack/Vite)"] --> Main["main.[hash].js (Core App)"] Webpack --> Vendor["vendor.[hash].js (React, Lodash)"] Webpack --> Async["chunk-auth.[hash].js (Loaded dynamically)"]
Depth (Senior Signal):
  • Dynamic imports for feature flags: if (feature.enabled) { const Module = await import('./Module') }
  • Preload key routes: or webpack magic comment /* webpackPrefetch: true */
Trap / Watch Out: Over-splitting creates too many small chunks — more HTTP requests can be slower than one slightly larger bundle. Profile before splitting.

Q52: What are common React performance pitfalls and how do you fix them?

Unnecessary re-renders, expensive computations in render, and large list rendering are the top three — each has a specific fix.

Core Points

  •  Unnecessary re-renders: use React DevTools Profiler to identify — fix with React.memo and stable
  • prop references
  •  Expensive computations in render: wrap with useMemo — recalculate only when inputs change
  •  Large lists: virtualize with react-window or react-virtual — render only visible rows
flowchart TD Issue["Unnecessary Re-renders"] --> DevTools["React Profiler Extension"] DevTools --> Identify["Identify component rendering too often"] Identify --> Fix["Apply React.memo / useCallback / useMemo"]
Depth (Senior Signal):
  • Avoid creating objects/functions inline in JSX: <Component onClick={()=> fn()} /> creates new
  • function every render
  • Use production build for performance testing — development mode adds significant overhead
Trap / Watch Out: Premature optimization is real — profile first, optimize second. Most React apps are fast enough without any manual optimization.
