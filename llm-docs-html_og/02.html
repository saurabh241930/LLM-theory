02. Tokenization & Context Windows

LLM Input Processing • Budgeting • Optimization
Cheat Sheet

What is a Token?

The atomic unit LLMs read. Not necessarily a word. It can be a part of a word, a whole word, or punctuation.

LLMs process Token IDs, not characters.

💡 "Hello" might be 1 token, but "f1f5f9" might be 3.

How Tokenization Works

Text passes through a Tokenizer (Vocabulary Lookup) before reaching the LLM.

  • Input: "The quick brown fox"
  • Tokenizer: Maps words to IDs (e.g., 1, 5, 9, 12).
  • LLM: Processes the sequence of IDs.

Context Window

The max tokens an LLM can process at once (Input + Output). Includes System Prompt, User Message, and History.

⚠️ Exceeding this limit causes ERROR or TRUNCATION.

Sliding Window Mechanism
Only tokens inside the frame are processed

Optimization Strategies

1. Summarization

Long doc (8K) → Summary (500 tokens).

2. Chunking (RAG)

Split into 2K chunks → Retrieve relevant ones.

3. Prompt Compression

Remove fluff/stopwords while keeping meaning.

Special Tokens

Invisible markers used by the model to structure the conversation.

  • <|start|> / <|end|>
  • <|im_start|> (Turn boundaries)
💡 They consume tokens but are invisible to the user.

<end>

RAG Implications (Budgeting)

You must fit Query + Chunks + System + History within the limit.

  • Bad: Retrieve 10 chunks → Expensive, hits limit.
  • Good: Retrieve 3 chunks → Fits, cheap, fast.

Common Pitfalls

  • Not counting tokens before API call (Surprise errors).
  • Forgetting special token overhead (Buffer needed).
  • Using old tokenizer data (Recount for new models).

Key Takeaways

✅Tokens ≠ Words: Variable length units (1 word ≈ 1.3 tokens).
✅Window Limits: Hard cap on input + output.
✅Optimization: Chunking/Summarization is mandatory for long docs.
✅Budgeting: Always estimate + buffer for special tokens.
