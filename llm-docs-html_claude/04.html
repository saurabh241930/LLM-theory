Chunking Strategies

Splitting Documents for Effective Retrieval-Augmented Generation: Methods, Trade-offs, and Best Practices

The Chunking Challenge in RAG

Retrieval-Augmented Generation (RAG) systems extend language models with the ability to reference external documents. However, documents—whether research papers, legal contracts, user manuals, or knowledge bases—often exceed the context window of language models. A document with 50,000 tokens cannot be provided directly to a model with a 4,000-token context window.

This fundamental constraint necessitates chunking: the strategic division of long documents into smaller, semantically coherent pieces that can be retrieved and provided to the model when needed. How documents are chunked directly impacts both the relevance of retrieved information and the model's ability to generate coherent answers.

The goal of chunking is deceptively simple yet challenging: preserve semantic meaning while ensuring pieces fit within context constraints. A chunk that is too small loses valuable context; one that is too large wastes tokens or doesn't fit at all. A chunk with artificial breaks (e.g., mid-sentence splits) becomes less useful. The right chunking strategy depends on the document structure, retrieval method, and downstream language model.

Core Tension: Chunking must balance context preservation (larger pieces are better) against context window constraints (smaller pieces are necessary). The optimal size depends on your specific system.

Fixed-Size Chunking

The simplest and most common chunking approach is to divide documents into fixed-size pieces. A document is split every N tokens (e.g., every 1,000 tokens), with optional overlap to preserve context at boundaries.

How It Works

A 10,000-token document with chunk size 1,000 and no overlap produces 10 chunks. A sliding window approach with 50% overlap would produce overlapping segments, so chunk 2 begins at token 500, chunk 3 at token 1,500, and so on. This overlap helps preserve meaning at chunk boundaries.

Advantages

  • Speed: Splitting requires no deep analysis—purely numerical operations.
  • Predictability: Output chunk count is entirely deterministic.
  • Simplicity: Easy to implement and understand.
  • Scalability: Works efficiently even on very large document collections.

Disadvantages

  • Semantic Fragmentation: Breaks may occur mid-sentence or mid-paragraph, losing meaning.
  • Inflexibility: Cannot adapt to natural document structure. A 500-word paragraph might span 5 chunks.
  • Boundary Loss: Without overlap, information at chunk edges can be lost entirely.

Despite these limitations, fixed-size chunking remains popular because its predictability makes retrieval and memory management straightforward. For many applications, the semantic cost is acceptable given the operational simplicity.

Semantic Chunking

Rather than dividing at fixed intervals, semantic chunking splits documents where topic shifts occur. The approach uses embeddings to measure semantic similarity between consecutive sentences and introduces a chunk boundary when similarity drops below a threshold.

The Algorithm

  1. Embed every sentence in the document using an embedding model.
  2. Calculate cosine similarity between consecutive sentences.
  3. When similarity drops below a threshold (e.g., 0.5), mark a chunk boundary.
  4. Recursively combine adjacent low-similarity groups if they're very small.

Advantages

  • Semantic Coherence: Each chunk represents a distinct topic, preserving meaning.
  • Variable Size: Chunks naturally adapt to content. A brief announcement might be one chunk; a complex explanation spans multiple.
  • No Artificial Breaks: Sentences stay intact—no mid-sentence splitting.

Disadvantages

  • Computational Cost: Embedding every sentence is expensive, especially for large document collections.
  • Dependency on Embeddings: Quality depends on the embedding model. Poor embeddings produce poor boundaries.
  • Threshold Sensitivity: Choosing the right similarity threshold requires tuning and experimentation.
When to Use: Semantic chunking is best for knowledge bases where semantic coherence is critical (e.g., medical documentation, legal contracts). The embedding cost is worthwhile if retrieval quality significantly improves.

Recursive Chunking

Recursive chunking attempts to preserve document structure by splitting hierarchically. It respects natural text boundaries while still adhering to size constraints.

How It Works

The algorithm tries splitting approaches in order, selecting the first that produces appropriately-sized chunks:

  1. First, try splitting by document structure markers (headers, section breaks).
  2. If chunks are still too large, recursively split by paragraph.
  3. If still too large, split by sentence.
  4. As a last resort, split by token if sentences are still oversized.

Advantages

  • Structure Preservation: Respects hierarchical document organization.
  • Flexibility: Adapts to various document types without configuration.
  • Balance: Attempts to honor semantic boundaries when possible, but falls back to size constraints when necessary.

Disadvantages

  • Complexity: More elaborate implementation than fixed-size chunking.
  • Unpredictability: Output structure varies based on document content.

Recursive chunking works well generically across different document types. It's particularly effective for structured documents like research papers (with sections and subsections) or software documentation (with headings and code blocks).

Document-Aware Chunking

When documents have explicit structure (Markdown headers, HTML elements, XML tags), chunking can leverage this structure. Each chunk preserves its source context—the header hierarchy from which it came.

Example: Markdown Documents

A Markdown document with multiple `## Section` headers can be chunked per section, with each chunk prefixed by its header. This metadata helps the retriever understand context and helps the model generate more contextualized responses.

Benefits

  • Context Preservation: The section title is always available as context for the chunk content.
  • Relevance Improvement: Metadata allows filters and better ranking by retrieval systems.
  • Traceability: Easy to link results back to the source section.

Implementation

Document-aware chunking requires parsing the document format (Markdown, HTML, XML, PDF). Once structural elements are identified, chunks are created at meaningful boundaries. The document structure becomes metadata attached to each chunk.

Limitation: Not all document formats have explicit structure. Unstructured text (emails, chat histories, scanned documents) cannot be chunked document-aware without additional processing.

Practical Guidelines for Chunk Sizing

Determining the optimal chunk size depends on multiple factors: the retrieval mechanism, the language model's context window, the embedding model, and the document type.

Recommended Sizes

A common heuristic is to use chunks that comprise approximately one-third of your language model's context window. If your model has a 4,000-token context, aim for chunks around 1,000-1,500 tokens. This leaves room for the query, other retrieved chunks, and generation.

Chunks smaller than 300 tokens often lack sufficient context, making them poor candidates for retrieval. Chunks larger than 3,000 tokens may not fit comfortably in context windows below 8,000 tokens.

Overlap Strategy

If using fixed-size or sliding window chunking, include 20-30% overlap. This ensures that information at chunk boundaries isn't lost. For instance, with 1,000-token chunks, use a 200-300-token overlap window.

Metadata and Traceability

Always attach metadata to chunks: source document name, source page or section, creation timestamp. This enables you to trace retrieved information back to its source and to identify and correct erroneous chunks later.

Validation and Iteration

Chunking decisions affect retrieval quality. Evaluate your chunking strategy empirically by measuring retrieval precision and recall. If many queries retrieve irrelevant chunks, try adjusting size, overlap, or strategy. This is inherently an iterative process without universal solutions.

Common Mistakes

Chunks Too Small

Fragments smaller than 200 tokens often lack sufficient context for meaningful retrieval. The search system must retrieve many fragments to assemble a coherent answer, increasing latency and complexity.

Chunks Too Large

Oversized chunks waste context window—they may contain relevant information mixed with irrelevant material. Large chunks also increase storage costs when vectorized.

No Overlap (Sliding Window Omission)

Without overlap, information at chunk boundaries can be entirely missed by retrieval. A retrieved chunk might be 5 tokens short of answering the query because relevant information exists in the adjacent omitted chunk.

Ignoring Document Structure

When documents have natural structure (sections, subsections), ignoring it by using only fixed-size chunking wastes this valuable signal. Document-aware approaches leverage structure better.

Missing Metadata

Chunks without source information become orphans. You cannot trace them back to original documents, making auditing and updates difficult.

Absence of Chunk Validation

Assuming a chunking strategy works without measuring retrieval quality is risky. Always validate chunks on representative queries before deploying to production.

Key Takeaways

Start Simple, Optimize Later: Begin with fixed-size chunking (easy to implement). Move to semantic or structural chunking if retrieval quality is insufficient.
Size Matters: Target chunks around 1/3 of your language model's context window. Most practical systems use 500-2,000 tokens per chunk.
Overlap Prevents Loss: Sliding window approaches with 20-30% overlap prevent valuable information from being lost at chunk boundaries.
Preserve Structure: When documents have structure (Markdown headers, sections), use document-aware chunking to preserve this context.
Always Attach Metadata: Source document, page, section—these enable traceability and debugging.
Validate Empirically: Measure retrieval quality on real queries. Chunking decisions are system-specific and require experimentation.
