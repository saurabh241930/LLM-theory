ReAct Pattern: Reasoning and Acting in Agent Loops

Enabling Agents to Reason, Plan, and Act Iteratively

ReAct: Enabling Reasoning and Acting in Agent Loops

ReAct (Reasoning + Acting) is a pattern for enabling language models to solve complex problems by alternating between thinking about what to do and taking actions. Rather than generating final answers directly, ReAct systems have the model generate reasoning steps, recognize when external tool use is needed, invoke tools, observe results, and reason about next steps in a loop. This pattern dramatically improves performance on tasks requiring planning, interaction with external systems, and iterative refinement.

The power of ReAct emerges from exposing reasoning. Traditional language models generate outputs in a single pass. ReAct systems generate visible reasoning—intermediate thoughts and plans—which can be inspected, validated, and corrected. When an intermediate reasoning step is wrong, the loop can be directed toward correction rather than compounding error downstream. This creates more reliable and interpretable AI systems.

The Original ReAct Contribution

The ReAct pattern was formalized in research showing that prompting language models to produce both thoughts (reasoning steps) and actions (tool invocations) together dramatically outperforms either alone. For example, when solving math word problems, prompting the model to show reasoning before answering yielded ~60% accuracy. Prompting the model to use a calculator tool when needed yielded ~65% accuracy. Combining both with ReAct structure (reason, then decide when to use calculator, observe results, continue reasoning) achieved ~85% accuracy. The combination is more than additive—the synergy comes from using reasoning to decide when tools help and using tool results to refine reasoning.

Key Insight: ReAct's power derives from interleaving reasoning and action, enabling the model to iteratively refine both its understanding and its plans based on intermediate results.

Core Components: Thoughts, Actions, Observations

ReAct systems operationalize reasoning and acting through three core components, repeated in loops until task completion.

Thoughts: Making Reasoning Explicit

Thoughts are reasoning steps the model generates before acting. Rather than jumping to actions, the model is prompted to think first: "I need to find the current population of Brazil. Let me search for this information." This explicit reasoning serves multiple purposes: (1) it guides the model's own behavior—thinking through the problem helps generate better actions, (2) it enables human oversight—readers can see the model's logic and spot errors, and (3) it often improves accuracy simply through forcing articulated reasoning.

The structure of thoughts matters. Some systems use natural language thoughts (any reasoning text), others structure thoughts more formally (step 1, step 2, etc.). Some systems include multiple thought formats: immediate thoughts (what to do next), longer-term planning (overall strategy), and correction thoughts (observations that the current approach won't work). This flexibility enables models to express different types of reasoning in appropriate formats.

Actions: Specifying What to Do

Actions are structured requests for tool invocations or information retrieval. A thought might be "I need to search for information about climate change." The action translates this into a structured request: ACTION: search("climate change effects on marine ecosystems"). The structure enables parsing and execution. Actions might be: (1) tool invocations (call a function), (2) API calls (query an external service), (3) database queries (retrieve information), or (4) information retrieval (search documents).

Effective action specification is crucial. Actions must be unambiguous—the system executing actions must understand what information is requested. Some systems use natural language actions (the executor uses language understanding to execute), others use structured formats (JSON, specific command syntax). Structured formats reduce ambiguity but require more engineering.

Observations: Incorporating Results

Observations are results returned from actions. After invoking a search action, the observation might be search results. After consulting a calculator, the observation is the computed value. Observations are fed back into the reasoning loop—the model sees results and generates new thoughts in response. This closes the feedback loop: reasoning informs actions, actions generate observations, observations inform refined reasoning.

How observations are presented affects downstream reasoning. Verbose observations with irrelevant information waste context and might distract the model. Terse observations might omit crucial details. Effective systems curate observations: return the 3-5 most relevant search results rather than 50, summarize complex information into key facts, and highlight unexpected findings that warrant investigation.

Thought

Model generates reasoning step: "What is needed next?"

Action

Model specifies tool invocation: "Search for X" or "Calculate Y"

Observation

System executes action and returns results to the model

Repeat or Conclude

Model uses observation to generate next thought, looping until task complete or maximum steps reached.

Effective ReAct Prompting Patterns

Getting ReAct systems to work well requires careful prompt engineering. The prompts must establish the format clearly, provide examples, and set expectations for when to conclude.

Format Specification and Examples

Prompts should specify the exact format for thoughts, actions, and observations. For example: "Generate thoughts in the format: Thought: [your reasoning]. Generate actions in the format: Action: search([query]). I will provide observations in the format: Observation: [results]." Following specification with examples greatly improves adherence: showing 1-2 complete trajectories (problem → thought → action → observation → thought → action → observation → final answer) trains the model to follow the pattern.

The format balance matters. Too rigid formats (e.g., exactly 3 thoughts per action) constrain flexibility for problems of varied complexity. Too loose formats (anything goes) result in inconsistent output that's hard to parse. Most effective systems provide structure (defined format for thoughts/actions/observations) while allowing variability in quantity and timing.

Stop Conditions and Termination Logic

ReAct loops could theoretically continue indefinitely—new thoughts leading to new actions indefinitely exploring. Systems must specify termination conditions: maximum number of steps, achieving a success criterion, recognizing that additional actions won't help. Clear stop conditions in prompts help: "Stop when you have found a definitive answer or determined that the answer cannot be found with available tools."

Tool Description and Availability

The model must understand what tools are available and when to use them. Prompt specification includes: "You have access to the following tools: [search tool - retrieves information from internet, calculator tool - computes mathematical expressions]. Use these tools when you need their specific capabilities." Clear tool descriptions with examples of usage dramatically improve appropriate tool invocation. Vague descriptions lead to tool misuse or failure to invoke tools when helpful.

Few-Shot Prompting for Consistency

Providing examples of the entire ReAct trajectory—thoughts, actions, observations, conclusions—for similar problems enables in-context learning. The model learns the task format and solution patterns from examples. For complex reasoning tasks, providing 1-3 examples often suffices; for diverse tasks, more examples provide better coverage of expected behaviors.

Prompt Complexity Trade-off: More detailed prompts improve adherence but consume context. Very long prompts leave less room for the actual reasoning loop and problem solving. Balance between specification and concision.

Loop Management and State Tracking

Implementing ReAct systems requires managing the reasoning loop—tracking state, maintaining context, and orchestrating transitions between thinking and acting.

Context Management and Loop History

As loops progress, context windows fill with thoughts, actions, and observations. For long reasoning chains, context becomes exhausted, preventing the model from considering all previous reasoning. Systems manage this through: (1) summarization—periodically summarize past reasoning and observations into concise summaries, (2) hierarchical reasoning—divide complex problems into sub-problems, reason over each, then synthesize, or (3) memory management—store intermediate results in external memory rather than context. Balancing context usage between problem context (what's being solved) and reasoning history (how it's being solved) is crucial.

Error Handling and Correction

Tools fail or return unexpected results. A search might return no results. A calculator might receive malformed input. ReAct systems must handle failures gracefully: (1) inform the model of the error, (2) enable the model to reason about alternatives, or (3) automatically retry with modified parameters. When tools fail repeatedly, systems should recognize dead ends and either suggest user clarification or conclude that the problem cannot be solved with available tools.

Loop Limits and Safeguards

Without safeguards, loops could run indefinitely. Systems implement step limits: maximum of 10 or 20 iterations, preventing runaway execution. Additionally, some systems implement loop detection: if the model repeats similar thoughts and actions without progress, the loop is terminated with a "stuck" conclusion. These safeguards prevent resource exhaustion and wasted computation.

Orchestration and State Management

Managing the ReAct loop requires: (1) parsing model output to extract thoughts, actions, and any final answer, (2) executing actions (tool invocations), (3) formulating observations, (4) determining whether to continue looping or terminate, and (5) if continuing, providing the model with the new observation and current state. This orchestration logic is error-prone; systems should validate that actions are executable before attempting execution, handle errors gracefully, and provide clear feedback when problems occur.

Advanced Patterns and Extensions

Beyond basic ReAct, several extensions and patterns enhance capabilities and reliability.

Plan-Then-Act Pattern

Basic ReAct generates plans incrementally (one step at a time). Extensions generate full plans before acting. The model might generate: "Step 1: Search for X. Step 2: Extract Y from search results. Step 3: Calculate Z using extracted Y. Step 4: Verify result makes sense." Then, the model executes the plan step by step. This pattern reduces backtracking because the full plan is visible upfront; if the plan seems infeasible, it can be revised before expensive execution.

Multi-Agent ReAct Patterns

Multiple agents can reason and act in coordination. One agent might be a researcher generating thoughts about what information is needed; another might be an executor performing actions; a third might be a validator checking results. These agents interact through shared observations and communication, enabling specialization and more robust reasoning. Multi-agent systems are more complex but can handle harder problems.

Reflection and Self-Correction

Advanced systems add reflection steps: after generating an answer, the model reviews its own reasoning and checks for errors. "Does my answer make sense? Have I considered alternative interpretations? Are there errors in my logic?" This self-critique loop often uncovers mistakes the model would otherwise miss. Some systems implement multiple reflection passes: generate answer, reflect, identify errors, generate corrected answer.

Deterministic Hybrid Systems

Pure language model reasoning is probabilistic and sometimes unreliable. Hybrid systems combine language models with deterministic components: the model reasons about what computation to perform, then a deterministic system performs it. For math problems, the model decides which mathematical operations to apply, then a symbolic math system executes them precisely. This combines language understanding flexibility with computational certainty.

ReAct Emerging Patterns: As practitioners build ReAct systems, new patterns emerge. Retrieval-AugmentedReAct (incorporating RAG with ReAct loops), Hierarchical ReAct (decomposing complex problems into subroutines), and Grounded ReAct (ensuring observations truly ground reasoning rather than the model hallucinating observations) are active research directions.

Evaluating and Improving ReAct Performance

ReAct systems are complex. Evaluating and debugging them requires systematic approaches.

Trajectory Analysis

Successful execution leaves a history—the complete thought-action-observation trajectory. Examining successful and failed trajectories reveals patterns. Successful examples might show clear reasoning, appropriate tool use, and logical progression. Failed examples might show: hallucinating tools that don't exist, repeating same actions without progress, or reasoning that doesn't match actions taken. Analyzing trajectories enables targeted improvements.

Bottleneck Identification

Failures can occur at multiple points: poor reasoning (model generates wrong thoughts), tool misuse (model invokes wrong tools or with wrong parameters), observation processing (model misinterprets results), or termination failures (reaches maximum steps before solving). Identifying where failures occur enables targeted fixes. If most failures are observation processing errors, improving that component pays dividends. If failures are reasoning errors, prompting improvements help.

Metrics

Basic metrics: Did the system reach a correct answer? Did it exceed maximum steps? Did it use tools appropriately? More nuanced metrics: How many tool invocations were wasted (didn't contribute to solution)? How many reasoning steps were extraneous? What's the efficiency (answer quality per resource consumed)? These metrics help optimize systems.

Human Oversight and Validation

For critical applications, human validation of ReAct trajectories is essential. Humans can verify: Is the reasoning sound? Are the tool invocations appropriate? Does the conclusion follow from evidence? This provides ground truth for evaluation and identifies failure modes that metrics miss.

Key Takeaways

ReAct Synergizes Reasoning and Acting: Combining visible thoughts with tool use outperforms either alone through iterative refinement.
Format Specification is Critical: Clear, consistent formatting for thoughts, actions, and observations enables reliable parsing and execution.
Examples Drive Behavior: In-context examples of complete ReAct trajectories teach the model the expected pattern and improve consistency.
State and Context Management Matter: Tracking loop state, managing context windows, and preventing infinite loops are essential engineering concerns.
Error Handling Enables Robustness: Graceful failure recovery, tool validation, and sensible defaults make systems reliable in practice.
Trajectory Analysis Reveals Improvement Opportunities: Examining successful and failed reasoning loops identifies bottlenecks for targeted optimization.
