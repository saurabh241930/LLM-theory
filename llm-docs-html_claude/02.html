Tokenization & Context Windows

How Language Models Break Down Text and Process Information Within Hard Memory Limits

Tokens: The Atomic Unit of Language Models

Language models do not process text character by character or even word by word. Instead, they work with tokens—variable-length chunks of text that serve as the fundamental unit of computation. A token might consist of a single character, part of a word, a complete word, or even multiple words with punctuation, depending on how the tokenizer is designed

Consider the word "hello." In most tokenizers, it will be represented as a single token. However, a complex word like "unfortunately" might require two tokens: "un" and "fortunately." This variable-length segmentation reflects a fundamental compromise: the vocabulary of a tokenizer is finite (typically 50,000 to 200,000 tokens), so infrequent character sequences must be broken into smaller pieces.

Under the hood, language models never actually "see" the text directly. The tokenizer first converts text into a sequence of token IDs—integer indices into the model's vocabulary. The model then uses an embedding layer to convert these IDs into high-dimensional vectors. This two-step process of string → tokenID → vector is invisible to users but absolutely critical to understanding how these systems work.

Critical Insight: The speed and cost of language model inference scale linearly with the number of tokens. Minimizing tokens is essential for production systems.

The Tokenization Pipeline

Tokenization occurs in a distinct preprocessing phase before text enters the language model. Different tokenizers employ different strategies, but most modern models use one of two approaches: Byte-Pair Encoding (BPE) or WordPiece tokenization.

Byte-Pair Encoding (BPE)

BPE begins with raw bytes and iteratively merges the most frequently-occurring adjacent byte pairs into new tokens. A tokenizer might learn that the bytes for "th" appear together so often that they should be merged into a single token. Over thousands of merge operations, the algorithm builds a vocabulary that balances frequency (common sequences become single tokens) with coverage (rare sequences can still be represented).

WordPiece Tokenization

WordPiece, used by BERT and similar models, starts with words and breaks them into subword units based on learned frequencies. It begins with character-level tokens and merges them, but maintains the assumption that word boundaries are significant. A word that doesn't exist in the vocabulary is prefixed with "##" to indicate it's a continuation.

Once tokenization is complete, the sequence of token IDs flows into the language model, where each ID is converted into a high-dimensional embedding vector. The entire computational engine of the model operates on these embeddings, not on the original text.

Tokenizer Variance and Its Implications

A critical practical detail that often catches practitioners off guard: different models use different tokenizers, and identical prompts produce different token counts across models. This variance has direct financial and performance implications, especially in cloud-based API environments where billing is token-based.

Why Token Counts Vary

OpenAI's GPT models, Anthropic's Claude, and Meta's Llama each developed their own tokenizers optimized for their training data and architectural considerations. A tokenizer trained primarily on English text might represent single English letters efficiently but require multiple tokens for less common scripts. Space characters are handled differently—some tokenizers include spaces with the preceding word, others treat them as separate tokens.

For a 100-word prompt, you might expect approximately 130 tokens. But actual counts could range from 120 to 150 depending on the tokenizer. When operating at scale with millions of API calls, this variance compounds into significant cost differences and affects latency calculations.

Context Windows: The Hard Memory Limit

Every language model has a context window—a maximum number of tokens it can process simultaneously. This is a hard limit imposed by the model architecture itself. The transformer architecture's attention mechanism has computational complexity that grows quadratically with sequence length, making longer contexts exponentially more expensive.

The context window encompasses everything: the system prompt that defines the model's behavior, the conversation history, the user's current query, and the space reserved for the model's response. All of these components compete for the same fixed token budget.

Common Context Window Sizes

  • GPT-3.5-turbine: 4,096 tokens (approximately 3,000 words)
  • GPT-4: 8,192 tokens standard, 32,768 tokens with 32K variant (limited availability)
  • Claude 2: 100,000 tokens (approximately 75,000 words)
  • Gemini Pro: 32,000 tokens
  • Llama 2-70B: 4,096 tokens

Exceeding the context window produces catastrophic failure modes: either the API returns an error, or the system silently truncates older context, causing the model to lose crucial information necessary for coherent response generation. This truncation often happens in multi-turn conversations where historical context gets progressively eliminated to make room for new messages.

Token Optimization and Context Management

Given that tokens directly determine latency and cost, and context windows impose hard limits, sophisticated applications must implement token optimization strategies. Three primary approaches dominate:

Strategy 1: Summarization

Pre-process documents offline to extract summaries before they enter the model's context. A 10,000-token document can typically be summarized to 3-5% of its original length while retaining the essential information. For historical conversation context, rolling summaries gradually replace old turn-by-turn exchanges with consolidated summaries, freeing tokens for new interactions.

Strategy 2: Retrieval-Augmented Generation (RAG)

Instead of including all relevant documents, retrieve only the most semantically similar chunks from a vector database. This allows applications to access knowledge bases orders of magnitude larger than the context window, retrieving only the relevant portions. A system might have access to millions of documents but include only 2-3 relevant ones in any given query.

Strategy 3: Prompt Compression

Carefully craft prompts to eliminate redundant words while preserving essential information. Stop words, articles, and verbose explanations can typically be removed without impacting instructions. Some advanced systems use AI-powered prompt compression that selects the most semantically important tokens while maintaining grammatical validity.

Special Tokens and Hidden Overhead

Beyond visible text, language models use invisible special tokens to structure conversations and delineate roles. These tokens consume from the context window budget but are typically invisible to end users, creating hidden overhead that practitioners must account for.

Common Special Tokens

  • <|im_start|> and <|im_end|>: Delimit individual turns in multi-turn conversations
  • <|system|>: Marks the beginning of the system prompt section
  • <|user|> and <|assistant|>: Distinguish between roles in conversations
  • Padding tokens: Added to ensure batch processing uses uniform lengths

A seemingly simple three-turn conversation might actually require 6-8 special tokens in addition to the visible text tokens. A system prompt of 10 natural language words plus the <|system|> and end markers might require 15 total tokens. This overhead compounds significantly in applications with many turns or large system prompts.

Practical Recommendation: Always add a 10-15% buffer to token estimates to account for special tokens, punctuation, and formatting characters. Real-world token counts typically exceed simple estimates.

Token Budgeting in RAG Systems

Retrieval-Augmented Generation systems must carefully balance between context richness and token constraints. A naive approach might retrieve the 10 most similar document chunks, but if each chunk is 500 tokens and the context window is 4,096 tokens, those chunks alone consume 78% of available context, leaving minimal space for the query and model output.

Effective RAG systems implement intelligent budgeting:

  • Reserve Output Space: Always allocate tokens for the expected response. A system generating outputs up to 500 tokens must reserve this space.
  • Limit Retrieved Chunks: Even if 10 chunks score well, retrieve only 2-3 if space is constrained. Quality of top results matters more than quantity.
  • Chunk Size Tuning: Smaller chunks (256 tokens) maximize retrieval flexibility but might lose context. Larger chunks (1024+) preserve coherence but dominate the context window.
  • Dynamic Budgeting: Some systems compute required output space dynamically based on query complexity, then allocate remaining space to retrieval.

Advanced RAG systems even compress retrieved chunks by removing redundant passages, preserving only the sentences directly relevant to answering the query. This might reduce 500-token chunks to 200-token compressed versions while maintaining answer quality.

Tokenization and Context Pitfalls

Pitfall 1: Underestimating Token Counts Simple word-count heuristics (dividing word count by 1.3-1.4) often underestimate actual token counts by 10-20%, especially for code, structured data, or non-English text. Always use official tokenizers for accurate counts.
Pitfall 2: Forgetting Special Token Overhead System prompts, role markers, and turn delimiters add invisible tokens. A 10-word system prompt might require 15-20 tokens after tokenization and special markers. This overhead often surprises developers.
Pitfall 3: Using Outdated Token Counts When upgrading to a new model version, token counts may change. An API endpoint that worked perfectly with GPT-3 might fail silently with GPT-4 if context calculation wasn't updated. Always requote after model upgrades.
Pitfall 4: Ignoring Token Variance Across Tokenizers The same prompt produces different token counts in different models (variance of 15-30%). Cross-model applications must test with each tokenizer independently rather than assuming consistent counts.

Key Takeaways

Tokens ≠ Words: Variable-length units that average 1.2-1.4 per English word but vary significantly based on content type and tokenizer choice.
Context Windows Are Hard Limits: The maximum tokens (input + output) is fixed by architecture. Exceeding it causes errors or silent truncation.
Different Models, Different Tokenizers: Identical text produces different token counts across models. Always verify with the specific model's tokenizer.
Optimization is Essential: Summarization, RAG retrieval, and prompt compression are not optional—they're necessary for production systems.
Special Tokens Add Hidden Overhead: System markers, role indicators, and turn delimiters consume budget but remain invisible to users.
Budget Carefully in RAG: Divide context window among query, history, documents, and output space. Naive approaches waste budget retrieving too many documents.
